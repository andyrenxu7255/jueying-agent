# 文档 14：数据库表设计与索引 v1.0

## 14.1 文档目的

本文件在《Database Logical Model v0.1》基础上，补齐可落地的 PostgreSQL 物理设计建议，覆盖：

- 主键与字段类型建议
- 主外键与唯一约束建议
- 必要索引建议
- pgvector 与 AGE 落地方式
- Day 1 需要新增的物理表

本文件仍是设计草案，不等同于最终 migration SQL，但已经足够指导后续生成 DDL。

本文件与以下补充文档强关联：

- 《文档 16》定义 `scope`、`owner`、`policy_snapshot` 约束
- 《文档 17》定义 WorkflowPlan / stage 契约中必须落表的字段
- 《文档 18》定义 `execution_session` 的最小结构
- 《文档 19》定义 `checkpoint` 的恢复字段
- 《文档 20》定义 `retrieval_trace`、`fact_conflict` 与 Evidence Pack 相关字段

## 14.2 物理设计原则

### 14.2.1 主键策略

- Day 1 统一使用 `uuid` 作为主键。
- 主键默认值使用 `gen_random_uuid()`。
- 对外展示 ID 可以继续包装为 `wf_xxx`、`ep_xxx`、`sk_xxx` 等业务标识，但数据库内键统一用 `uuid`。

### 14.2.2 时间字段

- 统一使用 `timestamptz`。
- 所有核心表必须有 `created_at timestamptz not null default now()`。
- 所有可更新对象必须有 `updated_at timestamptz not null default now()`。

### 14.2.3 状态与枚举策略

- Day 1 优先使用 `text + check constraint`，不急于上 PostgreSQL enum。
- 原因是 Workflow、skill、memory、artifact 状态仍可能演化，文本约束迁移成本更低。

### 14.2.4 JSONB 策略

- `metadata`、预算、约束、trace detail、provider detail 统一用 `jsonb`。
- 只有在查询路径稳定且高频时，才把 JSONB 字段提升为固定列。

### 14.2.5 大对象策略

- 原文、附件、patch、测试日志、checkpoint payload 大文件不直接存数据库正文。
- 数据库存元数据、hash、大小、存储引用。
- 对于小于 `32 KB` 的简短摘要可内联存于数据库。

### 14.2.6 向量策略

- Day 1 固定一套主 embedding 模型与维度，推荐 `1536` 维。
- 不在同一列混存多维向量。
- 若后续更换 embedding 模型，采用新列或新表版本化迁移，不在运行期混用。

### 14.2.7 共享与运行态范围规则

- Day 1 公共区只允许两类资源进入：`workflow_definition` 与 `skill`。
- Day 1 运行态对象默认只允许私有：`workflow_instance`、`workflow_stage`、`checkpoint`、`execution_session`。
- `artifact_object` 在 Day 1 默认作为私有或系统内部留痕对象存在，不作为公共资源暴露；即使服务于公共发布，也只通过 `workflow_definition` 或 `skill` 的安全结果暴露。

## 14.3 扩展与初始化建议

```sql
create extension if not exists pgcrypto;
create extension if not exists vector;
create extension if not exists pg_trgm;
create extension if not exists age;
load 'age';
set search_path = ag_catalog, "$user", public;
```

说明：

- `pgcrypto` 用于 UUID 生成与 hash 辅助。
- `vector` 用于语义候选召回。
- `pg_trgm` 用于名称、标题、关键文本的模糊匹配与全文补充。
- `age` 只为图增强查询服务。

## 14.4 Day 1 建议新增的物理表

相较《逻辑模型 v0.1》，Day 1 建议增加四张表：

| 表 | 目的 |
|---|---|
| `execution_session` | 承接开发阶段执行会话，不再只靠 `workflow_stage + checkpoint` 隐式表达 |
| `artifact_object` | 统一管理 patch、测试结果、日志、附件、报告、checkpoint payload |
| `workflow_event` | 结构化保存状态迁移、进度回写与阶段事件 |
| `fact_conflict` | 显式记录事实冲突对、裁决状态与人工确认链路 |

## 14.5 表设计建议

### 14.5.0 Day 1 冻结实现选择（消歧义）

以下条目在 Day 1 固定，不允许实现阶段自行改写：

1. `entity` 不启用硬唯一约束 `unique (scope_type, owner_user_id, entity_type, canonical_name)`，仅保留索引。
2. `skill` 不启用硬唯一约束 `unique (owner_user_id, name)`，仅保留索引。
3. `workflow_instance.scope_type` 固定 `private`（通过 check 约束落地）。
4. `artifact_object.scope_type` 固定 `private`，不作为公共区独立资源。

若后续要调整以上任意条目，必须先修改文档并通过 M0 一致性校验，不得在开发中临时改动。

### 14.5.1 身份与权限

#### `user`

- 主键：`id uuid pk`
- 关键字段：`org_id uuid`、`username text`、`display_name text`、`role text`、`status text`、`metadata jsonb`
- 约束：`unique (org_id, username)`
- 索引：`idx_user_role`、`idx_user_status`

#### `channel_identity`

- 主键：`id uuid pk`
- 外键：`user_id -> user.id`
- 关键字段：`channel_type text`、`external_identity text`、`binding_status text`、`metadata jsonb`
- 约束：`unique (channel_type, external_identity)`
- 索引：`idx_channel_identity_user`、`idx_channel_identity_binding_status`

#### `policy_snapshot`

- 主键：`id uuid pk`
- 外键：`user_id -> user.id`
- 关键字段：`role text`、`allowed_scopes jsonb`、`constraints jsonb`、`snapshot_hash text`
- 约束：`unique (snapshot_hash)`
- 索引：`idx_policy_snapshot_hash`、`idx_policy_snapshot_user`

说明：

- 物理设计中，`workflow_instance.policy_snapshot_id` 是权威关联。
- `policy_snapshot` 不再反向持有硬 FK 到 `workflow_instance`，避免循环外键。

### 14.5.2 Workflow 治理

#### `workflow_definition`

- 主键：`id uuid pk`
- 关键字段：`scope_type text`、`owner_user_id uuid`、`name text`、`workflow_type text`、`risk_level text`、`status text`、`version integer`、`definition_json jsonb`
- 外键：`owner_user_id -> user.id`
- 约束：`unique (owner_user_id, name, version)`
- 索引：`idx_workflow_definition_scope_owner`、`idx_workflow_definition_type_status`

说明：

- `scope_type = public` 只适用于可复用 template，不适用于运行态实例。

#### `workflow_instance`

- 主键：`id uuid pk`
- 外键：`workflow_definition_id -> workflow_definition.id`、`owner_user_id -> user.id`、`policy_snapshot_id -> policy_snapshot.id`
- 关键字段：`scope_type text`、`status text`、`workflow_plan_hash text`、`budget_json jsonb`、`input_summary jsonb`、`started_at`、`finished_at`
- 约束：`unique (workflow_plan_hash, owner_user_id, created_at)` 不建议做硬唯一；只保留普通索引
- 索引：
  - `idx_workflow_instance_owner_status`
  - `idx_workflow_instance_scope_status`
  - `idx_workflow_instance_started_at`
  - `idx_workflow_instance_policy_snapshot`

说明：

- Day 1 推荐对 `scope_type` 加 `check (scope_type = 'private')`。

#### `workflow_stage`

- 主键：`id uuid pk`
- 外键：`workflow_instance_id -> workflow_instance.id`、`checkpoint_id -> checkpoint.id`
- 关键字段：`stage_key text`、`stage_type text`、`seq integer`、`assigned_executor text`、`status text`、`input_refs jsonb`、`output_refs jsonb`、`stage_input_hash text`、`stage_output_hash text`、`tool_call_refs jsonb`、`evidence_refs jsonb`、`fact_write_refs jsonb`、`verification_refs jsonb`、`acceptance_result jsonb`、`next_action text`、`metadata jsonb`
- 约束：`unique (workflow_instance_id, seq)`、`unique (workflow_instance_id, stage_key)`
- 索引：`idx_workflow_stage_instance_status`、`idx_workflow_stage_executor_status`

#### `checkpoint`

- 主键：`id uuid pk`
- 外键：`workflow_instance_id -> workflow_instance.id`、`workflow_stage_id -> workflow_stage.id`
- 关键字段：`checkpoint_type text`、`resume_token text`、`state_hash text`、`policy_snapshot_hash text`、`status_snapshot jsonb`、`artifact_refs jsonb`、`fact_write_refs jsonb`、`verification_refs jsonb`、`evidence_pack_hash text`、`next_action text`、`tool_call_refs jsonb`、`notes text`
- 约束：`unique (resume_token)`
- 索引：`idx_checkpoint_workflow_stage`、`idx_checkpoint_created_at`

说明：

- `tool_call_refs` 和 `notes` 字段存储在 `metadata` JSONB 字段中，运行期可通过 metadata 访问。
- `checkpoint_type` 支持的值：`stage-enter`、`stage-exit`、`waiting-user`、`blocked`、`paused`、`repair`。

#### `workflow_event`

- 主键：`id uuid pk`
- 外键：`workflow_instance_id -> workflow_instance.id`、`workflow_stage_id -> workflow_stage.id`
- 关键字段：`event_type text`、`from_status text`、`to_status text`、`event_payload jsonb`、`occurred_at timestamptz`
- 索引：`idx_workflow_event_instance_time`、`idx_workflow_event_stage_time`、`idx_workflow_event_type`

#### `execution_session`

- 主键：`id uuid pk`
- 外键：`workflow_instance_id -> workflow_instance.id`、`workflow_stage_id -> workflow_stage.id`、`owner_user_id -> user.id`
- 关键字段：`status text`、`repo_ref text`、`branch_ref text`、`worktree_ref text`、`base_commit_hash text`、`stage_goal text`、`budget_json jsonb`、`acceptance_rules jsonb`、`backend_type text`、`policy_snapshot_hash text`、`checkpoint_id uuid`
- 约束：`check (status in ('created', 'preparing', 'ready', 'running', 'verifying', 'repairing', 'waiting_workflow', 'completed', 'failed', 'terminated'))`
- 索引：`idx_execution_session_stage`、`idx_execution_session_status`、`idx_execution_session_repo`

说明：

- `status` 状态迁移：`created -> preparing -> ready -> running -> (verifying | repairing) -> (completed | failed | terminated)`
- `waiting_workflow` 状态用于等待 Workflow 层决策

## 14.8 AGE 投影建议

### 14.8.1 图真相边界

- PostgreSQL 事务表是真相源。
- AGE 只保存图投影视图，不单独承载业务权威状态。

### 14.8.2 Day 1 投影对象

- `entity`
- `relation`
- `workflow_instance -> workflow_stage`
- `skill_version -> source`
- `fact -> evidence`

### 14.8.3 投影规则

1. 由后台投影任务从事务表读取变化。
2. 投影记录必须可回溯到 PostgreSQL 主键。
3. 删除或失效关系时，必须同步更新图投影。
4. 图查询一律受 `max_hops <= 2` 限制。

### 14.8.4 AGE 投影同步机制

#### 同步架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    PostgreSQL 事务表（真相源）                    │
│  entity / relation / workflow_instance / workflow_stage / fact  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 1. 写入时记录变更事件
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    projection_event 表                           │
│  记录待同步的变更事件（insert/update/delete）                      │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 2. BullMQ 任务队列
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Projection Worker                             │
│  消费事件，执行 AGE 图投影同步                                     │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 3. Cypher 写入
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Apache AGE 图投影                              │
│  顶点：entity_vertex / workflow_vertex / skill_vertex            │
│  边：relation_edge / stage_edge / evidence_edge                  │
└─────────────────────────────────────────────────────────────────┘
```

#### projection_event 表设计

```sql
create table projection_event (
    id uuid primary key default gen_random_uuid(),
    source_table text not null check (source_table in (
        'entity', 'relation', 'workflow_instance', 
        'workflow_stage', 'fact', 'skill_version'
    )),
    source_id uuid not null,
    operation text not null check (operation in ('insert', 'update', 'delete')),
    payload jsonb,
    status text not null default 'pending' check (status in (
        'pending', 'processing', 'completed', 'failed'
    )),
    retry_count int not null default 0,
    max_retries int not null default 3,
    error_message text,
    created_at timestamptz not null default now(),
    processed_at timestamptz,
    
    constraint uk_projection_event_pending unique (source_table, source_id, status) 
        where (status = 'pending')
);

create index idx_projection_event_status on projection_event (status, created_at);
```

#### 降级策略

| 场景 | 降级动作 | 影响 |
|------|----------|------|
| AGE 服务不可用 | 跳过图增强，只使用结构化+向量检索 | 检索结果无图增强 |
| 投影事件积压 > 1000 | 暂停新事件写入，优先处理积压 | 图数据暂时不更新 |
| 投影失败率 > 10% | 禁用 AGE，标记需要重建 | 图增强功能下线 |

#### 图投影重建

当 AGE 数据与 PostgreSQL 不一致时，执行全量重建：

```sql
-- 1. 清空 AGE 图
SELECT * FROM cypher('agent_harness_graph', $$
    MATCH (n) DETACH DELETE n
$$) as (a agtype);

-- 2. 重置 projection_event
UPDATE projection_event SET status = 'pending', retry_count = 0 
WHERE status != 'completed';

-- 3. 补充历史数据事件
INSERT INTO projection_event (source_table, source_id, operation, payload)
SELECT 'entity', id, 'insert', to_jsonb(e) FROM entity e WHERE status = 'active';

-- 4. 启动 Worker 重新消费
```

## 14.9 分区与归档建议

- Day 1 不急于做广泛分区。
- 仅对 `audit_event`、`workflow_event`、`retrieval_trace` 预留按月分区空间。
- 历史大对象依赖对象存储生命周期策略，不用数据库分表先行优化。
