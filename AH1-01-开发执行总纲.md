# 第零章：冻结决策基线

以下 16 条冻结决策在任何情况下都不可违反。若需变更，必须先修改本文档并通过 M0 一致性校验。

| 编号 | 冻结决策 | 违反后果 |
|------|----------|----------|
| F-01 | 单组织、多用户，不做多租户 SaaS | 架构偏离，权限模型失效 |
| F-02 | PostgreSQL 是唯一事实源 | 数据一致性崩溃 |
| F-03 | pgvector 只负责语义候选召回 | 向量库被当成事实源 |
| F-04 | Apache AGE 只做图增强，不是第二事实源 | 图数据与事务数据分裂 |
| F-05 | 不使用 Neo4j | 引入第二图数据库 |
| F-06 | 只保留"个人区 + 公共区"两种 scope | 权限模型复杂化 |
| F-07 | 普通用户默认严格隔离 | 越权读取 |
| F-08 | Workflow 不依赖 md/heartbeat 管理状态 | 状态不一致 |
| F-09 | Code Executor 与 OpenClaw 上下文隔离 | 上下文污染 |
| F-10 | 每阶段必须可恢复 | 恢复退化为重执行 |
| F-11 | 检索必须权限过滤，不能绕过 | 越权读取 |
| F-12 | Skill Day 1 默认私有 | 公共区污染 |
| F-13 | Memory 经验提示不可作为事实依据 | 事实层被经验覆盖 |
| F-14 | Hermes 不能直接写平台主事实 | 外部系统绕过治理 |
| F-15 | Replay 默认只读，禁止重执行时修改 | 历史数据被篡改 |
| F-16 | 预算必须可观测、可控制 | 成本失控 |

- 禁止把 AGE 用成独立图数据库产品

# 第二章：Workflow 状态机

## 2.1 状态定义（13 种）

| 序号 | 状态 | 触发条件 | 持续时间限制 |
|------|------|----------|-------------|
| 1 | `draft` | 用户发起请求 | ≤ 30s |
| 2 | `planned` | Planner 输出 WorkflowPlan | ≤ 60s |
| 3 | `running` | 阶段执行中 | 由 budgets 控制 |
| 4 | `verifying` | 进入验证阶段 | 由 stage timeouts 控制 |
| 5 | `repairing` | 修复阶段 | 由 repair_budget 控制 |
| 6 | `reporting` | 汇报阶段 | 由 stage timeouts 控制 |
| 7 | `waiting_user` | 需要用户输入/审批 | ≤ 7 天 |
| 8 | `blocked` | 外部依赖不可用 | ≤ 24h |
| 9 | `paused` | 用户主动暂停 | ≤ 30 天 |
| 10 | `succeeded` | 所有阶段完成 | 终态 |
| 11 | `failed` | 不可恢复的失败 | 终态 |
| 12 | `cancelled` | 用户取消 | 终态 |
| 13 | `archived` | 归档 | 永久 |

## 2.2 状态迁移规则（15 条）

| 编号 | 从 | 到 | 触发条件 | 审计事件 |
|------|----|----|----------|----------|
| T-01 | `draft` | `planned` | Planner 输出计划 | `workflow.planned` |
| T-02 | `planned` | `running` | 用户确认/自动开始 | `workflow.started` |
| T-03 | `running` | `verifying` | 进入验证阶段 | `workflow.verifying` |
| T-04 | `verifying` | `repairing` | 验证失败需修复 | `workflow.repairing` |
| T-05 | `repairing` | `verifying` | 修复后重新验证 | `workflow.re-verifying` |
| T-06 | `verifying` | `reporting` | 验证通过进入汇报 | `workflow.reporting` |
| T-07 | `reporting` | `running` | 汇报完成继续下一阶段 | `workflow.continued` |
| T-08 | `reporting` | `succeeded` | 所有阶段完成 | `workflow.succeeded` |
| T-09 | `running` | `waiting_user` | 需要用户输入 | `workflow.waiting_user` |
| T-10 | `running` | `blocked` | 外部依赖不可用 | `workflow.blocked` |
| T-11 | `running\|verifying\|repairing\|reporting` | `paused` | 用户暂停 | `workflow.paused` |
| T-12 | `paused` | `running` | 用户恢复 | `workflow.resumed` |
| T-13 | `planned\|running\|waiting_user\|blocked\|paused` | `cancelled` | 用户取消 | `workflow.cancelled` |
| T-14 | `planned\|running\|waiting_user\|blocked\|paused\|verifying\|repairing` | `failed` | 不可恢复失败 | `workflow.failed` |
| T-15 | `succeeded\|failed\|cancelled` | `archived` | 归档策略触发 | `workflow.archived` |

**实现约束：** 状态迁移必须使用 XState 定义，禁止手写 if-else 状态判断。

## 2.6 Planner 输出约束

**必须明确的内容（7 项）：**
1. Workflow 类型
2. 阶段链顺序
3. 每个阶段使用的 executor
4. 每个阶段是否需要检索
5. 每个阶段的验收条件
6. 超时、重试、修复上限
7. 需要用户确认或审批的节点

**禁止缺失的内容（5 项）：**
- 不允许缺失 `stage_type`
- 不允许缺失 `assigned_executor`
- 不允许缺失 `acceptance`
- 不允许缺失 `timeouts`
- 不允许缺失 `retry_policy`

**自主决策约束（4 条）：**
1. Planner 不得跳过 `Approval` 阶段
2. Planner 不得自行修改 `policy_snapshot`
3. Planner 不得超出 `budgets` 限制（超出时必须进入 `waiting_user`）
4. 每次决策必须写入 `workflow_event`

## 2.7 修复循环决策树

```
错误发生
  │
  ├── Transient（可重试）
  │     └── 重试次数 < max_retries → 重试
  │     └── 重试次数 >= max_retries → 进入修复评估
  │
  ├── Permanent（不可重试）
  │     └── 直接进入修复评估
  │
  └── System（系统级）
        └── 进入 blocked，等待系统恢复

修复评估
  │
  ├── 可修复
  │     └── 修复次数 < max_repairs → 进入 Repair 阶段
  │     └── 修复次数 >= max_repairs → 进入 waiting_user
  │     └── 连续 3 次修复失败 → 进入 waiting_user
  │
  ├── 需要用户输入
  │     └── 进入 waiting_user
  │
  └── 不可修复
        └── 进入 failed
```

**不稳定信号处理：**

| 信号 | 阈值 | 处理方式 |
|------|------|----------|
| 修复循环超过 max_repairs | - | 停止修复，进入 waiting_user |
| 连续 3 次修复失败 | - | 暂停，请求用户指导 |
| 上下文窗口使用 > 90% | - | 压缩上下文，若仍超限则 checkpoint 后暂停 |
| 同一文件反复修改 > 5 次 | - | 标记为"不稳定文件"，请求用户审查 |
| 测试通过率持续 < 50% | - | 回退到上一个 checkpoint，调整策略 |

## 2.8 Subagent 执行循环

| Subagent 类型 | 绑定 Executor | 职责 |
|---------------|---------------|------|
| `code-subagent` | `code-executor` | 代码开发自主编码循环 |
| `retrieval-subagent` | `retrieval-aware-executor` | 自主多轮检索 |
| `verification-subagent` | `verification-executor` | 自主运行测试并分析结果 |
| `repair-subagent` | `repair-executor` | 自主分析失败原因并修复 |
| `generic-subagent` | `generic-executor` | 规划、分析、汇报等 |

**Subagent 停止条件：**

| 条件 | 动作 |
|------|------|
| `acceptance.must_have` 全部满足 | 正常退出，输出结果 |
| `max_iterations` 达到上限 | 输出部分结果，标记 `needs_repair` |
| 上下文窗口使用 > 90% | 压缩上下文后继续，若仍超限则退出 |
| `hard_timeout_sec` 到期 | 强制退出，输出已有结果 |
| LLM 连续 3 次返回无效响应 | 退出，标记 `needs_repair` |

## 3.4 核心 API 端点

| 项目 | 内容 |
|------|------|
| 路径 | `POST /internal/channel-ingress/normalize` |

```json

  "raw_message": {},
  "channel_identity": "external_user_or_chat",
  "attachments": [],
  "session_hint": {}

```

响应 payload：
```json
{
  "user_id": "u_123 or null",
  "channel_type": "feishu",
```

# 第四章：权限与安全

## 4.1 Scope 定义

| scope token | 说明 |
|---|---|
| `private:{user_id}` | 用户私有作用域 |
| `public:workflow` | 公共工作流模板作用域（仅映射到 workflow_definition / template） |
| `public:skill` | 公共技能作用域 |

**明确禁止的 scope：** `department:*`、`group:*`、`share-with:user_x`、`team:*`、`org:*`

**Day 1 约束：** `workflow_instance`、`workflow_stage`、`checkpoint`、`execution_session` 不允许进入公共区。

## 4.2 Policy Snapshot 结构

```json
{
  "policy_snapshot_id": "ps_123",
  "user_id": "u_123",
  "role": "user",
  "acting_subject": "system:workflow",
  "allowed_scopes": ["private:u_123", "public:workflow", "public:skill"],
  "resource_rules": {
    "workflow_instance": ["read", "write", "update", "execute", "archive"],
    "skill": ["read"],
    "fact": ["read", "write"]
  },
  "constraints": {
    "max_graph_hops": 2,
    "allow_cross_user_read": false,
    "allow_public_publish": false
  },
  "snapshot_hash": "sha256:..."
}
```

## 4.3 policy_snapshot_hash 计算规则

**参与 hash 的字段：** `user_id`、`role`、`acting_subject`、`allowed_scopes`、`resource_rules`、`constraints`

**不参与 hash 的字段：** `created_at`、数据库自增版本号、临时 trace 信息

**算法：** 基于规范化 JSON 生成 SHA256，格式 `sha256:...`

## 4.4 角色定义

### 人类主体

| 角色 | 权限 |
|------|------|
| `user` | 只能访问自己的私有资源和公共区资源 |
| `admin` | 可跨用户读取、编辑、治理与发布，但必须审计 |

### 系统主体

| 主体 | 职责 |
|------|------|
| `system:gateway` | 仅负责接入、身份映射、消息回传 |
| `system:workflow` | 仅负责任务治理与状态推进 |
| `system:retrieval` | 仅负责受 snapshot 约束的检索 |
| `system:executor` | 仅负责阶段执行，不能绕开 snapshot |
| `system:admin-console` | 仅在 admin 代理场景下执行治理动作 |

**系统主体约束：** 不是超管账号，必须带 `acting_for_user_id` 或 `acting_for_admin_id`。

## 4.5 强制执行点

以下 7 个位置必须校验 `policy_snapshot_hash`：

1. 渠道身份解析后发起 Workflow 规划前
2. 进入 Retrieval 之前
3. 进入任何 Executor 之前
4. 执行工具调用前
5. 写入 `fact`、`memory_item`、`skill`、`artifact_object` 前
6. 发布公共 workflow 或 skill 前
7. checkpoint 恢复前

## 4.6 Scope 过滤 SQL

**普通用户：**
```sql
where (
  scope_type = 'private' and owner_user_id = :current_user_id
) or (
  scope_type = 'public' and :current_resource_kind in ('workflow_definition', 'skill')
)
```

**admin：**
```sql
where true
```
同时必须写审计，包含 `action = admin.cross_user_read`、`resource_ref`、`acting_admin_id`、`reason`。

**核心原则：** 检索不能先召回再过滤，必须先做 scope 过滤。

## 4.7 公共发布治理

公共区只允许两类资源：`workflow_definition` 和 `skill`，且必须满足：
1. 来源对象已经过验收或治理
2. 由 admin 审批或 admin 直接执行发布
3. 写入时保留来源链、证据链与发布审计

# 第六章：检索编排

## 6.1 检索链路分类

| 意图 | 首选链路 | 可选增强 | 默认禁用 |
|------|----------|----------|----------|
| `object-status` | 结构化 | 全文补充 | 向量全库 |
| `evidence` | 全文 | 向量召回、rerank | 图漫游 |
| `relation` | 结构化 + 图增强 | 全文补充 | 全库向量 |
| `similar-case` | 向量 | rerank、图补充 | 无约束 AGE |
| `dev-context` | 结构化 + 全文 + 向量 | 图增强 | 整仓注入模型 |
| `memory-hint` | memory 检索 | rerank | 直接覆盖主事实 |

## 6.2 检索编排流程（6 步）

| 步骤 | 名称 | 说明 |
|------|------|------|
| Step 0 | 权限过滤 | 输入 policy_snapshot_hash、allowed_scopes；输出可检索 scope 集合 |
| Step 1 | 结构化查询 | 有明确对象名/ID/owner 等线索时触发 |
| Step 2 | 全文检索 | 需要原文片段、查询含日志/文档/历史关键词时触发 |
| Step 3 | 向量召回 | 查询语义模糊、需要相似案例时触发；必须先 scope 过滤 |
| Step 4 | 图增强 | 需要 1-2 跳关系时触发；max_hops <= 2 |
| Step 5 | Rerank | 候选超过阈值、来源跨多个子系统时触发 |

**候选上限：** structured=20, fulltext=20, vector=30, graph=10, final_clip=12

## 6.3 Evidence Pack 最小输出字段

- `evidence_pack_id`
- `query_text`
- `intent_type`
- `scope_summary`
- `retrieval_steps`
- `items`（每个 item 可回溯到源记录，标出来源 scope）
- `clip_summary`
- `evidence_pack_hash`

## 6.4 Fact Write 模式

| 模式 | 说明 |
|------|------|
| `insert` | 新增事实声明 |
| `supersede` | 用新版本替代旧版本，但保留旧事实 |
| `conflict` | 发现冲突并登记 |
| `attach-evidence` | 为现有事实补证据 |

**Fact 状态流转：** `candidate` → `active` → `superseded` / `conflicted` / `rejected`

**冲突处理：** 新事实先写为 `candidate` → 创建 `fact_conflict` 记录 → 旧事实维持原状态 → 公共区发布需转人工确认

## 6.5 Memory 检索边界

**允许场景：** 用户偏好、历史经验提示、Dream 摘要线索、skill 形成原料

**禁止场景：** 客观审批结果、权限判断、主事实冲突裁决、公共发布最终结论

**Hermes Memory 映射：**

| Hermes 概念 | 平台概念 | 映射规则 |
|-------------|----------|----------|
| `memory.short_term` | 不映射 | 短期记忆不持久化 |
| `memory.long_term` | `memory_item` | 经 Adapter 转换，source_kind=hermes |
| `memory.dream` | `memory_item` | Dream 摘要经质量评估，memory_type=dream_summary |
| `memory.preference` | `memory_item` | 用户偏好直接写入，memory_type=user_preference |

**映射约束：** Hermes Memory 只能以候选形式进入平台；写入必须绑定 owner_user_id 和继承 policy_snapshot；写入必须生成 embedding。

## 6.6 Skill 封装规则

| Hermes 概念 | 平台概念 | 映射规则 |
|-------------|----------|----------|
| `skill.auto_generated` | `skill` + `skill_version` | 经 SkillExtraction 阶段评估后写入 |
| `skill.user_created` | `skill` + `skill_version` | 用户显式创建，直接写入 |
| `skill.community` | 不直接映射 | 公共 Skill 需经 admin 审核后发布 |

**Day 1 默认私有，admin 可发布到公共区。**

# 第七章：Checkpoint / Resume / Replay

## 7.1 Checkpoint 类型

| 类型 | 记录内容 |
|------|----------|
| Workflow Checkpoint | workflow_instance_id, 当前状态, 已完成阶段列表, 下一个待执行动作, policy_snapshot_hash |
| Stage Checkpoint | 当前阶段输入 hash, 输出 refs, evidence_pack_hash, artifact_refs, verification_refs, fact_write_refs |
| Execution Checkpoint | execution_session_id, repo_ref, branch_ref, worktree_ref, base_commit_hash, 最近 patch ref, 最近 test result ref |
| Waiting User Checkpoint | 缺失信息说明, 等待问题列表, 超时策略, 最近可交付物 refs |

## 7.2 必触发场景（6 种）

1. 进入 `waiting_user` 前
2. 进入 `blocked` 前
3. 进入 `paused` 前
4. 阶段成功完成后
5. Code Executor 完成关键 patch 与测试后
6. 发生可恢复失败且准备重试前

**禁止过度 checkpoint：** 不要每个 tool call 都写一次完整 checkpoint。

## 7.3 恢复流程

**恢复前检查（5 项）：**
1. `checkpoint_id` 存在
2. `policy_snapshot_hash` 一致
3. 所有关联 artifact ref 可读
4. 关联 repo/worktree/document/evidence 仍可访问
5. `resume_token` 未过期

**恢复动作顺序（5 步）：**
1. 装载 Workflow 与 Stage 结构化状态
2. 装载关联 artifact/evidence/verification refs
3. 若有执行会话，尝试恢复 worktree 或重新挂载执行环境
4. 根据 `next_action` 恢复到最近稳定边界
5. 写入恢复审计事件

**幂等规则：** 同一 `resume_token` 重放恢复请求必须返回同一恢复决策；重复请求不得重复写事实、重复发消息。

## 7.4 Replay 规则

**回放模式：**

| 模式 | 说明 |
|------|------|
| 审计回放 | 只重建事件链，不触发外部动作 |
| 调试回放 | 在隔离环境重演步骤，但不写真实事实 |
| 对比回放 | 比较新旧 planner/retrieval/executor 行为差异 |

**回放约束：** 默认只读，不得直接写主事实、公共资产或用户侧消息回执。

|--------|------|------|
| 文件系统 | Git Worktree | 每个执行会话独立工作目录 |
| 上下文窗口 | Subagent 独立上下文 | 不共享 OpenClaw 主上下文 |
| 进程 | 独立进程/容器 | 执行命令在隔离环境中运行 |

**上下文窗口管理策略：**

| 策略 | 触发条件 | 动作 |
|------|----------|------|
| 保留 | 始终 | 任务描述 + 验收条件 + 最近 3 轮交互 |
| 按需加载 | 读取文件时 | 只加载当前需要的文件片段，不预加载整个仓库 |
| 压缩 | 上下文使用 > 60% | 对历史交互生成摘要，替换原文 |
| 丢弃 | 上下文使用 > 80% | 丢弃重复/无关的工具输出 |
| 强制截断 | 上下文使用 > 95% | 保留任务描述+最新交互，截断其余 |

**代码上下文限制：** max_files=20, max_lines_per_file=200

## 8.3 调度策略

| 策略 | 说明 |
|------|------|
| 优先级调度 | 开发任务 > 分析任务 > 知识查询 |
| 预估时长排序 | 短任务优先（SJE） |
| 资源感知 | 根据当前 CPU/内存负载动态调整并发数 |
| 非开发任务绕行 | 知识查询/分析任务走 generic-executor |

**排队超限处理：**

| 等待时间 | 处理方式 |
|----------|----------|
| < 30s | 正常排队 |
| 30s - 2min | 发送进度通知 |
| > 2min | 触发告警 |
| > 5min | Workflow 进入 `blocked` |

## 9.3 降级矩阵

### 检索降级

| 失败点 | 降级策略 | 影响 |

| Rerank 失败 | 使用规则排序 | 排序质量下降 |
| Memory 检索失败 | 跳过 Memory | 经验提示缺失 |

### LLM 降级

| 场景 | 降级策略 |
|------|----------|

| Embedding 失败 | 跳过向量召回 |
| Rerank 失败 | 使用 BM25 排序 |

# 第十章：渠道接入与身份绑定

## 10.1 渠道接入统一输出

```json
{
  "channel_type": "wecom|feishu|web_portal",
  "channel_account_id": "acct_123",
  "channel_identity": "user_or_chat_id",
  "session_hint": {
    "conversation_id": "conv_123",
    "thread_id": "th_123"
  },
  "request_text": "...",
  "attachments": [],
  "raw_message_ref": "raw_123",
  "received_at": "2026-04-20T10:00:00Z"
}
```

## 10.2 身份绑定规则

1. 同一 `(channel_type, channel_identity)` 只能绑定一个 `user_id`
2. 未绑定身份不得直接创建 Workflow
3. 绑定冲突必须由 admin 处理

**绑定状态：** `pending` → `bound` / `disabled` / `conflicted`

## 10.3 Session Key 生成规则

- DM：`<channel_type>:<account_id>:dm:<channel_identity>`
- 群聊/频道：`<channel_type>:<account_id>:conv:<conversation_id>`
- 线程：在会话 key 后追加 `:thread:<thread_id>`

## 10.4 Session 到 Workflow 映射规则

**可复用条件（必须同时满足）：**
1. session_ref 对应存在活动 Workflow
2. owner_user_id 与当前用户一致
3. Workflow 状态属于 `running`、`waiting_user`、`paused`、`blocked`
4. 当前消息被识别为"继续当前任务"

**必须新建条件：**
1. 没有匹配的活动 Workflow
2. 当前消息意图明显切换到新任务
3. 原 Workflow 已 `succeeded`、`failed`、`cancelled` 或 `archived`

**明确禁止：**
- 禁止不同用户共享同一私有 Workflow
- 禁止通过公共群聊 thread 意外续接到他人私有开发任务

# 第十一章：Artifact 与对象存储

## 11.1 Artifact 类型

`raw_attachment`、`document_source`、`patch`、`test_result`、`command_log`、`verification_report`、`error_log`、`checkpoint_payload`、`final_report`、`evidence_excerpt`

## 11.2 存储分层

- **数据库层：** `artifact_object` 保存元数据（scope_type、owner_user_id、artifact_type、storage_ref、content_hash、content_size、mime_type）
- **对象存储层：** MinIO 保存实际正文

## 11.3 Key 命名规则

```text
org/<org_id>/wf/<workflow_id>/st/<stage_id>/<artifact_type>/<artifact_id>
```

## 11.4 写入流程

1. 组件产出 artifact 正文
2. 计算 `content_hash`（sha256）
3. 正文上传 MinIO
4. 数据库写 `artifact_object` 元数据
5. 若数据库落盘失败，触发补偿清理或重试

## 11.5 安全规则

1. Artifact 不得绕过 scope 与 owner 校验单独暴露
2. 涉及凭证、私钥的 artifact 必须标记敏感级别，默认禁止模型直接读取原文
3. `checkpoint_payload` 仅供恢复与调试使用，不得直接进入模型上下文
