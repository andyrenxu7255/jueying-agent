## 二、架构审计

### 2.1 七层架构评估

```
接入与总控层（OpenClaw Layer）     --> 渠道接入、身份解析、请求路由、结果回传
Workflow 治理层                    --> 任务生命周期、阶段编排、状态迁移、Checkpoint
Executor Layer                     --> 阶段内自治执行、工具调用、验证修复、Subagent
Fact / Retrieval Layer             --> 唯一事实源、混合检索、Evidence Pack、图增强
Memory / Skill Enhancement Layer   --> Hermes 适配、memory/dream/skill 候选治理
Provider Layer                     --> LLM/Embedding/Rerank/外部工具适配
Infrastructure Layer               --> PostgreSQL+pgvector+AGE、Redis、MinIO、SigNoz
```

**评估结论：架构设计优秀，分层清晰，职责边界明确。**

| 评估维度 | 评分 | 说明 |
|----------|------|------|
| 合理性 | 9/10 | 七层架构职责清晰，依赖方向正确（上层依赖下层） |
| 可扩展性 | 8/10 | 插件化渠道、动态 Executor 选择、Provider 路由均支持扩展 |
| 性能表现 | 7/10 | 单机 4C16G 20 用户目标合理，但 AGE + pgvector 叠加查询性能需实测 |
| 安全性 | 8/10 | 16 条冻结决策基线、7 个强制执行点、检索先过滤后召回 |

### 2.2 模块划分评估

**优势：**
- Workflow 状态机使用 XState 定义，禁止手写 if-else，保证状态迁移可验证
- 13 种状态、15 条迁移规则、16 种 Stage Type，覆盖完整
- Policy Snapshot 机制确保权限判定可追溯
- Checkpoint 4 类 6 种必触发场景，恢复前 5 项检查

**风险点：**

| 风险 | 严重程度 | 说明 |
|------|----------|------|
| AGE 稳定性 | 中 | 单机 4C16G 下 AGE 查询成本需 PoC 验证 |
| pgvector 性能 | 中 | 强权限过滤叠加向量召回可能带来性能压力 |
| Code Executor 资源 | 高 | 开发长任务可能优先耗尽 token/等待时间/测试资源 |
| Checkpoint 粒度 | 中 | 粒度设计不合理会导致恢复退化为重新执行 |
| 多用户隔离 | 高 | scope/owner/snapshot 字段体系不统一极易串数 |
| Hermes 边界 | 中 | 边界不清容易把 memory 系统误当真相源 |

### 2.3 数据流评估

**6 步检索编排流程设计优秀：**
```
Step 0: 权限过滤 → Step 1: 结构化查询 → Step 2: 全文检索 → Step 3: 向量召回 → Step 4: 图增强 → Step 5: Rerank → Clip → Evidence Pack
```

候选上限控制合理：structured=20, fulltext=20, vector=30, graph=10, final_clip=12

**降级矩阵设计完善：** 结构化 → 全文 → 向量 → 图增强 → Rerank → Memory，逐级降级

### 2.4 16 条冻结决策基线评估

所有 16 条决策基线均合理且不可回退，特别值得关注的：

1. **PostgreSQL 是唯一事实源** — 避免多源不一致
2. **检索必须权限过滤，不能绕过** — 先做 scope 过滤再检索
3. **Hermes 不能直接写平台主事实** — 防止记忆污染事实
4. **每阶段必须可恢复** — Checkpoint 保障
5. **预算必须可观测、可控制** — 成本治理

    r = subprocess.run(qc.get("command", ""), shell=True, capture_output=True, text=True, timeout=30)

if qc.get("type") == "exec":
