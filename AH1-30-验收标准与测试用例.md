使用人工标注的测试集评估检索质量：

```python
# 评估脚本示例
def evaluate_retrieval_quality(test_queries, ground_truth, top_k=10):
    """
    评估检索召回质量
    
    Args:
        test_queries: 测试查询列表
        ground_truth: 每个查询的相关文档 ID 集合
        top_k: 检索返回的候选数量
    
    Returns:
        dict: 包含 Recall@K, MRR, NDCG 等指标
    """
    results = {'recall@5': [], 'recall@10': [], 'mrr': [], 'ndcg@10': []}
    
    for query, relevant_docs in zip(test_queries, ground_truth):
        retrieved = retrieve(query, top_k=top_k)
        
        # Recall@K
        results['recall@5'].append(recall_at_k(retrieved, relevant_docs, k=5))
        results['recall@10'].append(recall_at_k(retrieved, relevant_docs, k=10))
        
        # MRR (Mean Reciprocal Rank)
        results['mrr'].append(reciprocal_rank(retrieved, relevant_docs))
        
        # NDCG@10
        results['ndcg@10'].append(ndcg_at_k(retrieved, relevant_docs, k=10))
    
    # 计算平均值
    return {k: np.mean(v) for k, v in results.items()}

# 验收标准
assert metrics['recall@10'] >= 0.85, "Recall@10 应 ≥ 85%"
assert metrics['mrr'] >= 0.70, "MRR 应 ≥ 70%"
```

```yaml
# k6 压测脚本示例
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate } from 'k6/metrics';

export let options = {
  stages: [
    { duration: '2m', target: 10 },  // 预热
    { duration: '5m', target: 20 },  // 负载阶段
    { duration: '2m', target: 20 },  # 持续阶段
    { duration: '1m', target: 0 },   # 降温
  ],
  thresholds: {
    http_req_duration: ['p(95)<3000'],  // P95 < 3s
    http_req_failed: ['rate<0.01'],       # 错误率 < 1%
  },
};

let errorRate = new Rate('errors');

export default function () {
  const payload = JSON.stringify({
    intent: 'knowledge-query',
    query: '什么是 Agent Harness 的核心架构？',
  });

  const params = {
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${__ENV.TOKEN}`,
    },
  };

  let res = http.post(`${__ENV.BASE_URL}/api/v1/workflows`, payload, params);

  check(res, {
    'status is 201': (r) => r.status === 201,
    'workflow created': (r) => r.json('id') !== undefined,
  }) || errorRate.add(1);

  sleep(Math.random() * 3);  // 模拟用户思考时间
}
```

## 30.6 缺陷管理流程

### 30.6.1 缺陷严重程度定义

| 严重程度 | 定义 | 响应时间 | 修复时间 | 示例 |
|----------|------|----------|----------|------|
| **Critical (致命)** | 系统崩溃、数据丢失、安全漏洞 | 1 小时 | 24 小时 | 数据库删除操作未授权 |
| **Major (严重)** | 核心功能不可用、无 workaround | 4 小时 | 3 天 | Workflow 无法创建 |
| **Moderate (一般)** | 功能异常但有 workaround | 24 小时 | 1 周 | 搜索结果排序不准确 |
| **Minor (轻微)** | UI 问题、体验不佳 | 3 天 | 2 周 | 错误消息不够友好 |
| **Trivial (建议)** | 优化建议、锦上添花 | - | 下版本 | 希望增加暗色主题 |

### 30.6.2 缺陷生命周期

```
New → Triaged → In Progress → Fixed → Verified → Closed
  ↓       ↓                          ↓
Rejected                     Reopened (if failed verification)
```
