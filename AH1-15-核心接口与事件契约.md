# 文档 15：核心接口与事件契约 v1.0

## 15.1 文档目的

本文件把《Adapter & Integration Contract V1》从概念描述进一步细化为统一契约，目标是让后续开发可以直接：

- 定义内部 API
- 生成 DTO / schema
- 约束事件总线 payload
- 编写 contract test

本文件定义的是“规范层契约”，不是最终传输实现。Day 1 推荐：

- 同步调用使用 HTTP/JSON 或进程内接口。
- 异步事件通过 Outbox + Redis Stream 或等价事件总线传播。

术语约束：

- `public:workflow` 在接口层统一表示可复用的 `workflow_definition` / template，不表示运行态 `workflow_instance`。
- `workflow_instance_id`、`workflow_stage_id` 在“预绑定 / 预规划”接口中允许为空；进入 Workflow 后必须有值。

## 15.2 统一信封结构

### 15.2.1 命令/查询请求信封

```json
{
  "request_id": "req_123",
  "trace_id": "tr_123",
  "actor": {
    "actor_type": "user|admin|system",
    "actor_id": "u_123",
    "channel_type": "wecom|feishu|web_portal|system"
  },
  "workflow_instance_id": "wf_123 or null",
  "workflow_stage_id": "st_123 or null",
  "policy_snapshot_hash": "sha256:... or null",
  "idempotency_key": "idem_123",
  "payload": {}
}
```

说明：

- `policy_snapshot_hash = null` 只允许出现在身份绑定前的接入规范化接口。
- 一旦进入 Workflow 规划、检索、执行、写入与恢复链路，`policy_snapshot_hash` 必须存在。

### 15.2.2 响应信封

```json
{
  "request_id": "req_123",
  "trace_id": "tr_123",
  "ok": true,
  "error": null,
  "payload": {},
  "meta": {
    "duration_ms": 123,
    "provider": "local"
  }
}
```

### 15.2.3 事件信封

```json
{
  "event_id": "evt_123",
  "trace_id": "tr_123",
  "event_type": "workflow.stage.started",
  "aggregate_type": "workflow_stage",
  "aggregate_id": "st_123",
  "workflow_instance_id": "wf_123",
  "workflow_stage_id": "st_123",
  "occurred_at": "2026-04-20T10:00:00Z",
  "payload": {}
}
```

## 15.3 错误码规范

错误码分层：

- `CHANNEL_*`：渠道接入错误
- `IDENTITY_*`：身份绑定错误
- `POLICY_*`：权限与策略错误
- `WORKFLOW_*`：规划与状态机错误
- `STAGE_*`：阶段执行错误
- `EXECUTOR_*`：执行器错误
- `RETRIEVAL_*`：检索错误
- `FACT_*`：事实写入错误
- `ARTIFACT_*`：对象存储与制品错误
- `INTEGRATION_*`：外部系统错误
- `SYSTEM_*`：系统级错误

通用约束：

1. 同一个失败只能有一个主错误码。
2. 附加上下文放在 `error.detail`。
3. 需要重试时必须设置 `retryable: true`。

错误体建议：

```json
{
  "code": "POLICY_DENIED",
  "message": "policy snapshot does not allow cross-user read",
  "retryable": false,
  "detail": {
    "resource_type": "fact",
    "resource_scope": "private:u_b"
  }
}
```

## 15.4 核心同步接口

### 15.4.1 Channel Ingress API

用途：把渠道侧消息标准化为平台请求。

推荐接口：`POST /internal/channel-ingress/normalize`

请求 `payload`：

```json
{
  "raw_message": {},
  "channel_identity": "external_user_or_chat",
  "attachments": [],
  "session_hint": {}
}
```

返回 `payload`：

```json
{
  "user_id": "u_123 or null",
  "channel_type": "feishu",
  "request_text": "...",
  "attachments": [],
  "session_ref": "sess_123",
  "identity_binding_state": "bound|pending|conflicted",
  "binding_action": "continue|binding_required|admin_resolution_required"
}
```

说明：

- 若 `identity_binding_state != bound`，请求不得直接进入 Workflow 规划。

### 15.4.2 Workflow Planning API

用途：把标准化请求转为 Workflow 实例。

推荐接口：`POST /internal/workflows/plan`

> **⚠️ 一致性说明**: 本接口定义以 [文档17](./AH1-17-Workflow-DSL与Planner契约.md) 为权威基准。Workflow DSL 定义了完整的请求/响应结构。

请求 `payload`：

```json
{
  "user_goal": "...",
  "user_id": "u_123",
  "channel_type": "web_portal",
  "policy_snapshot_hash": "sha256:...",
  "current_fact_refs": [],
  "visible_asset_refs": [],
  "task_type_hint": "development",
  "risk_level": "medium",
  "priority": "normal",
  "timeout_sec": 3600,
  "budget": {
    "retrieval": 20,
    "execution": 120,
    "time_sec": 1800
  }
}
```

**字段说明**:

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `user_goal` | string | ✅ | - | 用户目标描述 |
| `user_id` | string | ✅ | - | 用户标识 |
| `channel_type` | enum | ✅ | - | 渠道类型（见文档21）|
| `policy_snapshot_hash` | string | ✅ | - | 权限策略快照哈希 |
| `current_fact_refs` | array | ❌ | [] | 当前已知 Fact 引用 |
| `visible_asset_refs` | array | ❌ | [] | 可见资产引用 |
| `task_type_hint` | enum | ❌ | "knowledge-query" | 任务类型提示 |
| `risk_level` | enum | ❌ | "medium" | 风险等级（low/medium/high）|
| **`priority`** | **enum** | **❌** | **"normal"** | **优先级（low/normal/high/critical）- 见文档17 §17.4** |
| **`timeout_sec`** | **integer** | **❌** | **3600** | **工作流超时时间（秒）- 见文档28 §28.3.2.8** |
| `budget` | object | ❌ | - | 预算约束 |

返回 `payload`：

```json
{
  "workflow_definition_ref": "wd_123",
  "workflow_instance_ref": "wf_123",
  "workflow_plan_hash": "sha256:...",
  "workflow_plan": {},
  "stage_plan": []
}
```

**状态码映射**:

| 状态码 | 场景 | 错误码 |
|--------|------|--------|
| 201 | Workflow 创建成功 | - |
| 400 | 请求参数无效 | `WORKFLOW_INVALID_REQUEST` |
| 401 | 未认证或权限不足 | `IDENTITY_UNAUTHENTICATED` / `POLICY_DENIED` |
| 409 | 并发限制超出 | `WORKFLOW_CONCURRENT_LIMIT` |
| 422 | Planner 无法解析目标 | `WORKFLOW_PLANNING_FAILED` |
| 503 | LLM 服务不可用 | `INTEGRATION_LLM_UNAVAILABLE` |

推荐接口：`POST /internal/workflows/{workflow_id}/stages/{stage_id}/dispatch`

## 15.5 核心异步事件

### 15.5.1 Workflow 事件

| 事件 | 触发时机 | 最小 payload |
|---|---|---|
| `workflow.created` | Workflow 实例创建完成 | `workflow_id`、`workflow_type`、`owner_user_id` |
| `workflow.state.changed` | Workflow 状态迁移 | `from_status`、`to_status`、`reason` |
| `workflow.report.required` | 进入主动汇报 | `workflow_id`、`report_policy` |
| `workflow.archived` | 归档完成 | `workflow_id`、`archive_ref` |

### 15.5.2 Stage 事件

| 事件 | 触发时机 | 最小 payload |
|---|---|---|
| `workflow.stage.started` | 阶段开始 | `stage_id`、`stage_type`、`executor` |
| `workflow.stage.progressed` | 阶段产生结构化进度 | `stage_id`、`progress_pct`、`summary` |
| `workflow.stage.checkpointed` | checkpoint 落盘 | `stage_id`、`checkpoint_id` |
| `workflow.stage.completed` | 阶段通过验收 | `stage_id`、`acceptance_result` |
| `workflow.stage.failed` | 阶段失败 | `stage_id`、`error_code` |

### 15.5.3 Retrieval / Fact / Artifact 事件

| 事件 | 触发时机 | 最小 payload |
|---|---|---|
| `retrieval.executed` | Evidence Pack 构建完成 | `retrieval_trace_id`、`evidence_pack_hash` |
| `fact.written` | 事实写入成功 | `fact_refs`、`write_mode` |
| `fact.conflict.detected` | 发现事实冲突 | `old_fact_id`、`new_fact_id` |
| `artifact.created` | artifact 持久化完成 | `artifact_id`、`artifact_type` |

### 15.5.4 Code Executor 事件

| 事件 | 触发时机 | 最小 payload |
|---|---|---|
| `code.session.created` | 执行会话创建 | `execution_session_id`、`backend_type` |
| `code.progress.reported` | 代码执行进度回写 | `execution_session_id`、`summary` |
| `code.verification.completed` | 验证完成 | `result`、`verification_refs` |
| `code.repair.exhausted` | 修复次数耗尽 | `failure_summary` |

## 15.6 幂等性与重试规则

1. 所有有副作用接口必须传 `idempotency_key`。
2. `plan`、`dispatch`、`fact write`、`artifact create`、`code session create` 必须支持幂等重放。
3. 若请求因外部 provider 超时失败但实际可能已执行，服务端必须优先查询既有 `idempotency_key` 结果，不可直接重跑。

## 15.7 契约测试建议

每个接口都应至少有以下测试：

1. 正常输入可返回最小成功结构。
2. 缺失 `policy_snapshot_hash` 时被拒绝。
3. `idempotency_key` 重放时返回同一结果。
4. 越权 scope 时返回 `POLICY_DENIED`。
5. 异步事件 payload 能通过 JSON schema 校验。

## 15.8 Day 1 最先冻结的 schema

优先冻结以下 schema：

- Channel Ingress 请求/响应
- Workflow Planning 请求/响应
- Retrieval 请求/响应
- Fact Write 请求/响应
- Code Executor Session 创建与回写 schema
- Workflow / Stage / Code Progress 事件 schema

原因：这些契约一旦稳定，开发可以并行推进 Gateway、Workflow、Retrieval、Executor、Audit 五条子链路。
