# 文档 25：研发里程碑与任务拆解计划 v1.0

## 25.1 文档目的

本文件把前述实施文档转换为可执行研发计划，输出：

- 里程碑分期
- 每期目标与产出
- 每期任务拆解
- 进入下一期的验收门槛

## 25.2 总体里程碑

| 里程碑 | 目标 |
|---|---|
| M0 | 文档冻结与实施基线审校 |
| M1 | 接入层 + Workflow 主链路 |
| M2 | 事实层 + 检索主链路 |
| M3 | Code Executor 集成 |
| M4 | Hermes memory / skill / dream 治理接入 |
| M5 | 容量验证、治理收口与上线准备 |

## 25.3 M0：文档冻结与实施基线审校

### 25.3.1 目标

- 统一术语、边界、对象语义与文档依赖关系。
- 确认 `public workflow`、`policy_snapshot`、`artifact/document`、`checkpoint/replay` 等核心概念无冲突。
- 形成开发前冻结清单。

### 25.3.2 任务拆解

1. 校对总纲与文档 13-32。
2. 冻结权威术语与关系图。
3. 修正跨文档冲突。
4. 形成开发前自动一致性校验报告。

### 25.3.3 验收

- 文档之间无显著语义冲突。
- 开发前冻结清单明确。
- 后续开发只在冻结基线上推进。

## 25.4 M1：接入层 + Workflow 主链路

### 25.4.1 目标

- 渠道请求能进入平台。
- 能创建 Workflow。
- 状态机与 checkpoint 最小闭环可跑通。

### 25.4.2 任务拆解

1. 落 `user`、`channel_identity`、`policy_snapshot`、`workflow_definition`、`workflow_instance`、`workflow_stage`、`checkpoint`、`workflow_event` 表。
2. 实现 Channel Ingress 标准化接口。
3. 实现 Identity Resolver。
4. 实现 Workflow Planner 最小版本。
5. 实现 Workflow 状态机与状态迁移校验。
6. 实现阶段进度回传。

### 25.4.3 验收

- P0-1、P0-2 通过。
- `draft` / `planned` / `running` / `waiting_user` / `blocked` / `paused` / `verifying` / `repairing` / `failed` / `succeeded` 可区分。
- checkpoint 可恢复。

## 25.5 M2：事实层 + 检索主链路

### 25.5.1 目标

- PostgreSQL 成为事实源。
- 检索链路可生成 Evidence Pack。
- Fact Write 与冲突记录可落地。

### 25.5.2 任务拆解

1. 落 `document*`、`entity*`、`relation`、`fact*`、`retrieval_trace`、`artifact_object` 表。
2. 实现结构化查询服务。
3. 实现全文检索。
4. 接入 pgvector。
5. 实现 Evidence Pack Builder。
6. 实现 Fact Write 与 Conflict Detector。

### 25.5.3 验收

- P0-3 通过。
- 检索 trace 完整。
- 零越权读取。

## 25.6 M3：Code Executor 集成

### 25.6.1 目标

- 开发类任务可以进入 Code Executor。
- patch、测试结果、修复循环、checkpoint 能回写。

### 25.6.2 任务拆解

1. 落 `execution_session` 表。
2. 实现 Code Executor Session API。
3. 接入主 backend。
4. 实现 patch artifact、test result artifact、verification artifact。
5. 实现修复循环与上限控制。
6. 实现执行恢复流程。

### 25.6.3 验收

- P0-5 通过。
- 开发任务可跑通计划-实现-验证-修复闭环。

## 25.7 M4：Hermes 增强接入

### 25.7.1 目标

- memory / skill / dream 以平台治理方式进入系统。
- 不破坏事实真相源与权限模型。

### 25.7.2 任务拆解

1. 落 `memory_item`、`skill*` 表。
2. 实现 Hermes Adapter。
3. 实现 Memory Policy Guard。
4. 实现 Skill 草稿与公共发布治理。
5. 实现 DreamSummarization 与 SkillExtraction 后处理链。

### 25.7.3 验收

- P0-6 通过。
- Hermes 不能直接写主事实。
- 公共 skill 发布链路可审计。

## 25.8 M5：容量验证与治理收口

### 25.8.1 目标

- 20 运行态 Workflow 目标完成验证。
- 审计、日志、告警、限流完善。

### 25.8.2 任务拆解

1. 完成全部指标埋点与 dashboard。
2. 补齐告警策略。
3. 执行并发压测、恢复压测、检索压测。
4. 完成 public 发布治理与 admin 操作台。
5. 形成上线前 runbook。

### 25.8.3 验收

- 容量、恢复、权限、审计目标通过。
- 上线 runbook 完成。

## 25.9 跨里程碑工作流

持续工作流如下：

1. 每个里程碑先冻结契约与 migration。
2. 再开发服务与 adapter。
3. 再跑 PoC 与 contract test。
4. 最后补 dashboard、审计与 runbook。

## 25.10 优先级拆解

### 25.10.1 P0

- Workflow 状态机
- checkpoint / resume
- PostgreSQL 主表
- Retrieval 主链路
- Code Executor 最小集成
- 审计主链路

### 25.10.2 P1

- AGE 图增强
- Hermes memory 接入
- 公共区发布治理

### 25.10.3 P2

- 更复杂对象抽取
- 更强自动验证
- 更细粒度性能优化

## 25.11 交付顺序建议

1. 先完成 M1 + M2，形成"接入 -> Workflow -> Retrieval -> Fact" 主链路。
2. 再完成 M3，打通开发任务。
3. 再做 M4，把 memory / skill / dream 纳入治理。
4. 最后以 M5 收口性能、审计、告警、上线准备。

前置约束：

- 所有开发工作都必须在 M0 通过后开始。
