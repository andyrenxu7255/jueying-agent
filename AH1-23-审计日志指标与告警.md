# 文档 23：审计、日志、指标与告警设计 v1.0

## 23.1 文档目的

本文件定义平台可观测性与治理留痕的最小闭环，覆盖：

- 审计事件
- 结构化日志
- 指标体系
- 告警阈值
- Dashboard 视图

## 23.2 四层观测模型

1. 审计层：回答“谁在何时对什么做了什么”。
2. 日志层：回答“系统当时发生了什么”。
3. 指标层：回答“系统整体是否健康”。
4. 告警层：回答“何时需要人工介入”。

## 23.3 Trace 上下文

所有组件都应传递：

- `trace_id`
- `request_id`
- `workflow_instance_id`
- `workflow_stage_id`
- `execution_session_id` 可选
- `user_id`
- `policy_snapshot_hash`

## 23.5 结构化日志规则

### 23.5.1 日志字段

- `timestamp`
- `level`
- `service`
- `trace_id`
- `workflow_instance_id`
- `workflow_stage_id`
- `event`
- `message`
- `detail`

### 23.5.2 日志级别建议

- `DEBUG`：仅开发调试
- `INFO`：正常阶段流转、事件接收、健康状态
- `WARN`：降级、重试、轻微异常
- `ERROR`：阶段失败、外部依赖失败、恢复失败

## 23.7 告警建议

| 告警 | 条件 |
|---|---|
| Workflow 启动异常 | `workflow_start_latency_ms p95 > 3000ms` |
| Stage 长时间卡住 | 某阶段超过 `hard_timeout_sec` 未结束 |
| 检索退化异常 | `retrieval_degraded_total` 短时间内激增 |
| 权限拒绝异常 | `policy_denied_total` 激增，疑似误配置或越权攻击 |
| 审计失败 | `audit_write_fail_total > 0` |
| 执行器积压 | `execution_session_running_gauge` 持续满槽 |

## 23.8 Dashboard 建议

### 23.8.1 运营视图

- 今日 Workflow 总量
- 成功率
- 平均阶段耗时
- 当前运行态槽位占用

### 23.8.2 技术视图

- Retrieval p50/p95/p99
- Code Executor 成功率
- 修复循环次数分布
- Checkpoint 恢复成功率

### 23.8.3 安全治理视图

- 越权拒绝趋势
- admin 跨用户读取次数
- 公共发布次数
- 审计写入健康状况

## 23.9 保留策略建议

- 审计事件：至少 1 年
- 结构化运行日志：90 天热数据，之后冷归档
- 指标：按监控系统默认策略，可保留 30-180 天

## 23.10 Day 1 验证用例

1. 任意一次 Workflow 创建都能看到对应 audit、日志、指标。
2. 任意一次 `policy denied` 都能在审计和日志中检索到。
3. 检索降级时会打 `WARN` 且记录指标。
4. 审计写入失败能触发高优先级告警。
