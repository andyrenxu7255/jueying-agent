# 文档 21：渠道接入、身份绑定与 Session 映射设计 v1.0

## 21.1 文档目的

本文件细化 OpenClaw 接入层在本平台中的职责，明确：

- 渠道消息如何进入平台
- 外部身份如何绑定到内部 `user_id`
- `session_ref` 如何映射到 `workflow_instance`
- 什么时候复用会话，什么时候新建 Workflow

## 21.2 总体链路

标准链路如下：

1. 渠道消息进入 Channel Adapter。
2. Adapter 标准化消息体、附件与渠道上下文。
3. Identity Resolver 根据 `channel_identity` 查找或创建绑定。
4. Session Mapper 根据会话绑定策略生成或解析 `session_ref`。
5. Request Router 判断是继续现有 Workflow，还是创建新 Workflow。
6. OpenClaw 壳层把请求交给 Workflow 层，并持续回传进度与结果。

## 21.3 Channel Adapter 统一输出

每个 Channel Adapter 必须统一输出：

```json
{
  "channel_type": "wecom|feishu|web_portal",
  "channel_account_id": "acct_123",
  "channel_identity": "user_or_chat_id",
  "session_hint": {
    "conversation_id": "conv_123",
    "thread_id": "th_123"
  },
  "request_text": "...",
  "attachments": [],
  "raw_message_ref": "raw_123",
  "received_at": "2026-04-20T10:00:00Z"
}
```

## 21.4 身份绑定模型

### 21.4.1 绑定对象

身份绑定使用表：`channel_identity`

关键字段：

- `channel_type`
- `channel_identity`
- `user_id`
- `binding_status`

### 21.4.2 绑定状态

- `pending`
- `bound`
- `disabled`
- `conflicted`

### 21.4.3 绑定规则

1. 同一 `(channel_type, channel_identity)` 只能绑定一个 `user_id`。
2. 未绑定身份进入平台时，应先进入绑定前流程，不得直接创建 Workflow，也不得把未绑定请求伪装成 `waiting_user` Workflow。
3. 绑定冲突必须由 admin 处理。

### 21.4.4 绑定前流程

绑定前流程属于接入层，不属于 Workflow 运行态。

- 若身份未绑定：返回 `binding_required`
- 若身份冲突：返回 `admin_resolution_required`
- 只有身份被解析为确定 `user_id` 后，才允许生成 `policy_snapshot` 并进入 Workflow 规划

## 21.5 Session 模型

### 21.5.1 SessionRef 作用

`session_ref` 是渠道对话和平台 Workflow 之间的衔接层，用于：

- 识别消息属于哪个会话上下文
- 维持多轮对话连续性
- 决定是否复用当前 Workflow

### 21.5.2 推荐 SessionRef 组成

```json
{
  "session_ref": "sess_123",
  "channel_type": "feishu",
  "channel_account_id": "acct_123",
  "conversation_id": "conv_123",
  "thread_id": "th_123",
  "user_id": "u_123",
  "binding_key": "feishu:acct_123:conv_123:th_123"
}
```

### 21.5.3 Session Key 生成规则

Day 1 推荐规则：

- DM：`<channel_type>:<account_id>:dm:<channel_identity>`
- 群聊/频道：`<channel_type>:<account_id>:conv:<conversation_id>`
- 线程：在会话 key 后追加 `:thread:<thread_id>`

说明：

- 会话 key 只解决“对话连续性”。
- Workflow 是否复用还需要结合任务状态判断。

## 21.6 Session 到 Workflow 的映射规则

### 21.6.1 可复用现有 Workflow 的条件

只有同时满足以下条件，才允许复用：

1. `session_ref` 对应存在一个活动 Workflow。
2. 该 Workflow `owner_user_id` 与当前用户一致。
3. Workflow 状态属于：`running`、`waiting_user`、`paused`、`blocked`。
4. 当前用户消息被识别为“继续当前任务”而不是发起新任务。

### 21.6.2 必须新建 Workflow 的条件

1. 没有匹配的活动 Workflow。
2. 当前消息意图明显切换到新任务。
3. 原 Workflow 已 `succeeded`、`failed`、`cancelled` 或 `archived`。
4. 当前消息来自不同 `user_id`。

### 21.6.3 明确禁止的映射

- 禁止不同用户共享同一私有 Workflow。
- 禁止通过公共群聊 thread 意外续接到他人的私有开发任务。
- 禁止绕过 `session_ref` 直接靠文本相似度把消息挂到旧 Workflow。

## 21.7 渠道差异处理

### 21.7.1 企业微信

#### 接入协议

| 维度 | 说明 |
|------|------|
| 接入方式 | 企业微信回调 URL（Webhook） |
| 消息格式 | XML/JSON（企业微信 API v2） |
| 认证方式 | CorpSecret + Callback Token + EncodingAESKey |
| 事件类型 | 消息接收、事件通知（进入应用、审批状态等） |

#### 身份映射

| 企业微信字段 | 平台字段 | 说明 |
|-------------|----------|------|
| `FromUserName` | `channel_identity` | 企业微信 UserID |
| `AgentId` | `channel_account_id` | 应用 AgentId |
| `ChatId` | `conversation_id` | 群聊 ChatId |
| 无线程概念 | - | 企业微信不支持线程，用消息时间窗口模拟 |

#### Session Key 规则

- 单聊：`wecom:<corp_id>:dm:<user_id>`
- 群聊：`wecom:<corp_id>:conv:<chat_id>`
- 无线程支持：5 分钟内同一 session 的连续消息视为同一对话

#### 附件处理

| 附件类型 | 处理方式 |
|----------|----------|
| 图片 | 下载到 MinIO，返回 URL 引用 |
| 文件 | 下载到 MinIO，返回 URL 引用 |
| 链接 | 提取 URL，不下载 |
| 位置/名片 | 转为文本描述 |

#### 回传格式

- 文本消息：直接回复
- 长文本：截断 + 链接到 Web Portal
- 文件/报告：上传到企业微信素材库后发送

### 21.7.2 飞书

#### 接入协议

| 维度 | 说明 |
|------|------|
| 接入方式 | 飞书事件订阅（Event Subscription） |
| 消息格式 | JSON（飞书开放平台 API v2） |
| 认证方式 | App ID + App Secret + Verification Token + Encrypt Key |
| 事件类型 | 消息接收、机器人进群、审批等 |

#### 身份映射

| 飞书字段 | 平台字段 | 说明 |
|----------|----------|------|
| `event.sender.sender_id.open_id` | `channel_identity` | 飞书 Open ID |
| `header.app_id` | `channel_account_id` | 应用 App ID |
| `event.message.chat_id` | `conversation_id` | 群聊 Chat ID |
| `event.message.thread_id` | `thread_id` | 话题 Thread ID（飞书支持线程） |

#### Session Key 规则

- 单聊：`feishu:<app_id>:dm:<open_id>`
- 群聊：`feishu:<app_id>:conv:<chat_id>`
- 话题线程：`feishu:<app_id>:conv:<chat_id>:thread:<thread_id>`

#### 附件处理

| 附件类型 | 处理方式 |
|----------|----------|
| 图片 | 通过飞书 `im/v1/images` 接口下载到 MinIO |
| 文件 | 通过飞书 `im/v1/messages/{id}/resources` 下载到 MinIO |
| 富文本 | 提取纯文本 + 保留结构化标记 |
| 互动卡片 | 解析 action 后转为平台指令 |

#### 回传格式

- 文本消息：飞书文本消息
- 长文本：飞书富文本 + 折叠
- 进度更新：飞书互动卡片（可交互）
- 文件/报告：上传到飞书后发送文件消息

### 21.7.3 Web Portal

- 用户已登录，`user_id` 可直接确定。
- `session_ref` 由浏览器会话与当前页面任务上下文共同决定。

## 21.8 附件处理规则

1. 原始附件先以 `raw_attachment` 形式落 `artifact_object`。
2. 只有在需要进入长期检索或事实抽取时，才提升为 `document` / `document_version`。
3. Adapter 输出 `attachments` 只传引用，不传大文件正文。
4. 附件读取权限继承当前请求的 `policy_snapshot`。

## 21.9 回传规则

### 21.9.1 进度回传

OpenClaw 壳层负责把以下事件翻译为用户可见回执：

- 阶段开始
- 进入 `waiting_user`
- 进入 `blocked`
- 进入 `paused`
- 关键进度更新
- 最终结果

### 21.9.2 结果回传

结果回传必须包含：

- 当前 Workflow 状态
- 当前阶段摘要
- 是否需要用户动作
- 可选的 artifact / report ref

## 21.10 管理员操作

admin 可对任意 `session_ref` 关联的 Workflow 执行：

- 查看当前状态
- 暂停
- 恢复
- 取消
- 重新绑定 identity

所有管理员动作必须审计。

## 21.11 审计要求

以下动作必须写 `audit_event`：

- `identity.bound` — 新身份绑定
- `identity.conflict` — 身份绑定冲突
- `session.created` — session key 创建
- `session.workflow_routed` — session -> workflow 复用判断
- `admin.cross_user_session_read` — 跨用户 session 查询
- `channel.message_ingressed` — 渠道消息进入时间与路由结果

## 21.12 Day 1 验证用例

1. 同一用户在同一线程继续消息时，能够续接到原 Workflow。
2. 同一用户发起明显新任务时，系统会新建 Workflow。
3. 不同用户在同一公共群聊发消息时，不会误接到他人私有 Workflow。
4. 未绑定身份会进入绑定或等待流程，而不是直接落到匿名用户。
