# A. 总体判断

## A.1 当前方案最难点

1. **Workflow 状态机 + 自主规划循环**：这是平台核心，Planner 不是一次性输出而是持续决策循环，需要实现 16 种 stage_type、14 条状态迁移规则、自修复决策树、动态 Executor 选择、Subagent 机制。无上游可直接复用，必须全新构建。
2. **权限过滤先于检索**：所有检索必须先做 scope 过滤再召回，pgvector 的 HNSW 索引在强权限过滤下性能未验证，需要 PoC 先行。
3. **Code Executor 上下文隔离**：Code Executor 必须与 OpenClaw 主上下文完全隔离，独立上下文窗口管理、按需加载、压缩策略，实现复杂度高。
4. **Checkpoint/Resume 一致性**：恢复必须绑定原 policy_snapshot_hash，state_hash 计算规则必须稳定，幂等恢复不能产生重复写入。

  apps/
    gateway-adapter/          # 接入层：渠道消息标准化、身份解析、Session 映射

    contracts/                # 稳定 JSON schema、事件字典、错误码、DTO

    audit/                    # 审计写入、审计查询

  config/                     # 配置文件（default.yaml, production.yaml 等）
  docker-compose.yml          # 本地开发环境
  Dockerfile                  # 生产构建

# G. 开发执行约束（Agent 必须遵守）

## G.1 冻结决策不可违反

以下 16 条冻结决策基线在任何情况下都不可违反：

1. 单组织、多用户，不做多租户 SaaS
2. PostgreSQL 是唯一事实源
3. pgvector 只负责语义候选召回
4. Apache AGE 只做图增强，不是第二事实源
5. 不使用 Neo4j
6. 只保留"个人区 + 公共区"
7. 普通用户默认严格隔离
8. Workflow 不依赖 md/heartbeat 管理状态
9. Code Executor 与 OpenClaw 上下文隔离
10. 每阶段必须可恢复
11. 检索必须权限过滤，不能绕过
12. Skill Day 1 默认私有
13. Memory 经验提示不可作为事实依据
14. Hermes 不能直接写平台主事实
15. Replay 默认只读，禁止重执行时修改
16. 预算必须可观测、可控制

## G.2 开发顺序约束

1. **先契约后实现**：每个里程碑先冻结契约与 migration，再开发服务与 adapter
2. **先测试后交付**：每个接口至少有 5 类测试（正常输入、缺失 policy_snapshot_hash、idempotency_key 重放、越权 scope、事件 schema 校验）
3. **先 P0 后 P1**：P0 功能（Workflow 状态机、checkpoint、PostgreSQL 主表、检索主链路、Code Executor 最小集成、审计主链路）必须先完成
4. **先权限后功能**：任何新功能都必须先补齐 scope/owner/policy_snapshot/audit，不得放到"后续再补"

## G.3 代码规范约束

1. **TypeScript 主栈**：所有 services/libs 使用 TypeScript + Node 20
2. **Python 适配层**：hermes-adapter 使用 Python 3.11
3. **XState 状态机**：Workflow 状态迁移必须使用 XState 定义，禁止手写 if-else 状态判断
4. **Vercel AI SDK**：LLM 流式调用必须使用 AI SDK 的 streamText/tool-call，禁止直接调用 OpenAI SDK
5. **LiteLLM 代理**：所有 LLM 请求必须通过 LiteLLM 代理，禁止绕过代理直接调用 Provider
6. **Drizzle ORM**：数据库查询必须使用 Drizzle ORM，禁止手写裸 SQL（迁移脚本除外）
7. **BullMQ 队列**：任务调度必须使用 BullMQ，禁止自研调度循环
8. **node-casbin**：权限判定必须使用 casbin，禁止手写权限 if-else
9. **OpenTelemetry**：所有遥测必须通过 OpenTelemetry SDK，禁止自研埋点框架
10. **UUID 主键**：统一使用 `gen_random_uuid()`
11. **timestamptz 时间**：统一使用 `timestamptz`
12. **text + check 约束**：状态和枚举使用 `text + check constraint`
13. **JSONB 扩展**：metadata/budget/constraints/trace 使用 `jsonb`
14. **向量维度 1536**：Day 1 固定 text-embedding-3-small 1536 维
15. **sha256 hash**：所有 hash 使用 sha256
16. **32KB 内联阈值**：小于 32KB 可内联数据库，大于必须外置对象存储
17. **不添加注释**：代码中不添加注释，除非被明确要求
18. **LiteLLM 兼容模式**：LiteLLM 配置必须启用 `compatibility: "strict"`，确保 token usage 格式与 Vercel AI SDK 兼容
19. **BullMQ Worker 编译**：Worker 文件必须编译为 `.js` 后运行，生产环境禁止使用 `ts-node` 直接运行 Worker
20. **TypeScript 版本**：统一使用 TypeScript 5.4+，启用 strict 模式
21. **AGE 查询隔离**：AGE 图查询通过原生 Cypher 执行，结果映射回 PostgreSQL 主键后再通过 Drizzle ORM 查询完整数据，禁止通过 Drizzle 直接查询 AGE

## G.4 数据库迁移约束

1. 迁移脚本按序编号：`001_init_extensions.sql`、`002_identity_policy.sql`...
2. 每个迁移脚本必须幂等（使用 `IF NOT EXISTS`）
3. 扩展安装顺序：pgcrypto -> vector -> pg_trgm -> age
4. 表创建顺序严格按文档 14 §14.10
5. Day 1 冻结实现选择不可在代码中临时改动（entity 不启用硬唯一约束、skill 不启用硬唯一约束、workflow_instance.scope_type 固定 private、artifact_object.scope_type 固定 private）

## G.5 接口契约约束

1. 所有有副作用接口必须传 `idempotency_key`
2. `policy_snapshot_hash = null` 只允许出现在身份绑定前
3. 一旦进入 Workflow，`policy_snapshot_hash` 必须存在
4. 错误体必须包含 code/message/retryable/detail
5. 事件信封必须包含 event_id/trace_id/event_type/aggregate_type/aggregate_id/workflow_instance_id/occurred_at

## G.6 测试约束

1. 每个接口至少 5 类测试（文档 15 §15.7）
2. 每个 PoC 至少输出环境说明/测试步骤/原始结果/指标截图/风险判断/是否建议进入下一里程碑
3. 权限测试：普通用户 A 不能读取用户 B 的任何私有资源
4. 审计测试：任意一次 Workflow 创建都能看到对应 audit/log/metric
