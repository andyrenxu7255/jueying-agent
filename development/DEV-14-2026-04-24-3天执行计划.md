## 0. 每天开工固定流程（新窗口第一条）

1. 读取本文件：`development/DEV-14-2026-04-24-3天执行计划.md`
2. 先检查 Docker daemon：
   - `docker info`
   - 若失败：启动 Docker Desktop，再等待 `docker info` 成功
3. 执行环境闸门：
   - `npm run preflight:audit`
   - `npm run infra:bootstrap:full`
   - `npm run smoke:eval`
4. 若任一失败：先修复失败项，再重复步骤 2-3，直到全通过。
5. 通过后再进入当天任务。

说明：

- `smoke:eval` 通过仅代表系统基线可用，不代表“真实模型对话能力”已就绪。

## Day 1（04-24）：外部渠道接入闭环（飞书/企微）

### 目标

- 飞书与企微各完成「收消息 -> 系统处理 -> 回消息」最小闭环。
- 消息统一归一化到 gateway ingress 契约。

### 任务清单

1. 渠道配置与验签
   - 飞书 webhook 验签/challenge 回包
   - 企微 webhook 验签/解密（如有）
2. 渠道消息归一化
   - 抽取：`channel_identity`、`channel_type`、`channel_account_id`、`conversation_id/thread_id`
   - 统一送入：`/internal/channel-ingress/normalize`
3. 最小回包
   - 回显文本（包含 session_ref）
4. 错误处理
   - 签名失败 -> 401
   - 格式异常 -> 400
   - 重复事件 -> 幂等处理

### 验收闸门

- 本地模拟 payload 闭环先通过，再接真实渠道回调
- 飞书模拟消息：返回 200 + 正确回包
- 企微模拟消息：返回 200 + 正确回包
- `npm run smoke:eval` 仍 PASS

### 交付物

- 渠道适配代码（gateway）
- 渠道配置说明（新增到 development 文档）
- 当日结果回写到 `development/DEV-13-联调与测评报告.md`

## Day 2（04-25）：模型配置 + 对话/任务分流 + Workflow/Executor 主链路补齐

### 目标

- 配置模型 API 后可真实对话。
- 用户输入可区分「普通对话」与「任务请求」。
- workflow 不再只返回占位产物，dispatch 能真实调用 executor 最小执行入口。

### 任务清单

1. 模型配置
   - 支持环境变量读取 API Key
   - 缺 key 时友好降级提示（不崩）
2. 对话路径
   - 渠道消息 -> LiteLLM -> 回渠道
3. 任务分流
   - 规则：命令式/目标式输入进入 workflow 路径
   - 普通闲聊走对话路径
4. Workflow/Executor 主链路补齐
   - workflow `plan` 生成最小非空 `stage_chain`
   - executor 增加最小可执行入口（接收 dispatch，返回执行状态）
   - workflow `dispatch` 真正调用 executor，而不是仅返回占位 run_ref
5. 审计与日志
   - 记录请求类型（chat/task）
   - 记录模型调用成功/失败

### 验收闸门

 - 飞书/企微对话各 3 轮成功（前提：已配置有效模型 key）
 - 任务意图输入可生成非空 `stage_chain` 的 workflow plan
 - dispatch 后 executor 返回真实接收结果
 - `npm run smoke:eval` PASS
 - `npm run preflight:audit` PASS

### 交付物

- 模型配置能力完善
- chat/task 路由策略落地
- workflow/executor 最小执行主链路落地
- 结果回写 `development/DEV-13-联调与测评报告.md`

## 自纠错机制（每天都执行）

1. 任一命令失败：记录失败命令、错误摘要、影响范围。
2. 先做最小修复，不做无关重构。
3. 修复后按顺序复测：
   - 失败点局部复测
   - `npm run smoke:eval`
   - `npm run preflight:audit`
4. 若同一问题重复 >= 2 次：
   - 写入当天“重复故障”段落，给出根因与永久修复动作。
5. 若发现计划顺序与真实系统基线不匹配：
   - 优先修订本文件，再继续执行，避免后续窗口重复沿错误计划推进。
