# 图谱结构本体论分析报告 — 2026-05-11

> 分析对象：`development/app-graph/object-relationship-graph.md`（v2.3，生成于 2026-05-06，更新于 2026-05-11 审计修复同步） + `development/context-graph.json`（v1.9）
> 分析视角：本体论（Ontology）视角——关注概念的本质分类、形式化依赖和语义完整性
> 核心问题：多跳依赖的判断是否存在结构性缺陷？
> **状态更新（2026-05-11）**：本报告生成同日，SYSTEM-AUDIT-2026-05-11（50项发现）已完成修复。审计修复主要涉及安全实施层面（认证、网络隔离、凭据管理），对本报告的本体论分析结论无实质性影响——因为本体论缺陷是概念建模层的问题，与安全加固是不同的关注维度。

## 一、结论先行

**是的，你的直觉完全正确。** 当前图谱存在严重的本体论结构性缺陷，导致多跳依赖无法被系统化推导。根本原因不是"图不够大"，而是**图的节点类型和边类型没有被形式化定义**，使得：

- 单跳依赖能查表确认
- 两跳及以上依赖需要人工脑算，无法被工具/代码自动推导
- 概念间的语义约束（如"不相交""基数""传递性"）只存在于人脑中，未被显式声明

下面从 8 个维度展开分析。

## 二、核心缺陷清单

### 缺陷 1：三个抽象层级被揉在一个本体里

**现状**：`object-relationship-graph.md` 的 §1 系统拓扑和 §3 数据库 ER 图共享同一个命名空间。

**问题**：这违反了本体论的基本分层原则。

| 层级 | 当前混入的内容 | 正确做法 |
|------|--------------|---------|
| **L-A 基础设施拓扑** | Docker 容器、Kafka、Redis、MinIO、SigNoz | 独立的部署拓扑图 |
| **L-B 业务概念本体** | 12 个 DO、Workflow 状态机、故事线 | 核心本体文件 |
| **L-C 数据模型** | 47 张表、字段、索引、外键 | Schema 文档（从 schema.ts 自动生成） |

**后果**：当有人问"如果我把 ClickHouse 换成 Elasticsearch，哪些概念会受影响？"时，答案散布在基础设施拓扑和数据模型之间，无法系统定位。

**修复**：拆分为三个文件/章节，明确命名空间隔离。数据模型层通过 `派生自` 关联到概念层，概念层通过 `部署于` 关联到基础设施层。

> **同步注记（2026-05-11 审计修复）**：同日完成的系统审计修复已实施 Docker 网络隔离（agent-harness-net + agent-harness-backend 双网络），这是 L-A 基础设施拓扑层的一次具体改善。但概念层（L-B）和数据模型层（L-C）的分层缺失仍是未解决的本体论问题。

### 缺陷 2：依赖关系有类型但没有传递闭包规则

**现状**：§2 领域对象间约束关系定义了 10 条边，使用了 3 种关系类型：

```
CONSTRAINED_BY  DO-01 → DO-02, DO-04 → DO-02, DO-06 → DO-02, DO-01 → DO-03, DO-03 → DO-12
DEPENDS_ON      DO-06 → DO-09, DO-04 → DO-05, DO-04 → DO-07, DO-01 → DO-05
OBSERVES        DO-08 → DO-01, DO-04, DO-06, DO-07
```

**问题**：

1. **没有声明传递性**：`CONSTRAINED_BY` 是否具有传递性？如果 DO-01 → DO-03 → DO-12，那么 DO-01 是否传递依赖于 DO-12？当前图中无法判断，因为 DO-01 和 DO-12 之间没有直接边。

2. **没有声明合成规则**：如果 `A DEPENDS_ON B` 且 `B CONSTRAINED_BY C`，那么 `A` 和 `C` 之间是什么关系？当前不存在推导规则。

3. **OBSERVES 方向模糊**：DO-08 (AuditLogMetrics) OBSERVES DO-01, DO-04, DO-06, DO-07。但 AuditLogMetrics 到底是一个独立概念还是从属概念？它观察（observe）这些对象意味着什么？是依赖、是监听、还是只是记录？没有形式化语义。

**具体反例**：如果修改 `ApiContract`（DO-03）的信封格式，会影响谁？
- 直接：DO-01 (WorkflowLifecycle) CONSTRAINED_BY DO-03 ✓
- 传递：DO-01 → DO-03 → DO-12 (ApiVersionLifecycle) — 需要修改版本兼容策略 ✓
- 遗漏：DO-08 (AuditLogMetrics) — 审计日志记录的 action 字段可能包含信封信息，也需要同步更新 ✗（当前图谱无法推导）
- 遗漏：所有使用 `source_to_authority` 映射中关联了 AH1-15 的代码文件 ✗

**修复**：
1. 为每种关系类型声明属性：`isTransitive: boolean`, `isSymmetric: boolean`, `isReflexive: boolean`
2. 定义合成规则：`R1(A,B) ∧ R2(B,C) → R3(A,C)` 的显式推导链
3. 添加影响分析传播算法描述

### 缺陷 3：核心领域对象（DO）不完整

**现状**：12 个 DO 试图覆盖系统核心概念，但缺失了多个关键概念。

**缺失的重要概念**：

| 缺失概念 | 影响 | 应该关联的 DO |
|---------|------|-------------|
| **EvidencePack**（证据包） | §5.5 故事线中 retrieval-aware-executor 的核心产出物，但不在 DO 列表中 | DO-06 (RetrievalFactWrite) 的子概念 |
| **ExecutionContext**（执行上下文） | 每个 Executor 接收的 `{workflow_instance_id, stage, user_goal, context, policy_hash}` 是一个独立的概念实体 | DO-01 (WorkflowLifecycle) 与 DO-04 (ExecutionSession) 的交集 |
| **Heartbeat**（心跳） | §5.3 中 supervisor.recordHeartbeat() 是执行器阶段的核心状态追踪机制 | DO-01 的子概念 |
| **SkillCandidate**（技能候选） | Dream 模式中从交互模式检测生成的技能候选体，是 `scene_value_assessment` 的前置概念 | DO-01 (WorkflowLifecycle) 的衍生 |
| **RetentionPolicy**（保留策略） | memory_item 的 TTL 过期、audit_event 的归档策略 | 独立的生命周期管理 DO |
| **Notification/EventBus**（事件总线） | workflow_event + service_status_event 暗示了一个内部事件驱动机制 | 独立的通信 DO |

**后果**：概念缺失导致：
- 无法回答"EvidencePack 的生命周期是什么？"
- 无法回答"如果去掉 Heartbeat 机制，哪些功能会失效？"
- 新成员理解系统时需要从代码反向推导概念

**修复**：扩充到至少 18-20 个 DO，确保每个独立的业务概念都有权威定义。

### 缺陷 4：Is-a / Part-of / Instance-of 关系缺失

**现状**：图中只使用了 `CONSTRAINED_BY`、`DEPENDS_ON`、`OBSERVES` 三种关系。

**本体论分析**：概念之间有三种基本的本体论关系，当前全部缺失：

**Is-a（分类关系/继承）**：
- `CheckpointResumeReplay`（DO-05）和 `ExecutionSession`（DO-04）之间是什么关系？
  - Checkpoint 是 ExecutionSession 的一个子阶段？还是独立的概念？
  - 从代码看：`execution_session.checkpoint_id` → checkpoint 是 session 的一部分（Part-of）
  - 但 DO-05 的描述中 checkpoint 有独立的生命周期（可以被 resume）
  - **当前图谱无法回答这个问题**

**Part-of（组成关系）**：
- `EvidencePack` 是 `RetrievalFactWrite`（DO-06）的一部分吗？
- `AuditEvent` 属于哪个 DO？是 DO-08 的一部分还是跨 DO 的横切关注？
- `WorkflowStage` 是 `WorkflowLifecycle`（DO-01）的一部分吗？
  - 从 ER 图看：workflow_instance 1 → N workflow_stage，是 Part-of
  - 但从 DO 定义看，DO-01 描述的是 "WorkflowPlan → stage_chain → 状态机"，stage 是 lifecycle 的组成部分
  - **这些应该有明确的 Part-of 声明**

**Instance-of（实例化关系）**：
- `workflow_instance` 是 `workflow_definition` 的实例 — 这个在 ER 图暗示了（§3.2 中 1:N 关系），但在 DO 层面没有声明
- `skill_version` 是 `skill` 的实例 — 有 ER 图暗示，但 DO 层面没有
- **需要一个 Instance-of 矩阵来明确映射**

**修复**：为每个 DO-Do 之间的关系标注 `Is-a / Part-of / Instance-of / Constrains / Depends-on / Observes` 中的一个或多个。

### 缺陷 5：context-graph.json 的 authority_map 粒度不足

**现状**：
```json
"skill": "AH1-17-Workflow-DSL与Planner契约.md",
"dream": "AH1-17-Workflow-DSL与Planner契约.md",
```

**问题**：skill 和 dream 都映射到同一个权威文档（AH1-17），但它们的概念本质完全不同：

- **Skill**：可复用的能力单元，有完整的生命周期（创建→版本→审核→发布→评分→退役），涉及 tables: skill, skill_version, skill_source, skill_audit_record, skill_usage_stats, org_skill_registry, scene_value_assessment
- **Dream**：离线批处理模式，是一个 execution mode 而非独立概念，涉及 tables: dream_mode_config, memory_analysis_run, org_memory_summary

将两者映射到同一个权威文档，意味着：
1. 当有人问"skill 的审核流程是什么？"时，被导向 AH1-17 的 Workflow DSL 章节——但审核流程的细节实际分散在 AH1-16（权限）和代码实现中
2. 当 dream 模式需要修改时，权威文档指向不明确

**其他 authority_map 问题**：
```json
"resource": "AH1-26-Provider选择与配置.md"
```
"resource" 这个 key 太模糊——是指计算资源（CPU/GPU）、配额资源（resource_quota 表）、还是 Artifact 存储资源？应该拆分为：
- `compute_resource` → AH1-27（部署与运维）
- `quota_resource` → AH1-16（权限配额部分）
- `artifact_storage` → AH1-22

**修复**：
1. 对 authority_map 的每个 key 给出明确的概念定义
2. 允许一个概念映射到多个权威文档（多对一）
3. 将 `skill` 拆分出独立的权威文档映射
4. 将 `resource` 拆分为多个精确概念

### 缺陷 6：故事线（§5）与 DO 定义（§2）之间的引用断裂

**现状**：§5 有四条核心故事线，§2 有 12 个 DO，但两者之间的交叉引用几乎不存在。

**引用矩阵**（当前实际存在的引用）：

| 故事线 | 引用的 DO | 应该引用的 DO |
|--------|----------|-------------|
| §5.1 Chat 消息全链路 | DO-01, DO-02, DO-10 | DO-01, DO-02, DO-03, DO-08, DO-10, DO-11 |
| §5.2 Task 消息全链路 | DO-01, DO-02 | DO-01, DO-02, DO-04, DO-05, DO-08, DO-11 |
| §5.3 执行器自动执行 | DO-01, DO-04, DO-06, DO-07 | DO-01, DO-04, DO-05, DO-06, DO-07, DO-08, DO-11 |
| §5.4 自修复决策树 | 无 | DO-01, DO-05, DO-11 |

**断裂分析**：
- **DO-08 (AuditLogMetrics)**：在故事线中完全不出现，但实际每个步骤都产生审计事件。这是横切关注点在本体中被遗漏。
- **DO-11 (ErrorDegradePolicy)**：故事线 §5.4 的自修复决策树本质上就是错误降级策略的实现，但从未引用 DO-11。
- **DO-05 (CheckpointResumeReplay)**：只在 §5.2 的边缘提及（resume_token），但 checkpoint 机制贯穿整个执行流程。

**修复**：为每条故事线的每个步骤标注涉及的 DO，形成故事线-领域对象的交叉引用矩阵。

### 缺陷 7：读写矩阵（§6.2）与服务调用矩阵（§6.1）断层

**现状**：两个矩阵独立存在，没有建立关联。

**问题示例**：

当服务 A 通过 HTTP 调用服务 B 时，数据如何在两个矩阵之间流转？

```
§6.1: executor-gateway → fact-retrieval → POST /internal/retrieval/query
§6.2: executor-gateway 对 document_chunk 有 R 权限
       fact-retrieval 对 document_chunk 有 R/W 权限
```

但这两个矩阵之间的关系是隐式的：
- executor-gateway 调用 fact-retrieval 时，传递了什么参数？
- fact-retrieval 返回的 EvidencePack 包含哪些字段？
- 这些字段对应 document_chunk 的哪些列？

**更严重的断层**：web-portal 直接查询 PostgreSQL（§6.2 中标记了 R/W），但 §6.1 中没有 web-portal → PostgreSQL 的服务调用（因为它是通过共享库 `libs/shared/src/db` 的 ORM 连接）。这意味着：
- §6.1 矩阵只能反映 HTTP 服务调用
- §6.2 矩阵反映实际的数据访问（含直接 DB 连接）
- 两者之间有一个"直接数据库访问"的暗区。这不是安全缺陷（`libs/shared` 作为共享库抽象了数据库访问），但造成影响分析的不透明性——当修改 DB schema 时，从 §6.1 无法判断哪些服务受影响。

**修复**：
1. 在两个矩阵之间添加数据流箭头
2. 为每个 HTTP 端点标注"输入涉及的表"和"输出涉及的表"
3. 将直接数据库访问路径也纳入 §6.1 矩阵

### 缺陷 8：概念的多态性未被建模

**问题**：多个概念在不同上下文中扮演不同角色，但图谱只建模了一种角色。

**反例 1：`workflow_event` 表的双重角色**
- 作为 **审计追踪**：记录工作流状态变迁（对应 DO-08）
- 作为 **状态机驱动**：状态迁移的事件触发器（对应 DO-01）
- 当前图谱将 `workflow_event` 放在 §3.2 工作流治理模块中，仅与 DO-01 关联

**反例 2：`memory_item` 的多面性**
- 作为 **长期记忆**：被 hermes-adapter 管理，对应 DO 的记忆模块
- 作为 **上下文片段**：被 executor-gateway 在执行时检索，对应 DO-06 (RetrievalFactWrite)
- 作为 **反馈输入**：用户的交互修正会更新 memory_item，对应 Dream 模式
- 当前图谱只将其放在 §3.5 记忆与技能模块

**修复**：引入概念的多角色标注（Stereotype），或者使用 UML 的多重分类（Powertype）来建模。

## 三、多跳依赖判断的具体反例

### 反例场景：修改 `fact.confidence` 字段类型（real → numeric）

**一阶影响（当前图谱可查）**：
- `fact_evidence`（外键引用 fact）
- `fact_conflict`（外键引用 fact）

**二阶影响（当前图谱不可推导，需要人工分析）**：
- `memory_item.confidence` 也用 `real` 类型，语义上是否要同步修改？
- `scene_value_assessment.value_score` 也是 `real` 类型，是否有关联？
- `resource_quota` 的相关 numeric 字段是否需要统一精度？

**三阶影响（完全不可推导）**：
- `fact_evidence.excerpt` 存储的是文本证据摘要，如果 confidence 改为更高精度，证据的排序逻辑是否需要调整？
- `org_memory_summary.confidence` 的默认值 0.5 是否需要同步调整？
- 前端（web-portal）在展示 fact 时对 confidence 的显示格式是否需要变更？
- `workflow_instance.budget_json` 中可能包含与 confidence 相关的预算参数

**当前要回答这个问题需要**：
1. 扫描全部 47 张表的字段类型
2. 阅读所有涉及 confidence 的代码
3. 查看前端展示逻辑
4. 检查配置文件中的相关阈值

**正确做法**：本体中应该有：
```
confidence :: real[0.0, 1.0]    -- 在 Ontology 层面定义
  ↳ 出现在: fact, memory_item, entity_attribute, scene_value_assessment, org_memory_summary
  ↳ 影响: 排序算法、阈值判断、前端显示、审计规则
  ↳ 约束: 精度变更需要同步评估所有出现位置
```

## 四、优化方案

### 4.1 结构优化

**4.1.1 拆分本体文件**

```
development/ontology/
├── 00-domain-ontology.md        # 核心概念本体（纯业务概念，无实现细节）
├── 01-data-model.md             # 数据模型（从 schema.ts 自动生成或同步）
├── 02-service-topology.md       # 服务拓扑（从 docker-compose.yml 提取）
├── 03-infrastructure.md         # 基础设施（Docker/K8s/网络）
├── 04-dependency-closure.md     # 依赖闭包推导规则

```

每种关系类型需要声明元属性：

| 关系 | 传递性 | 对称性 | 含义 |
|------|--------|--------|------|
| Is-a | ✅ | ❌ | 分类继承 |
| Part-of | ❌ | ❌ | 组成关系 |
| Instance-of | ❌ | ❌ | 实例化 |
| Constrains-by | ✅ | ❌ | 约束依赖 |
| Depends-on | ✅ | ❌ | 运行时依赖 |
| Observes | ❌ | ❌ | 横切观察 |
| Implements | ❌ | ❌ | 接口实现 |
