### RED-C003: 5个意图分类函数可合并

**文件**: [apps/gateway-adapter/src/index.ts:L194-L365](file:///d:/teamclaw/agent-harness/apps/gateway-adapter/src/index.ts)

**问题描述**：意图分类分散在5个函数中：

| 函数 | 行数 | 分类结果 |
|------|------|----------|
| `isTaskIntent()` | L194-L203 | boolean |
| `isTaskDispatchIntent()` | L205-L216 | boolean |
| `isKnowledgeSubmitIntent()` | L220-L231 | boolean |
| `isQuickLookupIntent()` | L235-L246 | boolean |
| `classifyIntentWithLLM()` | L256-L365 | IntentClassification |

前4个是纯规则匹配，第5个是LLM分类。规则匹配先执行，LLM再覆盖。但规则匹配结果也需要LLM二次确认，造成双重判断。

**建议**：合并为单一 `IntentClassifier`

```typescript
class IntentClassifier {
  // 第一层：快速规则匹配（<1ms）
  quickRuleMatch(text: string): IntentType[]
  
  // 第二层：需要时才调用LLM
  async llmClassify(text: string, candidates: IntentType[]): Promise<IntentClassification>
  
  // 统一入口
  async classify(text: string): Promise<IntentClassification> {
    const candidates = this.quickRuleMatch(text)
    if (candidates.length === 1 && this.highConfidence(candidates[0])) {
      return this.buildFallback(candidates[0], 0.9) // 跳过LLM
    }
    return this.llmClassify(text, candidates)
  }
}
```

**收益**：
- 减少 ~100行重复逻辑
- 高置信规则匹配跳过LLM（估计30%请求），降低延迟200-500ms
- 降低LLM调用成本

身份检查 + 双向记忆写入出现在5个地方，共约40行重复模板。

    return { replyText: '身份尚未绑定...', modelCallOk: false }

// 路由注册

router.on('task', authenticated, withQuotaCheck, withMemoryPersistence, handleTask)

export function createMicroService(config: ServiceConfig): MicroService {

  const logger = createLogger(name)

  setupGracefulShutdown(server, logger)
  setupMetricsAggregation(logger)

  name: 'gateway-adapter',
  port: 3000,

## 五、性能瓶颈

### PERF-C001: gateway-adapter 单文件2230行阻塞事件循环

**问题**：gateway-adapter 中 LLM 意图分类（`classifyIntentWithLLM`）在主请求处理中同步等待，超时60秒。高并发下会阻塞后续请求。

**当前流程**：
```
请求 → 读取body → 意图分类(LLM, 可耗时500ms-60s) → 路由处理 → 回复
```

**建议**：意图分类异步化，请求先返回"处理中"

```
请求 → 快速规则匹配(<1ms) → 立即回复"已收到" → 异步LLM分类 → 异步路由处理
```

### PERF-C002: 内存存储无过期清理

**问题发现**：

| 服务 | 存储 | 过期策略 |
|------|------|----------|
| gateway-adapter | `gatewayState.dedupeCache` | 有TTL |
| gateway-adapter | `gatewayState.tokenCache` | 有TTL |
| **hermes-adapter** | `memoryStore` (Map) | **无TTL** |
| **web-portal** | `sessionStore` (Map) | 有TTL但无主动清理 |
| **mobile-app** | `deviceStore` (Map) | **无TTL** |
| **mobile-app** | `notificationStore` (Map) | 有MAX限制 |

hermes-adapter的`memoryStore`和mobile-app的`deviceStore`无TTL机制，长时间运行会内存泄漏。

**建议**：在 `@agent-harness/shared` 提供 `TTLMap<K,V>`

```typescript
// libs/shared/src/utils/ttl-map.ts
export class TTLMap<K, V> extends Map<K, { value: V; expiresAt: number }> {
  constructor(private defaultTTL: number) { super() }
  set(key: K, value: V, ttl?: number): this { ... }
  get(key: K): V | undefined {
    const entry = super.get(key)
    if (!entry) return undefined
    if (Date.now() > entry.expiresAt) { this.delete(key); return undefined }
    return entry.value
  }
}
```
