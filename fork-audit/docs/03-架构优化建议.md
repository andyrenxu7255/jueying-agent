## 一、当前架构评估

### 1.1 七层架构评估

当前架构分层：

```
L1: 接入与总控层 (gateway-adapter, feishu-longconn)
L2: Workflow治理层 (workflow-service)
L3: Executor层 (executor-gateway + 6种执行器)
L4: Fact/Retrieval层 (fact-retrieval)
L5: Memory/Skill层 (hermes-adapter, skill-library)
L6: Provider层 (LiteLLM)
L7: Infrastructure层 (PostgreSQL, Redis, MinIO, SigNoz)
```

**评估结论**: 架构设计优秀，分层清晰，职责边界明确。

| 评估维度 | 评分 | 说明 |
|----------|------|------|
| 合理性 | 9/10 | 七层架构职责清晰 |
| 可扩展性 | 8/10 | 插件化渠道、动态Executor选择 |
| 性能表现 | 7/10 | AGE+pgvector叠加查询需优化 |
| 安全性 | 8/10 | 16条冻结决策基线完善 |

## 二、架构优化建议

### 2.1 高优先级优化

#### OPT-001: 梦境模式数据库表精简

**当前状态**: 梦境模式新增9张表

| 表名 | 用途 | 可合并性 |
|------|------|----------|
| dream_mode_config | 配置 | 可合并到org_policy |
| memory_analysis_run | 分析运行 | 保留 |
| org_memory_summary | 记忆摘要 | 保留 |
| memory_access_log | 访问日志 | 可合并到audit_event |
| memory_compression_log | 压缩日志 | 可合并到memory_analysis_run |
| scene_value_assessment | 场景评估 | 可合并到skill_usage_stats |
| skill_audit_record | 技能审核 | 保留 |
| skill_usage_stats | 使用统计 | 保留 |
| org_skill_registry | 技能注册 | 可合并到skill表 |

**建议**: 9张表合并为4张核心表

```
保留:
- memory_analysis_run (分析运行)
- org_memory_summary (记忆摘要)
- skill_audit_record (技能审核)
- skill_usage_stats (使用统计)

合并:
- dream_mode_config → org_policy.metadata
- memory_access_log → audit_event
- memory_compression_log → memory_analysis_run.metadata
- scene_value_assessment → skill_usage_stats.metadata
- org_skill_registry → skill.scope_type='org'
```

**收益**: 减少5张表，降低维护成本，简化查询

---

#### OPT-002: 执行器类型精简

**当前状态**: 6种执行器

| 执行器 | 用途 | 使用频率 |
|--------|------|----------|
| generic-executor | 通用执行 | 高 |
| code-executor | 代码执行 | 中 |
| retrieval-aware-executor | 检索执行 | 高 |
| verification-executor | 验证执行 | 低 |
| repair-executor | 修复执行 | 低 |
| approval-executor | 审批执行 | 中 |

**问题**: verification/repair执行器使用频率低，可合并

**建议**:
- verification-executor合并到generic-executor
- repair-executor合并到code-executor
- 保留4种核心执行器

---

#### OPT-003: Workflow状态机简化

**当前状态**: 13种状态

```
draft → planned → running → verifying → reporting → succeeded → archived
                  ↓          ↓            ↓           ↓
              waiting_user  repairing    paused      failed
                  ↓          ↓
              blocked      paused
```

**问题**: verifying/repairing/reporting可合并为单一"processing"状态

**建议**: 简化为10种状态

```
draft → planned → running → processing → succeeded → archived
                  ↓          ↓            ↓
              waiting_user  paused      failed
                  ↓
              blocked
```

## 三、性能优化建议

### 3.1 数据库查询优化

#### PERF-001: pgvector查询优化

**问题**: 向量检索叠加权限过滤性能压力

**建议**:
- 添加复合索引: `(org_id, scope_type) + embedding`
- 使用分区表按org_id分区
- 实现查询结果缓存

---

#### PERF-002: AGE图查询优化

**问题**: 单机4C16G下AGE查询成本高

**建议**:
- 限制图查询深度max_hops≤2
- 实现图查询结果缓存
- 添加图投影事件批量处理

---

### 3.2 服务调用优化

#### PERF-003: 服务间调用链优化

**问题**: gateway→workflow→executor链路长

**建议**:
- 实现服务间调用缓存
- 添加请求合并机制
- 使用异步消息队列替代部分HTTP调用

---

### 3.3 内存管理优化

#### PERF-004: Hermes记忆压缩优化

**问题**: 4000字符阈值可能不够灵活

**建议**:
- 实现动态阈值（基于token预算）
- 添加压缩策略配置
- 实现分级压缩（轻度/中度/深度）

---

#### PERF-005: LLM调用优化

**问题**: 多次LLM调用延迟叠加

**建议**:
- 实现LLM调用结果缓存
- 添加批量调用支持
- 使用流式响应减少等待时间
