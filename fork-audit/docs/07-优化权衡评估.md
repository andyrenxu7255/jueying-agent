# 优化权衡评估报告

> **评估日期**: 2026-05-10 | **对比版本**: 原版 vs fork-audit-optimized

---

## 一、评估方法论

本报告采用 **"损失-收益"矩阵** 进行客观评估，逐项分析每个优化点的：
- **损失**: 功能缺失、灵活性降低、风险增加
- **收益**: 代码简化、性能提升、维护成本降低
- **净评估**: 损失是否可接受、收益是否值得

---

## 二、逐项权衡分析

### 2.1 ChannelAdapter 接口提取

**原版**: 飞书/企微 4对函数直接写在 gateway-adapter/index.ts 中

**优化版**: 提取 ChannelAdapter 接口 + FeishuAdapter + WecomAdapter

| 维度 | 损失 | 收益 |
|------|------|------|
| **功能** | 无损失 | 无变化 |
| **灵活性** | 原版可针对单个渠道微调（如飞书单独加日志） | 新增渠道需实现完整接口，无法"抄一半" |
| **代码量** | 无 | -200行重复代码 |
| **可维护性** | 修改需同步2处 | 修改接口即可 |
| **新增渠道成本** | ~150行（复制+微调） | ~30行（实现接口） |

**净评估**: ✅ **收益远大于损失**
- 损失的"灵活性"实际上是"不规范的灵活性"，容易导致不一致
- 新增渠道成本降低80%，这是长期收益

---

### 2.2 统一 DB 连接池管理

**原版**: 6个服务各自独立创建 Pool，总计最多54个连接

**优化版**: pool-manager 统一管理，按服务名分配

| 维度 | 损失 | 收益 |
|------|------|------|
| **功能** | 无损失 | 无变化 |
| **灵活性** | 各服务可独立配置 max 连接数 | 需通过 getPool(name, {max}) 参数传递 |
| **隔离性** | 服务间连接池完全隔离 | 共享同一 Pool 实例（但按 name 区分） |
| **DB连接数** | 最多54个 | 统一管理，可全局控制 |
| **代码量** | 无 | -100行重复代码 |

**潜在风险**:
- 原版各服务独立 Pool，一个服务连接泄漏不影响其他服务
- 优化版共享 Pool，理论上可能相互影响

**缓解措施**:
- pool-manager 按 name 区分，实际仍是独立 Pool 实例
- 可通过 `getPool('service-name', {max: 4})` 配置

**净评估**: ✅ **收益大于损失**
- "共享同一 Pool 实例"的担忧不成立，实际仍是独立实例
- DB连接数统一管理是运维收益

---

### 2.3 IntentClassifier 统一意图分类

**原版**: 5个独立函数 + classifyIntentWithLLM（100%调用LLM）

**优化版**: IntentClassifier 单一入口 + 两层策略（规则匹配→LLM）

| 维度 | 损失 | 收益 |
|------|------|------|
| **功能** | 无损失 | 无变化 |
| **LLM调用率** | 100% | ~70%（高置信规则跳过LLM） |
| **分类准确性** | LLM最终决定 | 规则匹配可能误判（但置信度<0.9时仍调用LLM） |
| **延迟** | 500ms-60s（LLM） | 高置信场景<1ms |
| **代码量** | 无 | -100行 |

**潜在风险**:
- 原版 LLM 分类有完整 prompt（~50行示例），优化版 prompt 简化
- 规则匹配可能漏掉边缘场景

**对比原版 prompt vs 优化版 prompt**:

| 内容 | 原版 | 优化版 |
|------|------|--------|
| 示例数量 | 12个 | 3个 |
| KEY DISTINCTION 说明 | 详细 | 简化 |
| task_dispatch vs task 区分 | 详细示例 | 简化 |

**缓解措施**:
- 优化版 prompt 可扩展，只需修改 classifier.ts 中的 systemPrompt
- 置信度阈值可调（当前0.9，可降低到0.85）

**净评估**: ⚠️ **收益大于损失，但需注意 prompt 简化**
- LLM调用率降低30%是显著收益
- prompt 简化可能导致边缘场景分类不准，建议后续补充示例

### 2.4 createMicroService 服务工厂

**原版**: 8个服务各自 ~30行 createServer/logger/health/shutdown 样板

**优化版**: createMicroService() 工厂函数

| 维度 | 损失 | 收益 |
|------|------|------|
| **功能** | 无损失 | 无变化 |
| **灵活性** | 各服务可自定义 server 配置 | 需通过 config 参数传递 |
| **中间件** | 各服务可自由添加中间件 | 需在 routes handler 中实现 |
| **代码量** | 无 | -240行样板 |
| **新增服务成本** | ~60行样板 | ~15行配置 |

**潜在风险**:
- 原版各服务可添加自定义中间件（如 cors、compression）
- 优化版 bootstrap.ts 未内置这些中间件

**缓解措施**:
- 可在 routes handler 中实现自定义逻辑
- 或扩展 createMicroService config 参数

**净评估**: ✅ **收益大于损失**
- 样板代码消除是显著收益
- 自定义中间件需求可通过扩展满足

---

### 2.5 Unified Executor 合并

**原版**: 6种执行器（generic/code/retrieval/approval/verification/repair）

**优化版**: unified-executor 单一入口 + 6种 mode

| 维度 | 损失 | 收益 |
|------|------|------|
| **功能** | 无损失 | 无变化 |
| **独立性** | 各执行器可独立部署/扩容 | 合并为单一服务 |
| **代码量** | 无 | -100行 |
| **LLM调用** | 3处 callLiteLLM 实现 | 1处 LlmClient |

**潜在风险**:
- 原版 verification-executor 和 repair-executor 可独立配置超时/重试
- 优化版统一配置，可能不适合所有场景

**缓解措施**:
- unified-executor 的 execute() 可接受 mode 参数
- 各 mode 的 timeout/maxTokens 可在 STAGES 中配置

**净评估**: ✅ **收益大于损失**
- verification/repair 实际使用频率低，合并合理
- 独立部署需求可通过 executor-gateway 多实例解决

---

### 2.6 TTLMap 内存存储

**原版**: hermes-adapter memoryStore 无 TTL，mobile-app deviceStore 无 TTL

**优化版**: TTLMap 带自动过期清理

| 维度 | 损失 | 收益 |
|------|------|------|
| **功能** | 无损失 | 无变化 |
| **内存安全** | 长时间运行内存泄漏 | 自动过期清理 |
| **性能** | 无额外开销 | 定期清理有轻微开销 |
| **数据持久性** | 数据永久保留（直到重启） | 数据按 TTL 过期 |

**潜在风险**:
- 原版 memoryStore 数据永久保留，可能用于长期分析
- 优化版 TTL 过期后数据丢失

**缓解措施**:
- TTL 可配置（默认24小时）
- 关键数据应持久化到 DB，而非依赖内存

**净评估**: ✅ **收益远大于损失**
- 内存泄漏是严重问题，TTLMap 是必要修复
- 长期数据应持久化，内存仅用于临时缓存

## 六、总结

### 6.1 损失-收益汇总

| 类别 | 损失 | 收益 | 净评估 |
|------|------|------|--------|
| ChannelAdapter | 微调灵活性 | -200行，新渠道-80%成本 | ✅ 收益远大于损失 |
| DB Pool | 配置灵活性 | -100行，连接数统一管理 | ✅ 收益大于损失 |
| IntentClassifier | prompt 详细度 | -100行，LLM调用-30% | ⚠️ 收益大于损失，需补充prompt |
| createMicroService | 中间件灵活性 | -240行，新服务-75%成本 | ✅ 收益大于损失 |
| Unified Executor | 独立部署灵活性 | -100行，LLM调用统一 | ✅ 收益大于损失 |
| TTLMap | 数据永久保留 | 内存泄漏修复 | ✅ 收益远大于损失 |

### 6.2 需补充项

| 优先级 | 功能 | 工作量 |
|--------|------|--------|
| 高 | logger/metrics 模块 | ~50行 |
| 高 | file-validator | ~30行 |
| 中 | IntentClassifier prompt 补充 | ~20行 |
| 中 | identityResolver 独立服务 | ~40行 |
| 低 | sessionMapper 独立服务 | ~20行 |

### 6.3 最终结论

**优化版本是值得采用的**，但需补充以下内容才能完全替代原版：

1. **日志/监控体系** - 原版 @agent-harness/shared 的 logger/metrics/alerts
2. **文件安全验证** - file-validator 的 validateFileForImport/sanitizeFileName
3. **IntentClassifier prompt** - 补充完整的分类示例

**预估补充工作量**: ~160行代码，约1-2天开发。
