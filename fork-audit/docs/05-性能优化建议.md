## 二、数据库性能优化

### 2.1 PostgreSQL优化

#### DB-001: 向量检索性能优化

**问题分析**:
- pgvector HNSW索引在权限过滤叠加时性能下降
- 当前索引: `embedding vector(1536)` 单列索引
- 查询需先过滤org_id/scope_type再向量检索

**优化方案**:

```sql
-- 方案A: 复合索引（推荐）
CREATE INDEX idx_document_chunk_org_scope_embedding 
ON document_chunk (org_id, scope_type) 
INCLUDE (embedding);

-- 方案B: 分区表
CREATE TABLE document_chunk_partitioned (
    ...
) PARTITION BY LIST (org_id);

-- 方案C: 查询缓存
-- 在Redis中缓存常用查询结果
```

**收益预估**: 向量检索性能提升50%

## 三、服务调用性能优化

### 3.1 HTTP调用优化

#### SVC-001: 服务间调用缓存

**问题分析**:
- gateway→workflow→executor调用链长
- 重复调用相同workflow状态查询

**优化方案**:

```typescript
// 在gateway中添加workflow状态缓存
const workflowStatusCache = new Map<string, {
  status: string,
  expiresAt: number
}>()

// 缓存有效期10秒
const CACHE_TTL = 10000

async function getWorkflowStatusWithCache(ref: string) {
  const cached = workflowStatusCache.get(ref)
  if (cached && cached.expiresAt > Date.now()) {
    return cached.status
  }
  const status = await fetchWorkflowStatus(ref)
  workflowStatusCache.set(ref, {
    status,
    expiresAt: Date.now() + CACHE_TTL
  })
  return status
}
```

## 四、内存管理优化

### 4.1 Hermes记忆优化

#### MEM-001: 动态压缩阈值

**问题分析**:
- 当前固定4000字符阈值
- 不同场景需要不同压缩策略

**优化方案**:

```typescript
// 动态阈值配置
interface CompressionConfig {
  lightThreshold: number    // 2000字符
  mediumThreshold: number   // 4000字符
  deepThreshold: number     // 8000字符
  tokenBudget: number       // 基于token预算动态调整
}

// 根据token预算选择压缩级别
function selectCompressionLevel(content: string, budget: number) {
  const len = content.length
  if (budget < 1000) return 'deep'
  if (len < config.lightThreshold) return 'none'
  if (len < config.mediumThreshold) return 'light'
  return 'medium'
}
```

## 五、资源调度优化

### 5.1 并发控制优化

#### RES-001: Workflow并发限制

**问题分析**:
- 当前无Workflow并发限制
- 可能导致资源耗尽

**优化方案**:

```typescript
// 全局Workflow并发限制
const MAX_CONCURRENT_WORKFLOWS = 15

// 使用信号量控制
const workflowSemaphore = new Semaphore(MAX_CONCURRENT_WORKFLOWS)

async function executeWorkflow(input: WorkflowInput) {
  await workflowSemaphore.acquire()
  try {
    return await doExecuteWorkflow(input)
  } finally {
    workflowSemaphore.release()
  }
}
```

---

#### RES-002: Executor资源隔离

**问题分析**:
- Code Executor可能占用过多资源
- 需与其他执行器隔离

**优化方案**:

```typescript
// Executor资源配额
interface ExecutorQuota {
  maxCpuPercent: number     // 50%
  maxMemoryMB: number       // 512MB
  maxExecutionTime: number  // 60s
}

// 资源监控与限制
async function executeWithQuota(input: ExecutorInput, quota: ExecutorQuota) {
  const monitor = startResourceMonitor(quota)
  try {
    return await doExecute(input)
  } catch (e) {
    if (monitor.exceeded) {
      return { status: 'resource_exceeded' }
    }
    throw e
  } finally {
    monitor.stop()
  }
}
```
