## 二、图谱冗余识别

### 2.1 高优先级冗余

#### GRP-001: context-graph.json 与 DEV-08§7 完全重复

**问题描述**:
- DEV-08§7包含"机器可读索引"JSON块
- context-graph.json包含相同内容
- 两处需同步维护，增加维护成本

**重复内容**:
```json
{
  "graph_version": "v1.9",
  "entry": "development/DEV-00-开发索引.md",
  "authority": { ... },
  "dev_mapping": { ... },
  "source_code_mapping": { ... }
}
```

**建议**:
- DEV-08删除§7内嵌JSON
- DEV-08添加引用: `详见 development/context-graph.json`
- 仅维护context-graph.json单一来源

---

#### GRP-002: object-relationship-graph.md§9 与 DEV-08§7 重复

**问题描述**:
- object-relationship-graph.md§9定义"文档权威源映射"
- DEV-08§7的authority字段定义相同内容
- 两处映射不一致时会导致混乱

**重复内容对比**:

| 对象 | object-relationship-graph.md§9 | DEV-08§7 authority |
|------|--------------------------------|---------------------|
| workflow | AH1-17 | AH1-17 |
| policy | AH1-16 | AH1-16 |
| contract | AH1-15 | AH1-15 |
| retrieval | AH1-20 | AH1-20 |
| executor | AH1-18 | AH1-18 |
| checkpoint | AH1-19 | AH1-19 |
| artifact | AH1-22 | AH1-22 |
| audit | AH1-23 | AH1-23 |
| provider | AH1-26 | AH1-26 |
| config | AH1-28 | AH1-28 |
| error | AH1-31 | AH1-31 |
| versioning | AH1-32 | AH1-32 |

**建议**:
- object-relationship-graph.md§9删除权威源映射表格
- 添加引用: `权威源映射详见 development/context-graph.json`

---

### 2.2 中优先级冗余

#### GRP-003: object-relationship-graph.md§11 与 DEV-08§6 重复

**问题描述**:
- 两处都定义"按任务加载的最小上下文包"
- 内容完全相同，表格格式一致

**建议**:
- object-relationship-graph.md删除§11
- 添加引用DEV-08§6

---

#### GRP-004: context-routing.json 与 DEV-10 内容重叠

**问题描述**:
- context-routing.json定义文件路径白名单
- DEV-10定义"工作区结构与读取白名单"
- 两处配置需同步

**建议**:
- DEV-10引用context-routing.json
- 仅维护context-routing.json

---

#### GRP-005: 多个图谱版本号不一致

**问题描述**:
- context-graph.json: v1.9
- object-relationship-graph.md: v2.2
- DEV-08: v1.9

版本号不一致导致无法判断哪个是最新。

**建议**:
- 统一版本号管理
- 在context-graph.json中添加version_sync字段

---

### 2.3 低优先级冗余

#### GRP-006: 图谱文件分散存储

**问题描述**:
- context-graph.json在development/
- object-relationship-graph.md在development/app-graph/
- 文件分散增加查找成本

**建议**:
- 将object-relationship-graph.md移到development/
- 或创建专门的graphs/目录

## 三、图谱结构优化建议

### 3.1 建议的图谱文件结构

```

├── graphs/
│   ├── context-graph.json          # 主索引（机器可读）
│   ├── context-routing.json        # 路由配置
```

## 四、图谱性能分析

### 4.1 当前加载策略问题

DEV-08定义的上下文加载策略：

```
默认只加载：1个对象权威源 + ≤2个直接依赖 + 1个DEV执行文档
```

**问题**:
1. 加载策略过于复杂，需要人工判断
2. 无自动化工具支持
3. 失效机制依赖人工触发

### 4.2 建议的优化方案

1. **自动化加载**: 开发脚本自动生成任务上下文包
2. **缓存机制**: 对常用任务上下文包进行缓存
3. **失效检测**: 自动检测权威文档变更并触发失效
