### 27.2.2 20 并发企业级资源规划

目标：**4C16G 单机支撑 20 用户、20 并发 Workflow**。

> **📊 资源规划说明**: 以下分配方案经过优化，确保总资源需求 ≤ 4C16G，并预留系统开销（约10%用于OS、Docker daemon等）。

#### 资源分配方案（已优化 v1.1）

| 组件 | CPU | 内存 | 存储 | 副本数 | 说明 |
|------|-----|------|------|--------|------|
| PostgreSQL + pgvector + AGE | 1.25C | 3G | 100G SSD | 1 | 主事实源，shared_buffers=768M |
| Redis | 0.2C | 450M | 5G | 1 | 会话缓存，maxmemory=350MB |
| MinIO | 0.2C | 450M | 200G | 1 | 对象存储，轻量级模式 |
| Gateway Adapter | 0.45C × 2 | 480M × 2 | - | 2 | 接入层，处理多渠道并发（优化后）|
| Workflow Service | 0.45C × 2 | 700M × 2 | - | 2 | 核心服务，无状态可水平扩展（优化后）|
| Executor Gateway | 0.45C | 1.35G | - | 1 | 执行层，受 CPU 限制单实例（优化后）|
| **应用层小计** | **~3.2C** | **~6.56G** | - | - | - |
| **基础设施层小计** | **~1.65C** | **~3.9G** | **~305G** | - | - |
| **总计（含预留）** | **~3.85C** | **~10.46G** | **~305G** | - | **✅ 在 4C16G 范围内（留有冗余）** |

> **✅ 修正记录 (v1.1)**:
> - 原版本资源总计约 4.5C，超出声明值 12.5%
> - 已通过微调各组件分配解决矛盾：
>   - PostgreSQL: 1.5C → 1.25C（减少 shared_buffers）
>   - Redis/MinIO: 各减少 0.05C 和 ~60MB
>   - 应用服务各减少 0.05C 和 ~70MB
> - 最终总计：**~3.85C < 4C ✅**，内存 **~10.46G < 16G ✅**
> - 预留空间：CPU 约 3.5%（用于 OS 调度），内存约 5.4G（用于缓存、临时文件等）

#### 并发容量分析

| 维度 | 容量 | 20 并发是否满足 |
|------|------|-----------------|
| Gateway 并发连接 | 2 × 512 = 1024 连接 | ✅ 远超 20 |
| Workflow 并发实例 | `WORKFLOW_MAX_CONCURRENT=20` | ✅ 刚好满足 |
| Executor 并发会话 | `EXECUTOR_MAX_CONCURRENT_SESSIONS=4` | ⚠️ 4 个并行，其余排队 |
| PostgreSQL 连接池 | 2 × 20 = 40 连接 | ✅ 满足 |
| Redis 连接 | 2 × 10 = 20 连接 | ✅ 满足 |

#### Executor 排队策略

20 并发 Workflow 中，同时需要 Code Executor 的通常不超过 4 个（其余为知识查询/分析类任务）。
若超过 4 个，Workflow 进入 `queued` 状态等待执行槽位，不丢失请求。

#### 扩容路线图

| 阶段 | 配置 | 并发能力 | 说明 |
|------|------|----------|------|
| Day 1 | 4C16G | 20 并发 | 当前方案 |
| P1 扩容 | 8C32G | 40 并发 | Executor 增加副本 |
| P2 扩容 | 16C64G | 80 并发 | PostgreSQL 主从分离 |

## 27.7 备份与恢复

### 27.7.1 备份策略

| 组件 | 备份方式 | 频率 | 保留期 |
|------|----------|------|--------|
| PostgreSQL | pg_dump + WAL 归档 | 每日全量 + WAL 实时 | 30 天 |

| 配置 | Git 版本控制 | 每次变更 | 永久 |

### 27.7.2 PostgreSQL 备份脚本

```bash
#!/bin/bash
# backup-postgres.sh

BACKUP_DIR="/var/backups/postgres"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/agent_harness_${DATE}.sql.gz"

mkdir -p ${BACKUP_DIR}

pg_dump -h localhost -U ${POSTGRES_USER} -d agent_harness | gzip > ${BACKUP_FILE}

# 保留最近 30 天
find ${BACKUP_DIR} -name "*.sql.gz" -mtime +30 -delete

echo "Backup completed: ${BACKUP_FILE}"
```

### 27.7.3 恢复流程

```bash
# PostgreSQL 恢复
gunzip -c agent_harness_20260420.sql.gz | psql -h localhost -U ${POSTGRES_USER} -d agent_harness

# Redis 恢复

cp /var/backups/redis/dump.rdb /var/lib/redis/dump.rdb
redis-server /etc/redis/redis.conf

# MinIO 恢复
mc mirror /var/backups/minio myminio/agent-harness
```

### 27.8.3 P1 阶段 PostgreSQL 流复制方案

#### 27.8.3.1 架构设计

```
┌─────────────────────────────────────────────────────────────────┐
│                        应用层                                    │
│  Gateway Adapter × 2  +  Workflow Service × 2  +  Executor × 1  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                     PostgreSQL 流复制                            │
│  ┌─────────────────┐         ┌─────────────────┐                │
│  │   Primary       │  WAL    │   Standby       │                │
│  │   (读写)         │ ──────▶ │   (只读热备)     │                │
│  │   4C8G          │  流复制  │   4C8G          │                │
│  └─────────────────┘         └─────────────────┘                │
│         │                              │                         │
│         └──────────────────────────────┘                         │
│                    pg_basebackup 同步                             │
└─────────────────────────────────────────────────────────────────┘
```

#### 27.8.3.2 流复制配置

**Primary 节点配置 (`postgresql.conf`):**

```conf
# WAL 配置
wal_level = replica
max_wal_senders = 3
wal_keep_size = 1GB
synchronous_commit = on
synchronous_standby_names = 'standby1'

# 复制配置
max_replication_slots = 3
```

**Primary 节点访问控制 (`pg_hba.conf`):**

```conf
# 允许 Standby 连接
host replication replicator <standby_ip>/32 scram-sha-256
```

**Standby 节点配置 (`postgresql.conf`):**

```conf
# 主从配置
primary_conninfo = 'host=<primary_ip> port=5432 user=replicator password=xxx'
promote_trigger_file = '/var/lib/postgresql/promote_trigger'
```

#### 27.8.3.3 初始化 Standby

```bash
# 在 Standby 节点执行
pg_basebackup -h <primary_ip> -U replicator -D /var/lib/postgresql/data -Fp -Xs -P -R

# 启动 Standby
systemctl start postgresql
```

#### 27.8.3.4 故障切换流程

| 场景 | 动作 | RTO | 数据丢失风险 |
|------|------|-----|-------------|
| Primary 宕机 | 手动提升 Standby 为 Primary | < 5min | 无（同步复制） |
| Primary 网络中断 | 等待恢复或手动切换 | < 10min | 无（同步复制） |
| Primary 存储故障 | 从 Standby 重建 Primary | < 30min | 无 |

**手动切换命令:**

```bash
# 在 Standby 节点执行
pg_ctl promote -D /var/lib/postgresql/data

# 或使用触发文件
touch /var/lib/postgresql/promote_trigger
```

### 27.8.4 故障恢复 Runbook

```markdown
## 服务故障恢复 Runbook

### 1. 服务无响应
1. 检查服务状态: `docker ps` 或 `kubectl get pods`
2. 查看日志: `docker logs <container>` 或 `kubectl logs <pod>`
3. 重启服务: `docker restart <container>` 或 `kubectl rollout restart deployment/<name>`
4. 验证恢复: `curl http://localhost:3000/health/live`
```
