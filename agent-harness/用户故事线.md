# JueYing (绝影) — 用户故事线

> 版本: 1.0.0 | 更新日期: 2026-05-03
> 品牌名称: JueYing (绝影) | 内部代号: agent-harness

---

## 系统背景概述

JueYing (绝影) 是一个**企业级 AI Agent 编排与执行平台**。用户通过飞书、企业微信等 IM 渠道与系统交互，系统使用大语言模型（LLM）将用户意图自动规划为多阶段工作流，调度多种专用执行器完成各阶段任务，并主动推送执行结果。

### 核心角色

| 角色 | 描述 |
|------|------|
| **平台管理员 (Admin)** | 系统的唯一管理角色，负责初始化部署、组织管理、用户管理、人设配置、策略审核、知识审批等全部管理操作 |
| **普通用户 (User)** | 通过飞书/企微与 AI 对话，提交长任务，响应工作要求 |

### 服务架构总览

系统由以下核心服务组成（采用 Docker Compose 容器化部署）：

| 服务 | 职责 |
|------|------|
| **gateway-adapter** (端口 3000) | 多渠道消息入口，身份解析与绑定，意图分类，路由到 workflow 或直接 LLM 对话 |
| **workflow-service** (端口 3001) | 工作流规划(planner)、监督(supervisor)、状态机(engine)，CRUD |
| **executor-gateway** (端口 3002) | 接收 dispatch，按阶段调度多种执行器，完成后回调 workflow |
| **fact-retrieval** (端口 3004) | 事实存储与向量检索，文档分块与索引、可选图检索和重排序 |
| **hermes-adapter** (端口 3005) | 会话记忆管理：存储、召回、压缩上下文 |
| **web-portal** (端口 3003) | Web 管理界面：登录/设置向导/工作流管理/策略管理/组织管理 |
| **feishu-longconn** | 飞书长连接 WebSocket 客户端，转发消息到 gateway |
| **LiteLLM Proxy** (端口 4000) | LLM 统一代理（支持 MiniMax-M2.7、Qwen3-Max、GLM-5 等多种模型） |

## 故事线一：平台初始化配置

### 1.1 场景描述

平台管理员首次部署 JueYing 系统，需要完成一系列初始化配置，使系统从"裸机"状态变为"可投产"状态。

### 1.2 初始化向导流程

管理员启动 Docker Compose 后，访问 Web Portal（`http://localhost:3003`），系统检测到尚未初始化，自动弹出**设置向导**，引导管理员完成以下 6 个步骤：

```
步骤一：数据库连接
  → 系统自动检测 PostgreSQL 连接状态
  → 连接成功后数据库迁移自动执行
  → 创建 agent_harness 数据库及所有表结构

步骤二：组织创建
  → 输入组织名称（如 "default"）
  → 输入显示名称（如 "Default Organization"）
  → 系统在 organization 表中创建组织记录

步骤三：管理员创建
  → 输入管理员用户名（如 "admin"）
  → 输入管理员密码
  → 密码使用 scrypt(N=16384) 哈希存储
  → 系统在 user 表中创建首个 admin 角色用户

步骤四：消息渠道配置
  → 配置飞书渠道：App ID、App Secret、Verification Token、Encrypt Key
  → 配置企业微信渠道：Corp ID、Agent ID、Secret、Token、AES Key
  → 写入 .env 文件持久化
  → 此步骤可跳过，后续在系统配置中完成

步骤五：LLM 模型配置
  → 输入 LiteLLM 地址（如 http://localhost:4000）
  → 输入默认模型名称（如 minimax-m2.7）
  → 配置 LiteLLM Master Key
  → 写入 .env 文件

步骤六：向量模型配置
  → 选择 Embedding 模式：deterministic（内置）或 provider（外部 API）
  → 如选择 provider：输入 Provider URL、Model、API Key、维度
  → 写入 .env 文件
```

### 1.3 环境变量 (.env) 配置体系

初始化完成后，`.env` 文件包含以下核心配置：

```bash
# 数据库
DATABASE_URL=postgresql://agent_harness:dev_password@postgres:5432/agent_harness

# LLM 模型
LITELLM_URL=http://litellm:4000
LITELLM_MASTER_KEY=litellm-dev-key
LITELLM_MODEL=minimax-m2.7
LITELLM_PLAN_MODEL=minimax-m2.7
LITELLM_CODE_MODEL=minimax-m2.7
LITELLM_PLAN_TIMEOUT_MS=60000

# 飞书渠道
FEISHU_APP_ID=cli_xxx
FEISHU_APP_SECRET=xxx
FEISHU_DOMAIN=feishu

# 服务间通信 URL（容器内均监听 3000 端口）
WORKFLOW_URL=http://workflow-service:3000
EXECUTOR_URL=http://executor-gateway:3000
FACT_RETRIEVAL_URL=http://fact-retrieval:3000
HERMES_URL=http://hermes-adapter:3000
GATEWAY_URL=http://gateway-adapter:3000

# JWT / 安全
JWT_SECRET=dev-secret-key
```

### 1.4 初始化完成后的状态

- Web Portal 登录页面可用
- 管理员可以登录并进入管理仪表盘
- 服务状态概览面板显示各服务健康状态
- 系统进入"可投产"状态

## 故事线二：组织与用户开通

### 2.1 场景描述

平台支持多个组织（部门）共用同一套基础设施，每个组织的数据完全隔离。管理员负责本组织内的用户管理、策略配置和任务分发。

### 2.2 组织管理（管理员操作）

管理员登录 Web Portal，进入"**组织管理**"页面：

```
创建新组织：
  → 输入组织名称（org_name），如 "sales-dept"
  → 输入显示名称（display_name），如 "销售部"
  → 系统在 organization 表中创建记录
  → 默认状态为 active

编辑组织：
  → 修改显示名称
  → 设置资源配额：
    · 用户上限（max_users），如 100
    · 每日 Workflow 上限（max_workflows_per_day），如 500
  → 修改状态：active（正常）/ suspended（暂停）/ deleted（软删除）

删除组织：
  → 软删除（status = 'deleted'）
  → 关联用户将无法登录
```

### 2.3 用户管理（管理员操作）

管理员进入"**用户管理**"页面：

```
创建用户：
  → 输入用户名
  → 输入密码（要求至少8位，同时包含字母和数字，不允许常见密码）
  → 选择角色：admin（管理员）或 user（普通用户）
  → 用户自动关联到当前组织
  → 系统在 user 表中创建记录
  → 密码使用 scrypt(N=16384) 哈希存储于 metadata 字段

用户列表：
  → 展示所有用户的用户名、角色、状态、所属组织
  → 支持分页查看（最多1000条）
```

### 2.4 成员邀请机制

```
创建邀请：
  → 输入被邀请人标识（invitee）
  → 选择角色（admin / user）
  → 设置过期时间（默认24小时）
  → 系统生成12位邀请码（invite_code）
  → 记录在 org_invitation 表中

撤销邀请：
  → 将邀请状态改为 revoked

邀请接受：
  → 被邀请人通过邀请码注册
  → 自动关联到对应组织
```

### 2.5 渠道身份绑定

每个用户可以通过飞书或企业微信与系统交互，系统需要将渠道身份与系统用户绑定：

```
飞书用户首次发消息：
  → 系统在 channel_identity 表中查找该飞书ID
  → 如果不存在：**自动创建用户并绑定**（生成 `u_feishu_{suffix}` 用户名）
  → 如果存在且已绑定（bound）：直接使用
  → 如果冲突（conflicted）：记录异常，等待管理员处理

绑定状态：
  → pending：待绑定
  → bound：已绑定
  → conflicted：冲突（同一渠道身份关联了多个用户）

身份管理页面：
  → 管理员查看所有渠道绑定记录
  → 对 pending/conflicted 状态可手动执行绑定
```

### 2.6 组织隔离机制

系统中的核心数据均按组织进行隔离：

| 数据类型 | 隔离方式 |
|----------|----------|
| 用户 (user) | 每个用户属于唯一组织（org_id） |
| 工作流实例 (workflow_instance) | 按 owner_user_id + org_id 隔离 |
| 记忆条目 (memory_item) | 按 owner_user_id + org_id 隔离 |
| 事实/知识 (fact) | 按 owner_user_id + org_id 隔离 |
| 文档 (document) | 按 owner_user_id + org_id + scope_type 隔离 |
| 组织任务 (org_task) | 按 org_id 隔离 |
| 审计日志 (audit_event) | 按 org_id 隔离 |

## 故事线三：配置 LLM 模型

### 3.1 场景描述

系统依赖大语言模型来完成对话生成、意图分类和工作流规划。通过 LiteLLM Proxy 统一管理和代理所有 LLM 请求。

### 3.2 LiteLLM 代理架构

```
                    ┌──────────────────┐
                    │   LiteLLM Proxy   │
                    │   (端口 4000)      │
                    └──┬───┬───┬────────┘
                       │   │   │
              ┌────────┘   │   └──────────┐
              ▼            ▼              ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │MiniMax   │ │Qwen3     │ │GLM-5     │
        │M2.7      │ │Max       │ │          │
        └──────────┘ └──────────┘ └──────────┘
```

### 3.3 模型配置项

通过 .env 文件或 Web Portal 的"系统配置"页面进行配置：

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `LITELLM_URL` | LiteLLM 代理地址 | `http://localhost:4000` |
| `LITELLM_MASTER_KEY` | LiteLLM 主密钥 | - |
| `LITELLM_MODEL` | 默认对话模型 | `minimax-m2.7` |
| `LITELLM_PLAN_MODEL` | 工作流规划专用模型 | `minimax-m2.7` |
| `LITELLM_CODE_MODEL` | 代码生成专用模型 | `minimax-m2.7` |
| `LITELLM_PLAN_TIMEOUT_MS` | 规划超时（毫秒） | `60000` |

### 3.4 模型调用路径

```
对话生成（Chat）：
  gateway-adapter → LiteLLM Proxy → LLM → 返回回复

意图分类（Intent Classification）：
  gateway-adapter → LiteLLM Proxy → LLM → 返回 chat/task

工作流规划（Plan Generation）：
  workflow-service/planner → LiteLLM Proxy → LLM → 返回 stage_chain JSON
  · 调用参数: temperature=0.3, max_tokens=16384, response_format=json_object
  · 支持重试: 最多2次, 退避策略 2s→4s, 带 jitter（规划超时60s）

执行器调用（Executor）：
  executor-gateway → LiteLLM Proxy → LLM → 返回任务结果
```

## 故事线四：配置向量模型 (Embedding)

### 4.1 场景描述

向量模型用于将文档内容和记忆条目转换为向量嵌入（embedding），存储在 PostgreSQL + pgvector 中，以支持语义相似度检索。

### 4.2 Embedding 配置项

通过 .env 文件或 Web Portal 配置：

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `EMBEDDING_MODE` | 模式：`deterministic`（内置）或 `provider`（外部API） | `deterministic` |
| `EMBEDDING_PROVIDER_URL` | 外部 Provider API 地址 | - |
| `EMBEDDING_PROVIDER_MODEL` | 外部 Provider 模型名称 | - |
| `EMBEDDING_PROVIDER_API_KEY` | 外部 Provider API Key | - |
| `EMBEDDING_DIMENSION` | 向量维度 | `1536` |

### 4.3 向量应用场景

```
文档索引流程：
  用户上传文档 → fact-retrieval 接收
    → 文档分块（document_chunk 表）
    → 每块计算 token_count
    → 调用 embedding 模型生成向量（1536维 pgvector）
    → 同时生成全文搜索向量（tsvector）
    → 存储到 document_chunk 表

记忆条目流程：
  对话消息 → hermes-adapter 接收
    → 创建 memory_item 记录
    → 调用 embedding 模型生成向量
    → 存储 embedding 到 memory_item 表
    → 用于后续上下文召回时的相似度匹配

事实条目流程：
  知识导入 → fact-retrieval 接收
    → 创建 fact 记录
    → 抽取实体（entity）
    → 建立实体间关系（relation）
```

### 4.4 deterministic 模式

当选择 `deterministic` 模式时，系统使用内置的确定性哈希算法生成伪向量，无需调用外部 API，适用于开发测试环境。

## 故事线五：配置 Rerank 模型

### 5.1 场景描述

重排序（Rerank）模型用于对初步检索结果进行精细化排序，提高最终返回给用户的知识片段的相关性和准确度。

### 5.2 Rerank 配置项

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `RERANK_MODE` | 模式：`deterministic` / `ollama` / `remote-provider` | `deterministic` |
| `RERANK_PROVIDER_URL` | 远程 Provider API 地址 | - |
| `RERANK_PROVIDER_MODEL` | 远程 Provider 模型 | - |
| `RERANK_PROVIDER_API_KEY` | 远程 Provider API Key | - |
| `RERANK_OLLAMA_URL` | Ollama 本地服务地址 | - |
| `RERANK_OLLAMA_MODEL` | Ollama 模型名称 | - |

### 5.3 Rerank 在检索流程中的位置

```
用户查询
  → 向量检索（pgvector 余弦相似度）：初筛 Top-N（如 50 条候选）
  → 全文检索（tsvector 文本匹配）：补充候选
  → Rerank 模型精排：对候选集重新打分排序
  → 返回 Top-K 结果（如 5 条最相关）
  → 记录检索追踪（retrieval_trace）
```

### 5.4 检索配置 Profile

系统支持三种检索策略配置：

| Profile | 说明 | 适用场景 |
|---------|------|----------|
| `precision-first` | 精确优先，少而精 | 事实查询 |
| `balanced` | 均衡模式 | 一般任务 |
| `comprehensive` | 全面模式，多而全 | 深度分析 |

## 故事线六：统一知识管理

### 6.1 场景描述

组织可以将各类知识文档导入系统，系统自动进行分块、向量化、实体抽取和关系构建，形成可被检索和引用的知识库。知识分为私有（仅本人可见）和共享（组织内可见）两种。

### 6.2 个人知识导入

普通用户通过 Web Portal 的"**知识导入**"页面操作：

```
导入流程：
  1. 输入标题（如 "Q1销售报告"）
  2. 输入/粘贴内容
  3. 选择来源类型：
    · manual：手动输入
    · document：文档上传
    · conversation：对话记录
  4. 选择可见范围：
    · private：仅自己可见
    · org：本组织内公开
  5. 勾选"自动抽取实体和关系"（默认开启）

后台处理链路：
  → POST /api/knowledge/import
  → 调用 fact-retrieval /internal/documents/index
    · 创建 document 记录（含 title, source_kind, scope_type, status）
    · 创建 document_version 记录（含 storage_ref, content_hash）
    · 内容分块为 document_chunk（含 chunkIndex, contentText, tokenCount）
    · 对每个 chunk 生成 embedding 向量
    · 生成全文搜索 tsvector
  → 如果勾选"自动抽取"，调用 /internal/entities/write
    · 创建 entity 记录
    · 建立 fact（事实三元组：subject → predicate → object）
    · 建立 relation（实体间关系）
```

### 6.3 共享知识库（管理员操作）

管理员通过 Web Portal 的"**共享知识库**"页面操作：

```
上传共享文档：
  → 输入标题
  → 输入内容
  → 选择来源类型（manual / template / guide / reference）
  → 文档 scope_type 设为 'shared'
  → 所有组织用户均可检索到此文档

管理共享文档：
  → 查看已共享文档列表
  → 支持移除（软删除，status = 'deleted'）

组织级查询：
  → 所有组织用户可通过 /api/knowledge/shared 接口查询共享文档
```

### 6.4 知识检索体系

```
事实表 (fact)：
  · subject_ref：主体引用
  · predicate：谓词（关系描述）
  · object_value / object_json：客体值
  · confidence：置信度
  · status：unconfirmed / active / superseded / deprecated / rejected / deleted

实体表 (entity)：
  · entity_type：实体类型
  · canonical_name：规范名称
  · 支持属性扩展（entity_attribute）

文档块表 (document_chunk)：
  · embedding：向量嵌入（1536维 pgvector）
  · search_tsv：全文搜索向量（tsvector）
  · 支持向量检索 + 全文检索混合查询

检索追踪表 (retrieval_trace)：
  · query_text：查询文本
  · intent_type：意图类型
  · result_summary：结果摘要
  · degraded：是否降级
  · duration_ms：耗时
```

### 6.5 主动提交知识（自然语言）

普通用户可以通过自然语言向管理员主动提交知识，无需 Web 界面：

```
用户在飞书发送：
  "我想提交一条知识：XX银行的采购决策流程一般是先技术评估
   再商务谈判，最后法务审核，整个周期大约2个月。"

智能体回复：
  "已收到你的知识提交，内容已存入临时审核池（仅你和管理员可见），
   管理员审批通过后将进入组织共享知识库。"

后台处理流程：
  → 系统识别 intent='knowledge_submit'
  → 写入 fact 表（scope_type='org', status='unconfirmed'）
  → 标记 source='user_submitted'
  → 关联 owner_user_id（记录提交人）
  → 临时审核池中展示：提交人、时间、内容摘要、状态

管理员审核流程：
  → 与定时抽取的知识条目在同一个「知识审核台」中展示
  → 可区分来源：dream_extraction / admin_extraction / user_submitted
  → 审核操作与普通知识条目一致（批准/退回/拒绝）

临时审核池：
  → 提交人可在「我的知识提交」页面查看自己提交但尚未审批的条目
  → 仅提交人本人和管理员可见
  → 审批通过后条目从临时池移除，进入主知识库
```

## 故事线七：子用户日常对话

### 7.1 场景描述

组织内的普通用户通过飞书与 AI 进行日常对话，询问问题、获取信息或闲聊。系统需要理解上下文、保持多轮对话连贯性。

### 7.2 飞书对话完整链路

```
第一步：用户发消息
  用户在飞书 App 中向机器人发送消息

第二步：飞书长连接接收
  feishu-longconn 服务通过 WebSocket 接收飞书事件
  将事件转发到 gateway-adapter（POST /channels/feishu/longconn/event）
  gateway 立即返回 200 OK（不阻塞飞书回调）

第三步：身份解析（异步处理）
  resolveIdentity()：
    → 查 channel_identity 表，根据 channel_type + external_identity 匹配
    → 如果找到且已绑定（bound）：直接使用
    → 如果未找到：自动创建用户 + 立即绑定
    → 如果冲突（conflicted）：记录异常

第四步：消息规范化
  normalizeMessage()：
    → 提取文本内容
    → 提取 session_hint（用于记忆关联）
    → 提取 conversation_id

第五步：意图分类
  classifyIntentWithLLM()：
    → 调用 LiteLLM 判断用户意图
    → 返回 chat（普通对话）或 task（长任务）

第六步（Chat 路径）：上下文召回 + 回复生成
  recallContext() → hermes /internal/memory/recall：
    → 根据当前用户和会话ID
    → 检索相关历史记忆（基于向量相似度）
    → 如果会话过长，自动压缩为摘要

  generateChatReply() → LiteLLM：
    → 构造 messages（system prompt + 压缩后的上下文 + 当前消息）
    → 调用 LLM 生成回复

  rememberContext() → hermes /internal/memory：
    → 存储本轮用户消息和助手回复
    → 生成 embedding 用于后续召回

第七步：发送回复
  sendFeishuTextReply() → 飞书 API：
    → 用户收到 AI 的回复内容
```

### 7.3 多轮对话记忆机制

```
会话创建：
  → 每个对话创建 session 记录（owner_user_id + session_id）

记忆存储：
  → 每轮对话存储两条 memory_entry：
    · role='user'：用户说的话
    · role='assistant'：AI 的回复
  → 每条记忆附带向量嵌入（用于语义检索）

上下文召回：
  → 新消息到来时，基于向量相似度检索相关历史记忆
  → 返回 Top-N 条相关记忆作为上下文

压缩摘要：
  → 当会话记忆条数超过配置上限时
  → Hermes 自动将早期记忆压缩为摘要
  → 保留摘要 + 最近记忆，确保上下文窗口不超限
```

## 故事线八：子用户长任务工作流生成与执行

### 8.1 场景描述

当用户提出的问题或请求较为复杂（需要多步骤推理、查找资料、生成报告等），系统自动识别为"长任务"，触发工作流规划引擎，将用户目标拆解为可执行的多阶段任务链。

### 8.2 任务触发与受理

```
第一步：意图识别为 task
  classifyIntentWithLLM() 返回 task

第二步：前置检查
  → 检查 identity_binding_state === 'bound'（必须已绑定身份）
  → 检查 org 限额（每日 workflow 数量是否超限）

第三步：创建任务规划
  → POST workflow-service /internal/workflows/plan
  → planner 接收 PlannerInput：
    {
      user_id: "用户ID",
      user_goal: "用户的任务目标描述",
      task_type_hint: "knowledge|analysis|development|sales|implementation",
      risk_level: "low|medium|high",
      policy_snapshot_hash: "sha256:xxx",
      budget: { time_sec: 3600, retrieval: 50, execution: 300 }
    }

第四步：返回受理回复
  → gateway 立即向用户飞书发送：
    "任务已受理，workflow=wf_xxxxx"
  → 用户收到确认消息

第五步：后台轮询结果
  pollAndReplyWorkflowResult()：
    → 每 10 秒查询 workflow 状态
    → 当 workflow 状态变为 completed/succeeded 时
    → 通过飞书向用户发送最终结果
```

### 8.3 LLM 工作流规划详解

```
Planner 工作流程：

1. Markdown 步骤解析（优先）
  → 如果输入包含 markdown_steps
  → 根据步骤名称自动推断 stage_type：
    · "澄清/明确" → IntentClarification
    · "检索/查找" → EvidenceRetrieval
    · "实现/开发" → Implementation
    · "验证/测试" → Verification
    · "报告/结果" → ResultReporting
    · ...（支持16种阶段类型）

2. 技能匹配
  → 从 skill 表中搜索匹配的技能
  → 基于关键词匹配（用户目标关键词 vs 技能名称/描述）
  → 如果匹配到相关技能，使用技能预定义的阶段模板

3. LLM 生成（核心路径）
  → 调用 LiteLLM，传入 system prompt + user prompt
  → system prompt 定义了：
    · 所有阶段类型和对应执行器映射
    · 标准 workflow 模板示例
  → user prompt 包含用户目标和上下文
  → LLM 返回 JSON 格式的阶段链

4. 默认降级
  → 如果 LLM 调用失败，使用默认阶段链：
    · knowledge 类型: IntentClarification → Approval → ResultReporting
    · analysis 类型: IntentClarification → EvidenceRetrieval → DecisionMaking → Approval → ResultReporting
    · development 类型: IntentClarification → PlanGeneration → Implementation → Verification → Approval → ResultReporting
```

| ObjectExtraction | generic-executor | 从内容中提取结构化对象 |
| ArchitectureDesign | generic-executor | 设计系统架构 |
| SpecGeneration | generic-executor | 生成规格说明书 |
| DecisionMaking | generic-executor | 基于证据做决策推理 |
| Implementation | code-executor | 编写/执行代码 |
| Verification | verification-executor | 验证结果正确性 |
| Repair | repair-executor | 修复发现的问题 |
| Approval | approval-executor | 等待人工审批 |

## 故事线九：工作流执行与汇报

### 9.1 场景描述

工作流被规划完成后，执行器网关按阶段顺序自动执行每个阶段，每个阶段完成后向工作流服务上报结果，全部完成后触发汇报流程，最终结果推送给用户。

### 9.2 自动执行编排

```
executor-gateway /internal/executor/dispatch 接收任务：

1. 立即返回受理
  → { dispatch_status: 'accepted', executor_run_ref: 'run_xxx', state: 'queued' }

2. 异步执行 autoExecuteWorkflowStages()：

  a. 从 workflow-service 拉取完整计划
      → GET /internal/workflows/{ref}
      → 解析 plan.stage_chain

  b. 提取 user_goal
      → 从 plan.goal.user_goal 提取
      → 注意：plan.goal 是对象，需提取子字段，防止出现 [object Object]

  c. 按序执行每个阶段：
      for (stage of stageChain) {
        if (stage.status !== 'running' && stage.status !== 'pending') continue;

        → 根据 stage.assigned_executor 选择执行器
        → 创建 ExecutorInput {
            workflow_instance_id,
            workflow_stage_id,
            stage,
            user_goal,
            policy_snapshot_hash,
            context: { owner_user_id, run_ref }
          }
        → executor.execute(input)
        → 调用 LiteLLM 完成阶段任务
        → 返回 { status, output, artifacts, fact_refs }
        → POST workflow /internal/workflows/{ref}/stages/{sid}/dispatch
          （上报阶段结果，含带重试机制，最多 5 次）
        → workflow 收到上报后记录 heartbeat

        if (result.status === 'failed' | 'waiting_user' | 'blocked')
          break; // 阶段链中止
      }

  d. 全部阶段完成后：
      → POST workflow /internal/workflows/{ref}/complete
      → workflow 触发状态机推进：
        running → verifying → reporting → succeeded
```

### 9.3 执行器类型详解

| 执行器 | 职责 | 调用方式 |
|--------|------|----------|
| **generic-executor** | 通用执行器，处理意图澄清、决策、报告等 | 直接调用 LLM + 构造输出 |
| **retrieval-aware-executor** | 检索感知执行器，先检索知识库再调用 LLM | fact-retrieval 检索 + LLM 推理 |
| **code-executor** | 代码执行器，生成并执行代码 | LLM 生成代码 + 沙箱执行 |
| **verification-executor** | 验证执行器，验证前置阶段的输出 | LLM 检查正确性/完整性 |
| **repair-executor** | 修复执行器，修复验证发现的问题 | LLM 生成修复方案 |
| **approval-executor** | 审批执行器，暂停等待人工审批 | 记录审批状态，等待管理员操作 |

### 9.4 结果汇报机制

```
Workflow 完成后汇报流程：

1. Gateway 后台轮询
   pollAndReplyWorkflowResult()：
     → 每 10 秒 GET workflow-service /internal/workflows/{ref}
     → 检查 status 是否为 completed / succeeded / failed

2. 飞书推送结果
   → workflow.status === 'succeeded':
     sendFeishuTextReply("任务执行完成: {result_summary}")
   → workflow.status === 'failed':
     sendFeishuTextReply("任务执行失败: {error_message}")

3. Web Portal 汇报
   → 工作流控制台查看所有工作流状态
   → 点击详情查看每个阶段的状态和输出
   → 审批台查看待审批的工作流

4. 报告策略 (report_policy)
   → on_waiting_user：需要用户输入时汇报
   → on_blocked：被阻塞时汇报  
   → on_final：最终完成时汇报
   → on_stage_complete：每个阶段完成时汇报（仅 Markdown 任务启用）
```

### 9.5 故障恢复机制

```
检查点 (checkpoint)：
  → 每个阶段的 on_enter / on_exit 可配置自动创建检查点
  → 记录状态快照、artifacts 引用、fact 引用
  → 通过 resume_token 恢复

重试策略 (retry_policy)：
  → max_retries：阶段失败后重试次数
  → max_repairs：Implementation 阶段可自动修复（最多5次）
  → retryable_errors：可重试的错误类型

暂停恢复：
  → 管理员可通过 Web Portal 暂停/恢复工作流
  → 暂停时保存完整状态
  → 恢复时从检查点继续
```

## 故事线十：管理员下发工作要求

### 10.1 场景描述

管理员通过指定沟通渠道（如飞书或企业微信对话）使用自然语言下发工作要求（如每日拜访总结、周报、项目进度反馈等）。系统通过 LLM 进行意图识别，自动创建任务、分配用户并发送通知。

典型 LUI 指令示例：
- "通知全员提交周报"
- "给销售团队下发任务：本周五前提交客户拜访报告"
- "下发工作要求：每天20点提交当日总结"

### 10.2 自然语言任务下发流程

管理员在飞书或企业微信中发送自然语言指令后，系统的处理流程如下：

```
系统处理流程：

1. LLM 意图分类
  → 管理员消息通过 gateway-adapter 的 classifyIntent() 进行意图分析
  → 识别 intent_type 为 "task_dispatch"（区别于普通 chat、task、knowledge_submit）
  → 区分要点：
    · task_dispatch：管理员向团队分配任务（如"通知所有人提交周报"）
    · task：用户希望 AI 为自己执行工作流（如"帮我分析数据"）

2. 身份校验
  → 检查发消息的用户 identity_binding_state 是否为 "bound"
  → 未绑定的用户将收到提示："身份尚未绑定，请先完成身份验证"
  → 已绑定用户继续执行

3. 创建组织任务
  → INSERT INTO org_task 表：
    · org_id：管理员所属组织
    · created_by：管理员用户 ID
    · title：LUI 消息前100字符
    · description：完整消息内容
    · task_type：'form'（表单收集）
    · schedule_type：'once'（单次执行）
    · status：'active'
    · prompt_message：原始消息内容
    · target_channels：['wecom', 'feishu']
    · source：'lui'（标记来源为自然语言界面）

4. 分配任务
  → POST /internal/tasks/assign（网关内部接口）
  → 查询目标组织的所有活跃用户
  → 避免重复分配（检查是否已有 pending/notified 记录）
  → 为每个用户创建 org_task_assignment 记录（status='pending'）

5. 发送通知
  → POST /internal/tasks/notify（网关内部接口）
  → 查找所有 status='pending' 的分配记录
  → 通过 gateway 向每个用户发送通知消息
  → 更新分配状态为 'notified'，记录通知时间
  → 通知内容格式：
    "📋 **{任务标题}**
     {提醒消息}
     请及时提交您的反馈。"

6. 反馈结果
  → 系统通过对话回复管理员：
    "✅ 工作要求已创建并下发！
     📋 任务: {任务标题}
     👥 已分配: N 人
     📢 已通知: N 人
     任务编号: {任务ID}"
```

### 10.3 任务调度器（定时任务）

系统内置任务调度器（Cron Scheduler），每分钟检查一次是否有待触发的周期性任务：

```
runTaskScheduler()（每60秒执行）：

  1. SELECT * FROM org_task WHERE status = 'active'
     AND schedule_type IN ('daily', 'weekly', 'cron')

  2. 对每个任务检查是否应该触发：
    · daily：按每日调度规则触发
    · weekly：按每周调度规则触发
    · cron：解析自定义 cron_expression

  3. 触发任务：
    a. 通过 /internal/tasks/assign 分配任务
    b. 通过 /internal/tasks/notify 发送通知

  4. 已过期的 once 类型任务自动归档
```

### 10.4 用户接收与响应

```
用户端体验：

1. 收到通知
  → 企业微信或飞书收到任务提醒消息
  → 通知内容包含任务说明和提交指引

2. Web Portal 查看"我的任务"
  → GET /api/tasks
  → 显示所有待处理任务：
    · 未完成：显示输入框 + 提交按钮
    · 已完成：显示完成时间

3. 提交反馈
  → 输入总结/反馈内容
  → POST /api/tasks/{assignment_id}/submit
  → 系统更新：
    · org_task_assignment.status = 'completed'
    · org_task_assignment.completed_at = NOW()
    · org_task_assignment.response_data = { summary: "用户输入的内容" }
```

### 10.5 管理员跟踪与统计

管理员通过 Web Portal 可以查看任务执行情况：

```
任务列表查看：
  → 显示所有任务的标题、类型、调度方式、状态
  → 显示完成进度（如 "15/30 完成"）
  → 支持操作：
    · 立即分发：手动触发一次分配+通知
    · 暂停：将任务状态改为 paused
    · 归档：将任务状态改为 archived

任务详情：
  → 查看每个用户的完成状态
  → 查看用户提交的反馈内容
  → 导出反馈数据
```

### 10.6 任务生命周期

```
创建 → active → paused / archived
  │                │        ▲
  │                └────────┘
  │              (暂停任务)
  ▼
active（运行中）：
  ├── 调度器按 Cron 表达式触发
  ├── 分配 → 通知 → 收集反馈

  └── 保留历史数据，不可恢复
```

## 故事线十二：PGSQL 全量数据存储与组织级权限隔离

### 12.1 场景描述

所有用户的日常对话记录、工作流执行日志、知识检索行为、文件上传记录等，全部实时同步写入 PostgreSQL 数据库。这不仅是一种数据持久化策略，更是权限隔离、跨库同步和管理审计的基础设施底座。

### 12.2 数据写入架构

```
所有用户交互行为的写入路径：

飞书/企微消息
  → gateway-adapter 解析与路由
    │
    ├── 对话记忆路径
    │   → hermes-adapter /internal/memory
    │   → 写入 memory_item 表（含 embedding 向量）
    │   → 建立 memory_source 引用（source_type, source_ref）
    │   → 记录 memory_usage_log（被哪些 workflow 引用过）
    │
    ├── 工作流执行路径
    │   → workflow-service 写入 workflow_instance（含 plan、status）
    │   → 每个阶段写入 workflow_stage（含 input/output hash）
    │   → 每个检查点写入 checkpoint（含状态快照、证据哈希）
    │   → 每个状态变更写入 workflow_event（事件溯源）
    │
    ├── 知识/文档路径
    │   → fact-retrieval 写入 document → document_version → document_chunk
    │   → 自动抽取 entity + entity_attribute
    │   → 自动建立 fact（三元组：主体→谓词→客体）
    │   → 自动建立 relation（实体间关系图边）
    │
    └── 审计路径
        → auditWriter 写入 audit_event（所有关键操作）
        → fact-retrieval 写入 retrieval_trace（每次检索快照）
```

### 12.3 权限隔离机制

PostgreSQL 的行级安全策略是权限隔离的核心：

```
组织级隔离（Row-Level Security）：
  · 每个表的 org_id 字段是隔离的主键维度
  · 查询时自动注入 WHERE org_id = $current_org
  · 跨组织查询需 admin 角色 + 明确授权

用户级隔离：
  · 私有数据 scope_type = 'private'，仅 owner_user_id 可见
  · 组织内共享 scope_type = 'org'，本组织成员可见
  · 全局共享 scope_type = 'shared'，跨组织可见（仅限 admin 发布的共享知识库）

角色级权限：
  · admin：读写本组织所有数据 + 管理策略 + 审核知识
  · user：读写本人数据 + 查看组织内共享数据
```

### 12.4 跨库同步能力

由于所有数据落地在标准 PostgreSQL，可以通过成熟的数据库工具实现跨库同步：

```
同步方案：
  · Logical Replication：原生的逻辑复制，支持表级别的实时同步到其他 PostgreSQL 实例
  · Foreign Data Wrapper (FDW)：访问外部数据源的联邦查询，跨数据库联接
  · pg_dump/pg_restore：定期全量或增量备份与迁移
  · 第三方工具：Debezium（CDC 变更数据捕获）、Airbyte（ETL 管道）

典型场景：
  · 将用户行为数据同步到数据仓库（ClickHouse）进行分析
  · 将审计日志实时同步到安全合规平台
  · 将知识库内容镜像到客户本地的私有化部署实例
  · Admin 通过 BI 工具（如 Metabase / Grafana）直连 PGSQL 进行组织级数据分析
```

### 12.5 Admin 全局查询能力

管理员通过 Web Portal 的各个管理页面，实质上在执行组织范围内的 PGSQL 查询：

| 数据域 | 管理页面 | 底层查询 |
|--------|----------|----------|
| 用户全貌 | 用户管理 | `SELECT * FROM user WHERE org_id = $1` |
| 身份绑定 | 身份绑定管理 | `SELECT * FROM channel_identity`（admin 全量） |
| 工作流全景 | Workflow 控制台 | `SELECT * FROM workflow_instance`（admin 全量） |
| 审计追踪 | 审计日志 | `SELECT * FROM audit_event ORDER BY occurred_at DESC` |
| 检索质量 | 检索追踪 | `SELECT * FROM retrieval_trace ORDER BY created_at DESC` |
| 任务反馈 | 任务分发 | `JOIN org_task + org_task_assignment` 统计完成率 |

## 故事线十三：定时知识抽取与管理员审核

### 13.1 场景描述

组织每天产生大量对话记录、上传文档和工作流产出物，其中蕴含着对组织有价值的业务知识。管理员可以配置定时抽取任务，由大模型在访问量较低的时段（如凌晨），自动扫描全部用户的对话日志和文件，提取其中有价值的知识条目，默认存入组织知识库但不激活全员共享。管理员通过审核台逐条审批后，决定是否发布为全员可见的共享知识。

### 13.2 定时抽取流程

```
第一步：配置抽取任务
  管理员通过 Web Portal「知识抽取」页面配置：
    · 调度时间：选择低峰时段（如每日02:00）
    · 扫描范围：全部用户 / 指定用户组 / 指定时间段
    · 抽取策略：
      · 业务规则类：从对话中提取流程、规范、决策逻辑
      · 客户信息类：从沟通中提取联系人、偏好、历史记录
      · 项目经验类：从工作流产出中提取案例、教训、成功模式
      · 技术方案类：从文档/代码中提取技术选型、架构决策
    · 抽取数量上限（如每批次最多100条候选）

第二步：LLM 批量抽取
  调度器触发 → 从 PGSQL 中读取目标数据：
    → memory_item：对话记忆（含 content_text + summary）
    → document_version：上传文档（含 storage_ref 指向 MinIO）
    → workflow_instance：工作流产出（含 plan.goal + stage_outputs）
    → fact：已提取的事实条目（避免重复抽取）

  LLM 进行批量知识萃取：
    → 分析每段内容的业务价值
    → 提取可结构化的事实三元组（subject→predicate→object）
    → 提取业务规则（条件→动作→适用范围）
    → 提取案例模板（场景→解决方案→效果）
    → 去重：与已有 fact 表比对 confidence 和 supersedes_fact_id

第三步：写入知识库（unconfirmed 状态）
  → fact 表的 status 字段新增 'unconfirmed' 状态
  → scope_type = 'org', status = 'unconfirmed'
  → 关联 source_ref 指向原始对话/文档/工作流
  → 关联 extracted_by = 'scheduled_cron_task'
  → 记录 confidence 评分（LLM 自我评估置信度）

第四步：管理员审核
  管理员通过「知识审核台」逐条查看：
    → 展示知识条目全文（subject → predicate → object）
    → 展示来源上下文（原始对话片段）
    → 展示 LLM 置信度评分
    → 支持操作：
      · 批准共享：status → 'active', scope_type → 'shared'
      · 批准但仅组织内：status → 'active', scope_type → 'org'
      · 退回修改：标记为 'needs_revision'，附修改意见
      · 拒绝：标记为 'rejected'，记录拒绝原因
    → 支持批量审批（按置信度排序，高置信度可批量通过）
```

### 13.3 知识生命周期管理

```
unconfirmed → active (管理员批准后激活)
unconfirmed → rejected (管理员拒绝)
active → deprecated (知识过期)
active → superseded (被新知识取代，supersedes_fact_id 记录)
active → deleted (手动删除或来源文档被删除)
```

### 13.4 调度器实现要点

```
分布式时间窗口设计：
  → 默认抽取时间窗口：每日 2:05 ~ 2:55
  → 每个用户在窗口内被分配一个独立的时间点（如 2:07, 2:12, 2:17 ...）
  → 同一时刻最多 N 个用户并发抽取（N 默认为 3，可按服务器规格调整）
  → 每个用户按固定间隔（如 5 分钟）错峰分配，避免内存和大模型并发瓶颈

管理员可自定义时间窗口：
  → Web Portal「知识抽取→调度配置」页面
  → 修改起止时间（如 1:05 ~ 3:45），扩大窗口以容纳更多用户
  → 修改并发上限 N

技术实现：
  → 复用 org_task 调度器框架（每分钟检查）
  → 新增 task_type = 'knowledge_extraction'
  → schedule_type = 'cron'，按用户分配的时间点触发
  → 触发后调用 fact-retrieval /internal/knowledge/extract 接口

接口处理流程：
  → 查询目标范围的所有 memory_item（按时间窗口过滤）
  → 拼接上下文窗口（利用204,800 tokens大窗口优势）
  → 批量调用 LiteLLM（批量模式，temperature=0.2）
  → 解析返回的结构化知识列表
  → 写入 fact 表（status='unconfirmed'）
  → 返回抽取摘要：{ extracted_count, avg_confidence, batch_id }
```

## 故事线十四：AGE 图查询与企业关系网络

### 14.1 场景描述

企业的业务数据天然具有图结构特征——行业包含多家客户，客户有多个联系人，联系人关联多个项目机会，项目涉及多个合同和交付物。传统的 SQL 布尔逻辑查询难以高效遍历这种多层级的复杂关系网络。通过 PostgreSQL + Apache AGE 扩展，系统支持 Cypher 图查询语言，使管理员和销售人员能够像查询"谁认识谁"、"哪些项目使用了某技术方案"这样直观地遍历关系网络，大幅提升信息检索效率。

### 14.2 AGE 图存储模型

```
图空间 (Graph) 命名规范：
  · knowledge_graph：统一知识图谱（当前实现）
  · 未来可扩展为 org_graph_{org_id}：每个组织独立的图空间
  · shared_graph：跨组织的共享知识图谱

顶点 (Vertex) 标签体系（代码实现 — 12 种）：
  · Entity：通用实体（基础类型）
  · Person：人物
  · Organization：组织/企业
  · Location：地点
  · Event：事件
  · Concept：概念
  · Technology：技术
  · Product：产品/解决方案
  · Client：客户企业（对应业务层 Organization_Client）
  · Contact：联系人（决策者、技术接口人、采购接口人等）
  · Opportunity：项目机会（含金额、阶段、预计成交日期）
  · Project：历史项目（含案例、成果、复盘）

业务层标签映射（文档描述 → 代码实现）：
  · Industry → Concept（行业作为概念实体）
  · Organization_Client → Client
  · Organization_Partner → Organization（合作伙伴作为组织实体）
  · Contract → Entity（合同作为通用实体，可扩展为独立标签）
  · Skill → Entity（技能模板作为通用实体）

边 (Edge) 类型（代码实现 — 16 种）：
  · RELATED：通用关联
  · works_for：人物 → 组织（工作关系）
  · located_in：实体 → 地点
  · uses：实体 → 技术/产品
  · manages：人物 → 项目/团队
  · depends_on：实体 → 依赖实体
  · co_occurs_with：实体 ↔ 实体（共现关系）
  · has_slot_value：实体 → 属性值
  · BELONGS_TO：客户 → 行业
  · EMPLOYES：组织 → 联系人
  · INVOLVED_IN：联系人 → 项目机会
  · INTERACTS_WITH：联系人 ↔ 联系人
  · REPORTS_TO：联系人 → 联系人（汇报关系）
  · PARTNERS_WITH：组织 ↔ 组织（合作关系）
  · COMPETES_WITH：组织 ↔ 组织（竞争关系）
  · SUPPLIES_TO：供应商 → 企业
```

### 14.3 典型图查询场景

```
场景一：人脉引荐（Referral Chain）
  问题：我想接触"XX银行"的CTO，我们公司谁接触过他们？或者谁认识能引荐的人？

  Cypher 查询：
    MATCH path = (me:Contact {user_id: 'xxx'})
      -[:REFERRED_BY*1..3]-(connector:Contact)
      -[:INVOLVED_IN]->(opp:Opportunity)
      <-[:INVOLVED_IN]-(target:Contact {title: 'CTO'})
      -[:EMPLOYS]->(c:Organization_Client {name: 'XX银行'})
    RETURN path, length(path) AS hops
    ORDER BY hops ASC
    LIMIT 5

  返回：
    · 最短引荐路径（几度人脉可达）
    · 中间引荐人的联系方式和过往互动记录
    · 与目标企业的历史合作项目摘要


场景二：项目案例匹配（Case Matching）
  问题：我们正在竞标一个"智慧城市"项目，有过类似成功案例吗？

  Cypher 查询：
    MATCH (opp:Opportunity {industry: '智慧城市'})
    MATCH (past:Project)-[:USES]->(prod:Product)
    WHERE past.status = 'won'
    MATCH (opp)-[:SIMILAR_TO]->(past)
    RETURN past.name, past.revenue, past.lessons_learned,
           collect(prod.name) AS products_used,
           opp.similarity_score
    ORDER BY opp.similarity_score DESC
    LIMIT 10

  返回：
    · 匹配的历史项目列表（按相似度排序）
    · 每个项目使用的产品和方案
    · 项目金额和复盘经验教训
    · 项目核心联系人（可直接咨询）


场景三：客户渗透分析（Account Penetration）
  问题：我们在"XX集团"有哪些联系人？覆盖了哪些部门？还有哪些空白？

  Cypher 查询：
    MATCH (c:Organization_Client {name: 'XX集团'})
      -[:EMPLOYS]->(contact:Contact)
    OPTIONAL MATCH (contact)-[:INVOLVED_IN]->(opp:Opportunity)
    OPTIONAL MATCH (contact)-[:SPONSORED_BY]->(proj:Project)
    RETURN contact.name, contact.department, contact.title,
           count(DISTINCT opp) AS active_opportunities,
           count(DISTINCT proj) AS past_projects
    ORDER BY contact.department, contact.title

  返回：
    · 按部门分组的联系人列表
    · 每个人的活跃商机和历史项目数
    · 标注未覆盖部门（如"财务部无联系人"）


场景四：竞争态势分析（Competitive Landscape）
  问题：在"金融科技"行业，我们和竞争对手各拿下了哪些客户？

  Cypher 查询：
    MATCH (industry:Industry {name: '金融科技'})
      <-[:BELONGS_TO]-(client:Organization_Client)
    OPTIONAL MATCH (client)<-[:COMPETES_WITH]-(competitor:Organization_Partner)
    MATCH (client)-[:EMPLOYS]->(c:Contact)-[:INVOLVED_IN]->(opp:Opportunity)
    RETURN client.name,
           count(DISTINCT opp) AS total_opportunities,
           sum(CASE WHEN opp.stage = 'won' THEN opp.revenue ELSE 0 END) AS won_revenue
    ORDER BY won_revenue DESC

  返回：
    · 行业内所有客户企业的商机分布
    · 已赢单金额排名
    · 竞争关系标注
```

### 14.4 图数据来源与更新

```
图数据自动构建来源：

1. 知识导入自动投影
   → 当用户导入知识并自动抽取实体时
   → entity 表的记录自动同步投影到 AGE 图空间
   → relation 表的关系自动投影为图边
   → projection_event 表记录每次投影操作

2. 对话自动抽取
   → LLM 从对话中识别实体（企业名、人名、产品名）
   → 识别关系（"XX是YY公司的CTO"、"我们上次和ZZ合作过"）
   → 写入 entity + relation 表 → 触发 AGE 投影

3. 工作流产出物解析
   → 销售分析类 workflow 的产出中提取客户、合同信息
   → 技术方案类 workflow 中提取产品和架构组件

4. 管理员手动构建
   → Web Portal 的「图谱管理」页面
   → 可视化添加/编辑顶点和边
   → 支持 CSV 批量导入企业名录和联系人

更新策略：
  → 增量更新：projection_event 记录变更，通过 applied 字段追踪
  → 定时全量刷新：在低峰期从 entity + relation 表全量重建图空间
```

## 故事线十五：销售短任务与长任务智能分流

### 15.1 场景描述

在销售场景中，用户通过飞书与 AI 的交互频率非常高，但大部分交互其实是短平快的查询——"查一下XX客户的联系方式"、"上次和YY开会讨论了什么"、"给我生成一份ZZ项目的周报"。这些请求如果都触发完整的长任务工作流（规划→澄清→检索→决策→报告），会极大浪费用户等待时间和系统资源。系统需要一套精确的分流机制，将真正的复杂分析任务与日常快速查询区分开来。

### 15.2 三级分流体系

```
Level 0：静态规则匹配（< 50ms）
  → 在调用 LLM 之前，先通过关键词和正则快速过滤
  → 明确属于快速查询的请求直接走 Chat 路径：
    · "查一下" + 实体名 → 知识检索型快查
    · "上次"、"之前"、"记得" → 记忆召回型快查
    · "生成" + "报告/周报/日报" → 模板化输出（非长任务）
    · "翻译"、"总结一下这段" → 单轮文本处理
  → 不启动 Planner，不创建 workflow_instance

Level 1：LLM 意图分类（约 2-5s）
  classifyIntentWithLLM() 细化判断：
    → chat：闲聊、问候、简单问答
    → quick_lookup：需要检索知识库但不需多阶段推理
      → 直接调用 retrieval-aware-executor 单轮执行
      → 立即返回结果，不创建持久化 workflow
    → task：真正的多阶段复杂任务
      → 完整的 Planner → Dispatch → Execute → Report 链路

  LLM 分类 Prompt 增强：
    在现有 intent classification prompt 中增加销售场景判断维度：
    · 需要多源信息交叉验证？→ task
    · 需要生成策略建议和风险评估？→ task
    · 只需检索和格式化展示现有数据？→ quick_lookup
    · 单轮即可完成的查询？→ chat

Level 2：用户主动声明（显式路由）
  → 用户可在消息前加前缀明确意图：
    · "/task 分析本季度的销售趋势" → 强制走 task 路径
    · "/find XX客户的最新合同" → 强制走 quick_lookup 路径
  → 用户在对话中自然表达"帮我做个详细分析" → LLM 识别为 task
```

### 15.3 销售场景典型案例区分

```
【快速查询 → chat/quick_lookup】           【长任务 → task workflow】
─────────────────────────────────         ───────────────────────────
"XX公司CTO叫什么？"                        "分析XX公司的决策链和切入策略"
"上次和YY开会讨论了什么？"                  "针对YY公司，制定下季度的销售计划"
"ZZ项目目前到哪个阶段了？"                 "评估ZZ项目的竞品态势和我们的优劣势"
"生成本周的拜访汇总"                       "基于本季度销售数据，预测下季度Pipeline"
"帮我翻译这段英文邮件"                     "帮我起草一份针对金融行业的解决方案白皮书"
"XX产品的报价是多少？"                     "对比分析我们三大产品线在各行业的渗透率"
"提醒我下午2点和YY开会"                    "复盘近期丢单原因，给出改进建议"
```

### 15.4 快速查询的响应优化

```
Chat/quick_lookup 路径的优化策略：

1. 上下文召回增强
   → recallContext 不仅召回对话记忆，还注入：
     · 当前用户的活跃项目列表（从图查询获取）
     · 当前用户的最近联系人（从图查询获取）
     · 组织级共享知识中与查询相关的 Top-5 片段
   → 使 LLM 在回答"查一下XX"时已有充足上下文

2. 结构化快速响应模板
   对于高频销售查询类型，LLM 返回结构化卡片而非自由文本：
   · 客户查询 → { name, industry, contacts: [...], recent_activities: [...] }
   · 项目查询 → { name, stage, revenue, next_action, deadline }
   · 联系人查询 → { name, title, phone, email, last_contacted, relationship_notes }

3. 调用去重与缓存
   → 相同查询在短时间内（如 5 分钟）返回缓存结果
   → Redis 缓存 query_hash → result 映射
   → 缓存命中时添加 "(结果来自缓存，更新于 X 分钟前)" 提示
```

### 15.5 分流统计与调优

```
管理员可在 Web Portal 查看分流统计数据：
  · chat 占比 / quick_lookup 占比 / task 占比
  · 各类别的平均响应时间
  · 分类准确率（如 LLM 判为 quick_lookup 但用户后续补充"详细分析一下"）
  · 按用户维度的使用习惯分析

基于统计数据持续优化路由规则：
  → 发现某类查询总是被误分类 → 更新 Level 0 规则
  → LLM 分类表现不佳 → 优化 classification prompt 和示例
```

## 故事线十六：工作流归档为 Skill 与组织技能库

### 16.1 场景描述

每次成功执行的工作流都是一次可复用的"业务经验"。系统的核心理念是：不让经验随工作流归档而沉寂，而是将其沉淀为可复用的 Skill（技能模板），经过用户本人确认后激活，自动归档到组织的 Skill 库。管理员审核后可以发布给全员使用，形成"执行→沉淀→审核→复用→迭代"的飞轮闭环。

### 16.2 工作流到 Skill 的转化流程

```
第一步：工作流执行成功（status = 'succeeded'）
  → executor-gateway 的 autoExecuteWorkflowStages 完成全部阶段
  → POST /internal/workflows/{ref}/complete
  → workflow 状态机推进到 succeeded

第二步：自动提取 Skill 候选（后台异步）
  SkillExtraction 阶段（在阶段链中增加可选阶段）：
    → 分析完整的 stage_chain + 各阶段输入输出
    → LLM 提取可复用的结构化模板：
      {
        skill_name: "销售数据分析与趋势预测",
        skill_type: "analysis",
        description: "分析指定时间段的销售数据，识别趋势并生成报告",
        task_type_hint: "analysis",
        stage_chain: [
          { stage_type: "IntentClarification", purpose: "明确分析维度和范围" },
          { stage_type: "EvidenceRetrieval", purpose: "检索销售数据和KPI" },
          { stage_type: "DecisionMaking", purpose: "识别趋势和异常" },
          { stage_type: "ResultReporting", purpose: "生成分析报告" }
        ],
        input_parameters: {
          time_period: "分析时间段（如Q1 2026）",
          product_lines: "产品线范围（可选，默认全部）",
          compare_with: "对比基准（同比/环比/预算）"
        },
        expected_output_format: "结构化分析报告",
        tags: ["销售", "数据分析", "趋势", "报告"],
        source_workflow_ref: "wf_xxx"  // 来源追溯
      }

第三步：用户确认（待确认状态）
  → 通过飞书向用户推送 Skill 确认卡片：
    "🎯 你的工作流「销售数据分析」已成功执行！
     系统从中提取了一个可复用的技能模板「销售数据分析与趋势预测」，
     是否激活到你的技能库？"
    → [确认激活] [暂不需要] [修改参数再确认]

  → 用户确认后：
    · skill 表写入记录（scope_type='private', status='active'）
    · skill_version 表写入 v1 定义
    · 关联到原 workflow_instance

第四步：自动归档到组织技能库
  → 用户激活后，系统自动将 Skill 推入组织技能库：
    · 在 skill 表中创建组织副本（scope_type='org', status='pending_review'）
    · 保留来源引用（extracted_from_skill_id → 用户的 private skill）
    · 通知管理员审核队列有新技能待审核

第五步：管理员审核与发布
  管理员通过「技能审核台」：
    → 查看技能名称、描述、阶段链、输入参数
    → 预览技能模板（模拟输入参数看 LLM 规划结果）
    → 编辑优化（调整 prompt 模板、添加文档说明）
    → 操作选项：
      · 批准发布：scope_type → 'public', status → 'active'（全员可见）
      · 仅组织内部：scope_type → 'org', status → 'active'
      · 退回优化：填写修改建议
      · 拒绝：附拒绝原因
    → 发布后，所有组织成员在创建任务时，Planner 可自动匹配到该 Skill
```

### 16.3 Skill 复用与迭代

```
Skill 的匹配与使用：
  用户在飞书提出任务目标 → Planner 阶段：
    → findMatchingSkills() 搜索 skill 表
    → 匹配算法：
      · 关键词匹配（目标描述 vs skill_name + description + tags）
      · 类型匹配（task_type_hint vs skill_type）
      · 语义匹配（embedding 相似度）
    → 匹配到的 Skill 作为 workflow 规划的首选模板
    → LLM 只需在模板基础上微调，而非从零规划
    → 大幅降低规划延迟，提升规划质量

Skill 的版本迭代：
  用户使用某个 Skill 执行工作流后：
    → 如果在执行过程中手动调整了阶段参数
    → 工作流成功完成后，系统识别到"模板变异"
    → 生成本地改进版本（skill_version v2, status='draft'）
    → 用户确认后激活新版本
    → 旧版本保留（通过 skill_version.version 区分）
    → 管理员看到后可以决定是否将改进版提升为组织标准模板

Skill 的评分与推荐：
  → 每次 skill 被匹配和使用，记录 usage_log
  → 统计指标：
    · 使用次数、成功率、平均执行时间
    · 用户满意度评分（工作流完成后邀请用户评分）
  → 高评分 Skill 在匹配时获得加权
  → 管理员可以置顶推荐 Skill（pinned=true）
```

### 16.4 组织技能库全景

```
组织技能库的层次结构：

Level 1: 预制公共技能 (scope_type='public')
  → 系统内置的 15 个预制技能（Document Pro、Google Workspace 等）
  → 来自 ClawHub 技能市场安装的技能
  → 所有组织的所有成员可见和使用

Level 2: 组织共享技能 (scope_type='org')
  → 从本组织成员工作流中沉淀的技能（经管理员审核发布）
  → 本组织内所有成员可见和使用
  → 带有组织特有的业务适配（如「XX银行投标分析模板」）

Level 3: 个人私有技能 (scope_type='private')
  → 成员个人从工作流中激活的技能
  → 仅本人可见和使用
  → 可以随时"捐献"到 Level 2（提交组织审核）

技能的生命周期流转：
  private ──捐献──→ org ──管理员推荐──→ public (ClawHub 发布)
                                                      ↓
                                             其他组织通过 ClawHub 安装
```

### 16.5 与工作流规划器的集成

```
Planner 内部集成的 Skill 优先逻辑（已有代码基础）：

plan() 方法中的优先级：
  1. markdown_steps（用户显式指定的步骤 → 最高优先级）
  2. matchedSkills（匹配到的 Skill 模板 → 优先使用）
  3. LLM 生成（无匹配时从零规划 → 兜底）

当 Skill 匹配成功时：
  → buildPlanFromSkill() 将 Skill 的阶段链转换为 WorkflowPlan
  → 标记 source='skill_match' 在审计日志中
  → 用户无需重复描述已在 Skill 中固化的最佳实践

当 Skill 匹配但用户需求有差异时：
  → LLM 在 Skill 模板基础上进行适应性修改
  → 标记 source='skill_adapt' 
  → 保留 Skill 的核心架构，仅微调参数和上下文
```

## 故事线十七：技能公网安装与多路检索 LLM 决策

### 17.1 场景描述

组织成员在 ClawHub 技能市场检索和安装公开技能时，系统默认通过多个国内镜像站点并发检索，将多路返回的候选技能列表交由大模型横向对比，自动推荐最佳匹配的技能进行安装。对于需要 API Key 的技能（如调用第三方付费服务），个人用户无权限安装，必须由管理员统一配置后，以组织级共享技能的方式下发给全员使用。

### 17.2 多路检索与 LLM 决策流程

```
第一步：用户发起技能搜索
  用户在飞书输入：
    "/install 一个能做竞品分析的技能"
  或通过 Web Portal「技能市场」页面搜索

第二步：多路并发检索
  网关同时请求多个检索源：

  请求1 → 国内镜像站
    GET https://mirror-cn.clawhub.com/api/skills/search?q=竞品分析
  请求2 → 官方源（兜底）
    GET https://mirror-cn.clawhub.com/api/skills/search?q=竞品分析

  → 全部请求设置 10 秒超时
  → 哪个先返回就用哪个的结果进候选池
  → 最多收集 N 个源的返回（去重合并）

第三步：LLM 横向对比决策
  将多路返回的候选技能列表提交给 LiteLLM：

  Prompt：
    "以下是来自多个技能市场的搜索结果。请比较这些技能候选，
     根据以下维度选出最匹配的一个：
     1. 功能覆盖度：是否完整覆盖用户需求
     2. 质量评分：star 数、下载量、评分
     3. 兼容性：是否与当前系统兼容（stage_chain 完整性）
     4. 维护活跃度：最近更新时间
     5. 安全性：是否包含可执行代码或外部 API 调用

     用户需求：{user_query}
     候选列表：{merged_candidates_json}

     返回 JSON：{ selected_skill_id, reason, confidence, alternative_ids }"

  → LLM 返回决策结果
  → 向用户展示推荐理由（如 "选择了来自镜像站B的「Competitor Analyzer v2.3」，理由：..." ）

第四步：安装前权限检查
  检查技能类型：
    · type='public_free' → 个人可直接安装
    · type='requires_api_key' → 阻止安装，提示：
      "该技能需要配置第三方 API Key，个人无法直接安装。
       请联系管理员在「组织配置→API密钥管理」中统一配置后下发。"
    · type='org_managed' → 仅管理员可安装
```

### 17.3 API Key 技能的统一管理

```
管理员操作流程：

1. 配置 API Key
   → Web Portal「组织配置→API密钥管理」页面
   → 添加密钥：名称、服务商（如"SerpAPI"、"NewsAPI"）、Key 值
   → 密钥加密存储（AES-256），仅运行时解密注入
   → 按技能维度关联：哪些技能可使用此 Key

2. 安装 API Key 技能
   → 管理员在技能市场中搜索到 requires_api_key 类型的技能
   → 安装时选择关联已配置的 API Key
   → 安装后 scope_type='org'，全组织成员可见
   → 成员使用时，系统自动注入组织共享的 API Key

3. 审计与安全
   → API Key 的使用频率和调用量在 audit_event 中记录
   → 管理员可设置调用上限（如每日最多1000次）
   → Key 泄露风险：运行时注入，不落盘到工作流日志
```

### 17.4 镜像站容错与降级

```
多路检索的容错策略：

正常模式：
  → 3 个源并发请求
  → 任意一个在 10 秒内返回 → LLM 对比决策

部分降级（1个源超时）：
  → 用已返回的 2 个源的结果进行对比

完全降级（全部超时）：
  → 降级到官方源，增加超时到 30 秒
  → 如仍失败，向用户返回："技能市场暂时不可用，请稍后重试"

镜像站健康监控：
  → 每次请求记录各镜像站响应时间
  → 持续超时的镜像站自动标记为 degraded
  → 管理员可在配置中新增/移除镜像站地址
```

## 故事线十八：管理员下发智能体人设与用户画像自治

### 18.1 场景描述

在组织场景中，智能体的行为风格和核心价值观由管理员统一管控——每个用户的智能体人设（soul、identity、性格、语气风格、行为边界）由管理员集中设定，用户个人无权修改。但用户自己的画像数据（姓名、职位、技能标签、偏好、工作习惯等），由用户本人的智能体根据日常交互自动总结和更新，形成持续演进的动态用户画像，这部分用户有完全自主权。

### 18.2 智能体人设的层级模型

```
┌──────────────────────────────────────────────────────┐
│  Level 1: 系统基座人设（管理员设定，全局生效）         │
│  · 基础行为准则：诚实、合规、安全                        │
│  · 通用的语气风格：专业、友善                          │
│  · 能力边界声明：我能做什么/不能做什么                   │
│  · System Prompt 基础模板                             │
└──────────────────────────────────────────────────────┘
                        ↓ 继承并覆盖
┌──────────────────────────────────────────────────────┐
│  Level 2: 组织人设（管理员设定，本组织生效）           │
│  · 组织身份：智能体代表哪个组织                         │
│  · 行业知识偏好：如"你是金融行业的销售助理"              │
│  · 组织价值观与禁忌：如"不讨论竞品负面信息"              │
│  · 组织级 System Prompt 片段                          │
└──────────────────────────────────────────────────────┘
                        ↓ 继承并覆盖
┌──────────────────────────────────────────────────────┐
│  Level 3: 用户人设（管理员为每个用户设定，用户不可自行修改）  │
│  · 角色定位：如"你是张三的销售助理"                     │
│  · soul：核心性格描述（如"细致耐心、善于数据分析"）       │
│  · identity：身份定位（如"资深B2B销售顾问"）            │
│  · 语气风格：专业度、亲和度、简洁度                      │
│  · 行为边界：什么话题可以展开/什么话题需要回避            │
│  · 响应偏好：主动建议型 / 被动响应型                    │
│  · 用户专属 System Prompt 片段                        │
└──────────────────────────────────────────────────────┘
```

### 18.3 管理员配置流程

```
管理员通过 Web Portal「智能体人设管理」页面：

创建用户人设模板：
  → 输入 soul 描述（如"你是一个温暖但严谨的销售教练，
       善于从数据中发现问题，用提问引导销售自我反思，
       而非直接给出答案。"）
  → 输入 identity 描述（如"你有15年B2B大客户销售经验，
       曾在IBM和Oracle担任过区域销售总监。"）
  → 选择语气风格滑块：
     · 专业度：████░░░ 60%
     · 亲和力：███████░ 80%
     · 简洁度：███░░░░ 30%  （越长越详细）
  → 设置行为边界：
     · 允许主动建议：☑ 是
     · 允许情感陪伴：☑ 是
     · 禁止讨论话题：✎ 竞品贬低, 政治观点, ...
     · 必遵合规准则：✎ GDPR, 个人信息保护法, ...

分配人设给用户：
  → 从用户列表中选择目标用户
  → 关联已创建的人设模板
  → 或为一组用户批量下发同一人设
  → 用户侧的人设字段变为 readonly

用户端体验：
  → 用户可在个人页面查看自己的人设描述（只读）
  → 灰色锁定标识："此人设由管理员统一管理"
  → 如有需求，可提交"人设修改申请"（附带理由）
  → 管理员审核后决定是否变更
```

### 18.4 用户画像自治

```
与管理员管控的人设不同，用户画像由用户本人完全掌控：

用户画像包含：
  · 基础信息：姓名、部门、职位
  · 技能标签：如 ["大客户销售", "金融行业", "招投标"]
  · 工作偏好：如 偏好上午做深度分析，下午做客户拜访
  · 沟通风格：如 喜欢结论先行再展开细节
  · 当前焦点：如 本月重点在"XX银行"项目推进

画像的自动演进：
  智能体根据日常互动自动提取和更新：
    → 对话中识别用户提到的关键词："我最近在跟进XX项目"
    → 自动添加 '当前焦点': 'XX项目'
    → 用户确认后生效

    → 从成功工作流中识别用户擅长的领域
    → 自动更新技能标签
    → 用户确认后生效

画像的控制权：
  → 用户可在 Web Portal「我的画像」页面随时手动修改
  → 用户也可以完全关闭自动画像功能
  → 画像数据仅用户本人和管理员可见
  → 管理员可见但不可修改（只读视角）
```

### 18.5 人设与画像在对话中的注入

```
gateway-adapter 构造 LLM 消息时的拼接顺序：

  system_prompt =
    Level 1 基座人设（平台级，不可被覆盖的核心）
    + Level 2 组织人设（行业、价值观）
    + Level 3 用户人设（soul、identity、语气、边界）
    + 用户画像摘要（动态，当日有效）

  user_prompt =
    压缩后的对话上下文
    + 当前消息

  人设的生效时机：
    → 首次对话：注入全套人设
    → 多轮对话：仅注入压缩版人设摘要（节省 token）
    → 工作流启动：注入人设摘要到 planner input
```

## 故事线十九：梦境——每日记忆总结与压缩归档

### 19.1 场景描述

每天夜间，系统为每个活跃用户运行一次"梦境"任务——扫描当日所有对话日志和记忆条目，进行三层处理：① 提取有价值的知识和技能候选并归档；② 对超过 4000 字符的长篇记忆进行压缩，核心内容转为 Markdown 文件持久化存储（MinIO），原记忆条目替换为摘要 + 文件链接；③ 将当日的碎片记忆压缩为一份"今日摘要"，帮助用户次日快速回顾昨天发生了什么。

### 19.2 梦境执行流程

```
触发条件：
  → 每个用户的「梦境」在用户当日最后一条消息产生后
    · 加上一个冷却窗口（如 2 小时无新消息）自动触发
    · 或固定在凌晨 3:00 对所有当日活跃用户批量执行

梦境三步走：

┌─ 第一步：收集 ─────────────────────────────────────┐
│  查询当日该用户的所有 memory_item：                    │
│    → 按时间窗口（今日 00:00 ~ 当前时间）               │
│    → 按重要性排序（含 embedding 相关度、是否被引用）     │
│    → 收集关联的 document（今日上传的文档）             │
│    → 收集关联的 fact（今日新增的知识条目）             │
└────────────────────────────────────────────────────┘

┌─ 第二步：压缩 ─────────────────────────────────────┐
│  对每个 memory_item 进行字符数判断：                   │
│                                                      │
│  · len(content_text) < 4000 → 保留原文              │
│  · len(content_text) >= 4000 → 触发压缩：            │
│                                                      │
│    1. 调用 LiteLLM 生成摘要（2000 字符以内）          │
│       Prompt: "请将以下对话记录压缩为不超过2000字的摘要，│
│                保留核心决策、关键数据和行动项"          │
│                                                      │
│    2. 原文保存为 Markdown 文件 → MinIO                │
│       key: memories/{org_id}/{user_id}/{date}/{memory_id}.md │
│                                                      │
│    3. 更新 memory_item：                              │
│       · content_text = 摘要文本                      │
│       · compressed = true                            │
│       · archived_file_ref = "minio://memories/...md" │
│       · original_char_count = 12000                  │
│       · compressed_char_count = 1856                 │
│                                                      │
│  类似 Hermes 的 2000 字符上限，本系统采用 4000 字符     │
│  阈值，给日常对话提供更宽裕的压缩窗口                   │
└────────────────────────────────────────────────────┘

┌─ 第三步：抽取 ─────────────────────────────────────┐
│  LLM 分析压缩后的记忆集合，抽取：                       │
│                                                      │
│  1. 知识条目（→ fact 表）                             │
│     → 业务规则、客户偏好、项目决策等                  │
│     → status='unconfirmed'，待管理员审核             │
│                                                      │
│  2. 技能模板（→ skill 表）                            │
│     → 今日工作流中形成的可复用方法                    │
│     → scope_type='private', status='draft'          │
│                                                      │
│  3. 今日摘要（→ hermes daily_summary）               │
│     → "今天你做了3件事：                               │
│        1. 分析了Q1销售数据并输出了趋势报告              │
│        2. 整理了XX客户的项目信息                      │
│        3. 提交了每日拜访总结"                         │
│     → 次日用户发第一条消息时，自动推送昨日摘要          │
└────────────────────────────────────────────────────┘
```

### 19.3 记忆文件归档机制

```
归档文件结构（MinIO 存储）：

  memories/
    {org_id}/
      {user_id}/
        {yyyy-MM}/
          {yyyy-MM-dd}_dream_summary.md   ← 每日梦境总结
          {memory_id}_original.md         ← 被压缩的原记忆文件
          {memory_id}_extracted_facts.md  ← 抽取的知识条目记录


文件链接在记忆中的形态：

  压缩前：
    memory_item.content_text = "（12000 字长篇对话记录...）"

  压缩后：
    memory_item.content_text = "（1856 字摘要文本）"

    + memory_item.archived_file_ref = "minio://.../{memory_id}_original.md"

  用户在对话中引用该记忆时：
    智能体回复："关于那次讨论，我已将详细记录归档，
                [点击查看完整记录]({archived_file_ref})"
```

### 19.4 字符阈值与触发机制

```
阈值设计参考：

  Hermes 默认上限：2000 字符（短记忆模型）
  本系统阈值：4000 字符（长记忆模型，更好保留上下文）

触发逻辑：

  INSERT / UPDATE memory_item 时：
    if (new_content_text.length >= 4000 && !item.compressed) {
      // 标记为 "待压缩"，但不阻塞当前写入
      item.needs_compression = true;
    }

  梦境批量处理时：
    SELECT * FROM memory_item
    WHERE needs_compression = true
      AND created_at >= today_start
      AND owner_user_id = $user_id

    批量压缩 → 写入归档文件 → 更新摘要 + 清除 needs_compression


强制压缩（管理员操作）：
  → 管理员可对某个用户的全部记忆执行批量压缩
  → Web Portal「记忆管理」页面 → 「压缩全部记忆」
  → 适用于存储占用过高的场景
```

## 故事线二十一：B2B 销售管理日常与工作流可观测闭环

### 21.1 场景总述：老板只做决策，Agent 承担经营节奏

一家 B2B 软件公司有 28 名销售，分成华东、华南、北方三个小组。老板每天真正有时间处理销售的窗口只有早上 20 分钟和晚上 15 分钟，他不想打开一堆 CRM 页面，也不想听每个销售逐条报流水账。他只关心三件事：本周目标有没有跑偏，哪些客户需要他拍板，哪些异常如果今天不处理会影响成交或回款。JueYing 在这里的角色不是一个被动问答机器人，而是一个经营节奏助手：老板提出经营意图，Agent 先查是否已有被确认的销售管理 workflow；如果有，就按既有路径执行；如果没有，就自主生成路径，执行中自己处理缺数据、阶段失败、证据不足等异常，最终把过程、结果和需要老板决策的点推回来。

这个故事线以 B2B 销售管理为基准，采用“老板决策、经理盯过程、销售做动作、Admin 治理规则、运维保障可观测、开发保证路径可演进”的逻辑。它不是“系统支持工作流”这种技术描述，而是一组日常使用场景：周一晨会、每日八访、卡单救援、折扣审批、回款追踪、周五复盘，以及第一次做新任务后如何沉淀成下次可复用的 workflow。

### 21.2 老板视角：周一 8:30 的经营驾驶舱

周一早上 8:30，老板在企微里发一句话：“本周把华东区回款风险降下来，两个重点客户必须推进到 closing，告诉我今天要盯谁。”系统收到后，第一步不是立刻让大模型自由发挥，而是先在个人私有 workflow、组织 workflow、公共技能中查找是否有“周销售经营复盘”“回款风险排查”“重点商机推进”之类的已确认路径。若上周老板已经确认过“销售漏斗晨会复盘 workflow v1”，系统会直接匹配它，并在回复中说明“已沿用你上次确认的销售漏斗复盘路径”。若没有命中，Planner 会生成首跑路径：读取销售目标、召回本周 pipeline、按客户阶段和最后跟进时间打红黄绿、识别回款和折扣异常、输出老板只需决策的清单。

老板收到的不是表格堆叠，而是一份可行动简报：第一段告诉他本周目标差距，比如“华东区本周需新增 42 万回款，当前确定性回款 18 万，缺口 24 万”；第二段列出三条红色异常，比如“A 客户合同已签 16 天未回款，销售承诺本周三到账但财务未见付款凭证”；第三段列出需要老板拍板的事项，比如“B 客户要求 12% 折扣，低于毛利红线 4 个点，建议只批准 8%，并要求客户本周五前付款”。老板只需要回复“同意 B 客户 8%，A 客户让销售经理今天 17 点前给回款证据”，系统就把这些决策拆成销售经理和销售个人的执行任务，并继续盯完成状态。

### 21.3 销售经理视角：9:00 晨会和 20:30 夕会

销售经理每天 9:00 打开销售群，不再先问“大家今天打算干什么”。Agent 已经根据每日八访节奏生成个人动作清单：新人优先 4 个新客户触达，潜力销售补 2 个二访，核心销售集中推进 2 个 closing 客户。经理看到的是团队地图：谁今天缺新线索，谁的关键客户 7 天没联系，谁的 proposal 卡了 10 天，谁昨天承诺给客户方案但没有上传证据。经理可以直接在群里下发：“今天华东组先处理红色客户，张三 10 点前联系 C 客户 CFO，李四补 D 客户 ROI 测算，王五下午约 E 客户采购负责人。”

晚上 20:30，经理不再逐个翻聊天记录。系统汇总当天承诺动作和实际证据：哪些电话有记录，哪些拜访有纪要，哪些报价有附件，哪些客户推进了阶段。一个销售说“客户说下周再看”，Agent 会追问关键缺口：客户是哪类决策风格，是预算没确认、决策人缺席，还是竞品压价？如果销售只写了模糊记录，系统会把该条标成黄色，要求补全下一步时间和责任人。经理的夕会因此从“听汇报”变成“处理例外”：只讨论红色卡单、重要折扣、回款风险和需要资源支持的客户。

### 21.4 一线销售视角：客户跟进不靠记忆，下一步不靠猜

一线销售小周上午 10:20 刚打完 A 客户电话，他在飞书里发：“记录 A 客户，采购认可方案，CFO 要看预算，下周三前给反馈，我今晚发 ROI 摘要。”系统把这句话拆成客户阶段变化、下一步、承诺时间、涉及角色和风险点：A 客户从 `presented` 进入 `proposal`，关键决策人 CFO 尚未确认，下一步是今晚发送 ROI 摘要，若 24 小时没有上传或发送证据，系统会提醒小周和经理。第二天如果小周问“今天先联系谁”，系统不会给一堆客户名单，而会按温度排序告诉他：“先打 A 客户 CFO，因为你昨天承诺发 ROI；再跟 B 客户采购确认评审会时间；新客户触达还差 2 个，建议补制造业名单里的 F 和 G。”

当小周遇到卡单，他可以直接说：“D 客户 proposal 后 12 天没回，怎么办？”系统先查是否已有“卡单诊断 workflow”。如果没有，就自主生成诊断路径：查看最近跟进记录、判断客户类型、检查是否缺决策人、预算、时间压力或商业价值证明，再给出下一步话术和补证据动作。比如系统判断 D 客户是偏分析型，建议小周补一页完整价格拆解和竞品对照，并把“周四 15:00 前发送技术/财务明细”写成待办。执行结果如果有效，小周确认这条路径后，下次类似“proposal 后没回应”的场景就会优先沿用。

### 21.5 Admin 视角：把销售管理方法变成组织规则

Admin 不参与每个客户沟通，但他要把组织管理规则配置清楚：销售目标分层、角色权限、知识审核、workflow 审核、折扣红线、回款证据要求、技能是否可提升为组织通用路径。比如公司决定“低于 35% 毛利的报价必须老板审批”“超过 14 天未回款必须红色预警”“所有客户采购流程信息必须进入知识审核池”，Admin 在 Web Portal 里配置这些规则，并确认销售团队的智能体身份统一为“严谨但不替代人拍板的 B2B 销售教练”。

当某个销售经理第一次跑完“月度成交率下降原因分析”并确认效果好，系统会生成一个私有 draft workflow。Admin 在技能审核台看到它的来源、阶段链、样例产出、适用场景和风险说明：它是否读取了正确的组织数据，是否要求用户确认关键假设，是否把客户隐私限制在组织内，是否能对多个团队复用。通过审核后，这条路径被提升为组织 workflow，其他经理下次提出“分析本月销售漏斗为什么掉了”时，系统优先用组织确认版，而不是重新让模型临时想办法。

### 21.6 工作流可观测故事：首次任务如何变成可复用 workflow

周三下午，销售总监发起一个新任务：“分析本月销售漏斗为什么成交率下降，给出明天晨会的主管动作。”这是第一次出现的管理问题。系统先查找既有 workflow：个人私有库没有，组织库没有，公共技能只有泛化的销售分析技能，匹配分不足以直接复用。于是系统进入首跑模式，自主生成执行路径：先澄清分析口径，再检索本月线索、阶段停留、拜访记录、报价和回款数据，然后识别漏斗掉点，最后输出晨会动作。

执行过程中，EvidenceRetrieval 阶段发现部分销售没有填写丢单原因。系统不会直接失败，而是把缺口标成黄色，使用可用字段先做初步诊断，同时在结果里明确说明“丢单原因字段覆盖率不足，只能作为辅助判断”。如果某个阶段执行失败，Executor 会尝试一次自主修复，把原失败、修复动作和修复后产出记录到过程里。最终用户收到的完成消息包含“执行过程”：检索了哪些证据、哪些阶段完成、哪些异常被修复、哪些假设需要人工确认，以及最终建议的晨会动作。

销售总监看完后觉得这条路径符合团队习惯，回复“确认工作流 wf_xxx”。系统把该次成功路径激活为他的私有 workflow，并保留来源、阶段链、过程摘要和确认时间。下个月他再次提出类似问题时，系统会先匹配这条已确认 workflow；如果执行过程中出现新的变体，比如这次重点不是成交率而是回款率，系统会沿用主体路径，同时把差异记录为可演进版本，等待用户确认后再升级。

### 21.7 运维与开发视角：用故事线做发布基准

运维验收这条故事线时，不只看服务是否启动，还要看销售任务从渠道入口到最终回执是否能被追踪。核心检查包括：飞书/企微事件可异步确认，重复事件不会重复执行，Workflow `/progress` 返回阶段状态和过程摘要，Supervisor 能看到心跳，完成消息能说明过程与结果，缺凭据时能明确区分“外部渠道不可用”和“核心工作流故障”。新增的 `smoke:workflow-observability` 用来固定这条产品路径，确保以后不会把用户确认、过程摘要、私有/组织技能匹配或自主修复误删。

开发者接手需求时，以这条销售故事线作为代码基准：任何改动都要回答“老板能不能只看异常和决策点”“经理能不能围绕红黄绿处理例外”“销售能不能得到下一步动作”“用户能不能看见系统怎么完成任务”“确认后的路径下次是否真的优先复用”。因此发布前检查不再只是编译和单测，还包括 lint、type-check、unit test、M0 校验、context audit、渠道烟测、评测烟测和 workflow 可观测烟测。

| 角色 | 真实故事线 | 验收重点 | 本轮覆盖 |
|------|------------|----------|----------|
