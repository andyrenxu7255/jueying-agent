# 代码级架构评估

对 Agent Harness V1 项目 (~9000 行 TypeScript) 进行源码级扫描，发现 13 项可优化点。

## 冗余代码 (RED)

### RED-C001: gateway-adapter 渠道代码 80% 重复
- **位置**: `apps/gateway-adapter/src/index.ts` L1239-1592
- **问题**: Feishu 和 Wecom 的 getAccessToken / sendTextMessage / downloadFile 函数逻辑几乎相同
- **数据**: 4 对函数平均重复率 80%，约 200 行冗余
- **方案**: ChannelAdapter 接口抽象 + FeishuAdapter / WecomAdapter 实现

### RED-C002: 6 个服务独立创建 DB Pool
- **位置**: hermes-adapter/db.ts, fact-retrieval/db.ts, workflow/persistence/db.ts, hermes-adapter/index.ts 等
- **问题**: 每个服务各自 new Pool()，无统一配置管理
- **方案**: PoolManager 集中管理

### RED-C003: 5 个意图分类函数重复
- **位置**: `apps/gateway-adapter/src/index.ts` L197-368
- **问题**: isTaskIntent / isTaskDispatchIntent / isKnowledgeSubmitIntent / isQuickLookupIntent + classifyIntentWithLLM 共 170 行
- **方案**: IntentClassifier 模块

### RED-C004: generic-executor / verification-executor / repair-executor 3 份 callLiteLLM
- **位置**: 3 个 executor 文件
- **问题**: 相同的 fetch + AbortController + JSON 解析代码
- **方案**: LlmClient 统一客户端

### RED-C005: 8 个服务重复 bootstrap 代码
- **位置**: 各服务 index.ts 中 createServer / health check / graceful shutdown
- **问题**: 每个服务 ~30 行相同模式代码
- **方案**: createMicroService 工厂函数

## 性能问题 (PERF)

### PERF-C001: LLM Intent 分类阻塞事件循环
- **问题**: 每次用户消息同步等待 LLM 响应 (60s timeout)
- **方案**: 规则先行分类 (confidence >= 0.85 跳过 LLM)

### PERF-C002: memoryStore / notificationStore 无 TTL
- **位置**: hermes-adapter MemoryEntry[], mobile-app PushNotification[]
- **问题**: Map 只增不减，存在内存泄漏
- **方案**: TTLMap

### PERF-C003: verifyFeishuSignature 使用 timingSafeEqual
- **问题**: 已经使用了安全的比较，但签名验证依然每次重复计算
- **方案**: 缓存已验证的 nonce (已存在于 gateway-state.ts)
