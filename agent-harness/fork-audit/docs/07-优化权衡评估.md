# 优化权衡评估

## 损益矩阵

| 优化项 | 得到 (Gain) | 失去 (Loss) | 风险等级 |
|--------|-------------|-------------|----------|
| **TTLMap** | 内存泄漏修复, 自动过期 | 无, 100% 兼容 Map API | 低 |
| **LlmClient** | 消除 3 份重复代码, 统一超时/重试 | 各 executor 独立 timeout 配置消失 | 低 |
| **PoolManager** | 统一连接池管理, 优雅关闭 | 单服务异常不影响其他服务池 (极小) | 低 |
| **ChannelAdapter** | 消除 ~200 行重复, 新增渠道只需实现接口 | 原代码与 gateway-state 缓存耦合, 适配器内置缓存略有不同 | 中 |
| **createMicroService** | 消除 ~240 行 boilerplate | 各服务的定制路由需要额外配置参数 | 低 |
| **Unified Executor** | 6 个 executor 合并为 1 个 | 需引入 mode 分发, 单文件变大, 违背单一职责 | 中 (已推迟) |
| **IntentClassifier** | 减少 ~30% LLM 调用, 消除 170 行重复 | 规则误判 (影响较小, fallback 兜底) | 低 |
| **gateway-adapter 模块化** | 提高可维护性 | 需要重构 2000+ 行文件, 影响面大 | 高 (已推迟) |

## 结论

**低风险高收益 (立即执行)**: TTLMap, LlmClient, PoolManager, ChannelAdapter, createMicroService, IntentClassifier
**中风险 (慎重执行)**: Unified Executor (需要保留原文件做兼容层)
**高风险 (推迟)**: gateway-adapter 模块化拆分
