# 文档 13：仓库复用与改造边界清单 v1.0

## 13.1 文档目的

本文件用于把《Agent Harness V1 实施规格包》里的“参考 OpenClaw / opencode / claw-code / Hermes / PostgreSQL 生态源码”进一步收敛为可执行的仓库级决策，避免后续开发智能体把“阅读源码”误解为“直接把现有仓库拼起来”。

本文件回答三个问题：

1. 哪些仓库是 Day 1 直接复用目标。
2. 哪些仓库只作为模式参考，不直接嵌入。
3. 平台主仓库应该承载哪些新代码。

## 13.2 总体决策

### 13.2.1 主仓库策略

Day 1 推荐采用“新主仓库 + 外部仓库参考/适配”的路线。

- 新主仓库承载：Workflow 治理、Fact / Retrieval、Policy、Audit、Adapter、Portal、数据库迁移、PoC 脚本。
- `openclaw` 主要提供渠道接入、Gateway、会话入口、配对、安全默认值与控制面模式参考。
- `opencode` 主要提供 Code Executor 需要的 project / session / worktree / patch / tool 协议参考。
- `claw-code` 主要提供开发执行闭环、权限模式、恢复策略的参考实现。
- `hermes-agent` 主要提供 memory / dream / skill / checkpoint / session context / code execution 的增强型实现参考。
- `pgsql`、`pgvector`、`AGE` 作为基础设施与能力来源，不作为应用仓库直接 fork 开发对象。

### 13.2.2 复用分类定义

| 分类 | 含义 | 适用情形 |
|---|---|---|
| 直接复用 | 直接以库、协议、脚本、子进程或明确导入方式接入 | 成本低、边界清晰、升级风险可控 |
| 薄适配 | 保留外部模块核心行为，外围包一层 Adapter / Facade / Contract | 行为可用，但平台上下文、策略、审计需要补齐 |
| 参考重写 | 只复用设计思路、对象模型或执行经验，不直接集成代码 | 语言栈不同、内聚过高、产品语义不匹配 |
| 禁止复用 | 明确不进入 Day 1 范围 | 会冲突主边界，或引入第二事实源 / 第二总控 |

## 13.3 目标主仓库建议结构

Day 1 推荐主仓库目录结构如下：

```text
agent-harness/
  apps/
    gateway-adapter/
    web-portal/
  services/
    workflow/
    fact-retrieval/
    executor-gateway/
    hermes-adapter/
  libs/
    contracts/
    policy/
    audit/
    shared/
  db/
    migrations/
    queries/
    age/
  pocs/
  docs/
```

说明：

- `apps/gateway-adapter` 负责接住 OpenClaw 风格渠道消息并标准化为平台请求。
- `services/workflow` 是统一任务治理内核。
- `services/fact-retrieval` 负责结构化、全文、向量、图增强与 Evidence Pack。
- `services/executor-gateway` 负责通用执行器与 Code Executor Adapter。
- `services/hermes-adapter` 负责把 Hermes 的 memory / dream / skill 输出治理化。
- `libs/contracts` 放稳定 JSON schema、事件字典、错误码与 DTO。
- `db` 放 PostgreSQL 逻辑落地、索引、AGE 投影脚本。

## 13.4 仓库级复用边界

### 13.4.1 OpenClaw

已观察到的关键触点：

- `src/channels/`
- `src/gateway/`
- `src/sessions/`
- `src/routing/`
- `src/tasks/`
- `docs/concepts/architecture.md`

边界决策如下：

| 仓库 | 模块 | 动作 | 分类 | 原因 |
|---|---|---|---|---|
| openclaw | `src/channels/` | 复用渠道标准化、会话 envelope、线程绑定、allowlist、mention gating 思路 | 薄适配 | 与多渠道接入高度相关，但要改成平台 `user_id` / `policy_snapshot` 语义 |
| openclaw | `src/gateway/` | 复用 Gateway WS 控制面、连接鉴权、配对、安全默认值模式 | 薄适配 | 很适合作为接入与总控壳层，但不能直接承担 Workflow 治理 |
| openclaw | `src/sessions/` | 复用 session key、session lifecycle、transcript event 思路 | 薄适配 | 对 `session_ref -> workflow` 映射有帮助 |
| openclaw | `src/routing/` | 复用 route resolution、bound account、session continuity 思路 | 参考重写 | 平台要引入 `channel_identity -> user_id -> workflow` 新链路 |
| openclaw | `src/tasks/` | 只参考 owner access、task registry、audit pattern | 参考重写 | 现有任务语义不是本平台的 Workflow 状态机 |
| openclaw | memory / plugin / skill 体系 | 仅参考扩展机制，不作为主记忆底座 | 禁止直接复用为真相层 | 与本平台冻结边界冲突，容易把 OpenClaw 自身能力误当平台核心 |

明确禁止：

- 禁止把 OpenClaw 原生 task / memory / cron 体系直接当成平台 Workflow 主治理层。
- 禁止把 OpenClaw 自带 workspace/skills 结构直接当成企业共享资产真相源。

### 13.4.2 opencode

已观察到的关键触点：

- `packages/opencode/src/project/`
- `packages/opencode/src/session/`
- `packages/opencode/src/worktree/`
- `packages/opencode/src/patch/`
- `packages/opencode/src/tool/`
- `packages/opencode/src/server/`
- `specs/project.md`

边界决策如下：

| 仓库 | 模块 | 动作 | 分类 | 原因 |
|---|---|---|---|---|
| opencode | `src/project/` | 参考 project / directory / VCS / bootstrap 模型 | 薄适配 | 适合映射平台 `repo_ref`、`workdir_ref`、`branch_ref` |
| opencode | `src/session/` | 参考 session、prompt、run-state、retry、summary、todo 思路 | 薄适配 | 对 Code Executor 执行会话设计价值高 |
| opencode | `src/worktree/` | 复用工作树隔离思路 | 参考重写 | 平台需要绑定 Workflow、checkpoint、artifact、policy |
| opencode | `src/patch/` | 复用 patch 生成/应用协议 | 直接复用协议或薄适配 | 可直接服务于 patch refs 与 diff artifact |
| opencode | `src/tool/` | 复用工具协议面与工具注册模式 | 薄适配 | 可作为 Code Executor 工具面设计参考 |
| opencode | `src/server/` | 参考控制面、adapter、event 路由 | 参考重写 | 平台自身还有 Workflow、Fact、Policy、Audit 额外语义 |
| opencode | `packages/app`、桌面、UI | 不纳入 Day 1 | 禁止直接复用 | 与平台关键链路无关，会稀释实现重心 |

推荐结论：

- Day 1 可以优先从 `opencode` 学习和抽象 Code Executor 的会话模型、工具面、patch 与 worktree 设计。
- 不建议 Day 1 把整个 `opencode` 作为平台内嵌开发内核强行集成。

### 13.4.3 claw-code

已观察到的关键触点：

- `rust/crates/runtime/src/`
- `rust/crates/tools/src/`
- `rust/crates/commands/src/`
- `USAGE.md`
- `PHILOSOPHY.md`

边界决策如下：

| 仓库 | 模块 | 动作 | 分类 | 原因 |
|---|---|---|---|---|
| claw-code | `runtime/src/permissions.rs`、`permission_enforcer.rs` | 参考权限模式与执行限制 | 参考重写 | 与平台 `policy_snapshot` 强相关，但语言栈不同 |
| claw-code | `runtime/src/session.rs`、`session_control.rs` | 参考执行会话与控制策略 | 参考重写 | 对 Code Executor session lifecycle 有价值 |
| claw-code | `runtime/src/recovery_recipes.rs` | 参考修复与恢复策略 | 参考重写 | 可转化为平台 `Repair` 与恢复规则 |
| claw-code | `runtime/src/task_registry.rs` | 参考任务包与 registry 思路 | 参考重写 | 不直接映射平台 Workflow 表模型 |
| claw-code | CLI 二进制 | 可作为后续可选 backend | 薄适配 | 适合通过子进程 adapter 接入，不宜嵌入主业务逻辑 |
| claw-code | 生态哲学层 | 仅保留对多 agent 自治闭环的启发 | 参考重写 | 平台 Day 1 先做单平台统一治理，不直接复制整套自治生态 |

推荐结论：

- `claw-code` 更适合作为 Code Executor 的备选独立 backend，不适合作为平台主仓库的核心框架。
- 若 Day 1 追求集成速度，优先选 TypeScript 主栈适配；若后续需要更强 CLI 隔离，可增加 `claw-code` backend。

### 13.4.4 Hermes Agent

已观察到的关键触点：

- `gateway/session.py`
- `gateway/`
- `tools/checkpoint_manager.py`
- `tools/code_execution_tool.py`
- `tools/memory_tool.py`
- `tools/skill_manager_tool.py`

边界决策如下：

| 仓库 | 模块 | 动作 | 分类 | 原因 |
|---|---|---|---|---|
| hermes-agent | `gateway/session.py` | 参考 session context、来源描述、动态 prompt 注入模式 | 参考重写 | 平台也有渠道来源和 session 映射，但需要企业权限语义 |
| hermes-agent | `tools/checkpoint_manager.py` | 参考 shadow git checkpoint 机制 | 薄适配 | 对开发任务文件级 checkpoint 很有价值 |
| hermes-agent | `tools/code_execution_tool.py` | 参考远端执行、RPC 封装、脚本化工具调用 | 参考重写 | Windows 不支持、本平台不能把它直接当总执行层 |
| hermes-agent | memory / skill / dream 相关工具 | 通过 `Hermes Adapter` 对接候选结果 | 薄适配 | 很符合增强来源定位，但必须经过平台 metadata / policy / audit |
| hermes-agent | gateway 整体 | 不作为平台主入口 | 禁止直接复用为总控 | 与 OpenClaw 角色冲突，不应形成双总控 |

推荐结论：

- Hermes 最适合作为增强层提供者，而不是入口、真相源或权限系统。
- 任何 Hermes 输出只允许以候选对象进入平台，由平台完成写回与发布治理。

### 13.4.5 PostgreSQL / pgvector / Apache AGE

| 组件 | 动作 | 分类 | 原因 |
|---|---|---|---|
| PostgreSQL | 作为唯一事实源 | 直接复用 | 符合冻结决策 |
| pgvector | 作为语义候选召回能力 | 直接复用 | 满足少而准的向量检索能力 |
| Apache AGE | 作为 PostgreSQL 内图增强能力 | 直接复用 | 只做 1-2 跳图增强，不做第二事实源 |
| PostgreSQL 上游源码 | 仅作能力参考 | 参考重写 | 应用层不需要 fork 数据库本体 |

## 13.5 Day 1 建议的复用顺序

### 13.5.1 第一优先级

- OpenClaw 的接入层、Gateway 控制面、安全默认值与会话映射模式。
- PostgreSQL + pgvector 的主事实与语义召回能力。
- opencode 的 project / session / patch / tool / worktree 设计。

### 13.5.2 第二优先级

- Hermes 的 checkpoint、memory / dream / skill 候选生成能力。
- AGE 的 1-2 跳关系增强能力。

### 13.5.3 第三优先级

- claw-code 作为备选独立 Code Executor backend。
- 更深层的多 agent 协同或自治编排能力。

## 13.6 仓库改造总原则

1. 不在 Day 1 深 fork 上游大仓库。
2. 不把跨仓库复用建立在“复制整目录进来”之上。
3. 优先抽象稳定契约，再接入具体 backend。
4. 新主仓库内部必须先稳定 `contracts -> policy -> workflow -> retrieval -> executor` 顺序。
5. 所有外部能力都必须在平台侧补齐：`user_id`、`policy_snapshot_hash`、`workflow_instance_id`、`audit trace`。

## 13.7 明确禁止的改造方式

- 禁止把 OpenClaw 改造成长任务 Workflow 内核。
- 禁止把 Hermes 改造成主事实源与主权限系统。
- 禁止把 opencode / claw-code 的执行状态直接当平台状态真相源。
- 禁止把 AGE 用成独立图数据库产品。
- 禁止为了快速接通而把 `scope / owner / policy_snapshot` 放到“后续再补”。

## 13.8 自建特征定义与边界

### 13.8.1 自建特征定义

"自建特征"指平台从零构建、不依赖任何上游仓库的核心能力。这些能力是 Agent Harness 的差异化价值，也是不直接复用 OpenClaw/Hermes 的根本原因。

### 13.8.2 自建 vs 继承分类

| 能力 | 分类 | 来源 | 说明 |
|------|------|------|------|
| Workflow 状态机 | **自建** | 全新 | OpenClaw 的 task 体系不满足需求 |
| Planner 自主决策循环 | **自建** | 全新 | 无上游可复用 |
| Subagent 执行循环 | **自建** | 全新 | OpenClaw 无此概念 |
| 动态 Executor 选择 | **自建** | 全新 | 无上游可复用 |
| Policy Snapshot | **自建** | 全新 | OpenClaw 无权限快照概念 |
| Checkpoint / Resume | **自建** | 全新 | OpenClaw 无结构化检查点 |
| 混合检索编排 | **自建** | 全新 | OpenClaw 无多源检索编排 |
| Skill 自动封装 | **自建** | 参考 Hermes | Hermes 有 skill 概念但封装逻辑不同 |
| Memory 管理 | **自建** | 参考 Hermes | Hermes 有 memory 但映射规则需自建 |
| Dream 摘要 | **自建** | 参考 Hermes | Hermes 有 dream 但集成方式需自建 |
| 渠道接入 | 继承+适配 | OpenClaw | 复用渠道标准化思路，适配平台语义 |
| Gateway 控制面 | 继承+适配 | OpenClaw | 复用连接鉴权，适配多副本 |
| Session 管理 | 继承+适配 | OpenClaw | 复用 session key，适配 workflow 映射 |
| Code Executor | 继承+适配 | opencode | 复用 worktree/patch 协议，适配平台策略 |
| 审计日志 | **自建** | 全新 | OpenClaw 审计不满足企业级需求 |

### 13.8.3 自建特征边界

1. 自建特征不引入上游仓库的内部依赖。
2. 自建特征通过稳定契约（接口/事件）与继承+适配层交互。
3. 自建特征优先保证正确性，性能优化在 PoC 验证后进行。
4. 自建特征必须包含完整的错误处理、降级策略和审计记录。

## 13.9 本文输出给开发计划的直接结论

后续开发计划必须按以下边界输出：

- 主开发仓库：新建平台主仓库。
- OpenClaw：接入层与 Gateway 壳层优先复用。
- opencode：Code Executor 设计优先参考与薄适配。
- claw-code：备选 backend，不进入 Day 1 主链路。
- Hermes：仅作为增强来源接入，不进入真相层。
- PostgreSQL / pgvector / AGE：直接作为事实与检索底座。
