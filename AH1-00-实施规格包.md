# 文档 1：Frozen Decisions / 冻结决策基线

## 1.1 不可回退基线

以下内容视为 V1 冻结决策，后续开发计划、模块设计、仓库改造不得擅自偏离：

1. **部署模型**：单组织、多用户；不做多租户 SaaS。
2. **数据主事实**：PostgreSQL 是唯一事实源。
3. **向量能力**：`pgvector` 只负责语义候选召回。
4. **图能力**：`Apache AGE` 只作为 PostgreSQL 体系内图增强能力，不是第二事实源。
5. **图数据库选择**：明确不使用 Neo4j。
6. **接入与总控**：OpenClaw 负责多渠道接入、会话入口、顶层调度、任务分发、结果汇总。
7. **统一治理底座**：Workflow 是平台统一任务治理底座，不是开发专用编排器。
8. **执行分层**：必须存在独立 Executor Layer，负责阶段内自治执行。
9. **开发分支**：开发类高复杂任务走 Code Executor 分支，不让 OpenClaw 直接裸调复杂开发 CLI。
10. **Hermes 定位**：Hermes 是 memory / dream / skill 的增强来源，不是平台主事实源，不是权限真相源，不是完整企业级记忆底座。
11. **md 定位**：md 继续生成，但只是结果物、留痕、归档、汇报和重抽取证据源，不是主状态源。
12. **共享模型**：只保留“个人区 + 公共区”；公共区仅包含 `workflow` 与 `skill`，其中 `workflow` 在 V1 中语义上指可复用的 `workflow_definition` / template，不包含运行态 `workflow_instance`。
13. **权限模型**：普通用户默认严格隔离；只有 admin 具备跨用户读取、编辑、治理权限。
14. **长任务治理**：必须基于 Workflow、Checkpoint、结构化状态与审计，不依赖 md 文件和 heartbeat 模式充当主控制机制。
15. **平台闭环**：所有任务都要有执行 / 自检 / 自修复 / 主动汇报 / 归档闭环。
16. **检索目标**：上下文尽量少，但召回必须精准、有限、可控、可追溯、可审计。

## 1.2 后续开发中禁止擅自做的事

- 禁止引入 Neo4j 或任何第二事实源。
- 禁止把 AGE 升级成独立主图数据库。
- 禁止把 md、日志或对象存储重新当作主事实状态源。
- 禁止让 Hermes 直接写主事实表并绕过平台 policy / metadata / audit。
- 禁止让 OpenClaw 直接裸调开发 CLI 充当完整执行层。
- 禁止把系统扩展为多租户。
- 禁止新增部门共享、组共享、指定人可见等复杂共享层。
- 禁止 memory 黑箱默认注入模型上下文，必须经过检索编排与审计。
- 禁止绕开 policy snapshot 进行检索、调用工具、读取 memory、写入事实。

# 文档 2：PDR V1（产品定义）

## 2.1 产品定位

Agent Harness 是一个面向**企业内部单组织、多用户**场景的 Agent Harness / Agent 执行平台。平台主体基于 OpenClaw 承接多渠道接入，在其上补齐：

- 长任务治理
- 多用户隔离与统一权限控制
- 事实层与证据层闭环
- 精准、有限、可控的混合召回
- 开发类长任务执行能力
- 记忆与技能沉淀治理

## 2.2 Day 1 目标用户

- 销售
- 售前
- 开发
- 实施
- 项目经理
- leader / 管理者
- admin / 平台治理人员

## 2.3 Day 1 核心能力

### 2.3.1 多渠道接入

- 企业微信
- 飞书
- Web Portal

### 2.3.2 统一 Workflow 治理

- 任务接收后生成 Workflow 实例
- 阶段化执行
- 检查点保存
- 超时、重试、恢复、暂停、人工挂起
- 主动汇报与归档

### 2.3.3 Executor Layer

- 通用执行器
- 检索感知执行器
- 验证执行器
- 修复执行器
- Code Executor

### 2.3.4 事实层与召回

- 原始证据入库
- 结构化对象与关系维护
- 混合检索
- Evidence Pack 输出
- 事实冲突记录
- 召回审计与回放

### 2.3.5 记忆与技能沉淀

- Hermes memory / dream / skill 接入
- 平台侧 metadata / scope / policy / audit 补齐
- Workflow skill 与 Dream 两条来源链分离管理

### 2.3.6 单机目标

目标为一台 **4C16G** 服务器支撑：

- 20 用户可用
- 20 个 Workflow 运行态并发

说明：该数值是**工程目标**，不是已验证结论，必须通过 PoC、压测与限流策略验证。

## 2.4 非目标

- 不做多租户 SaaS
- 不重造完整 AI IDE
- 不重造向量数据库与图数据库
- 不重造 Hermes 整套记忆引擎
- 不做复杂共享层级
- 不追求 Day 1 覆盖所有业务对象抽取

## 2.5 成功标准

1. OpenClaw、Workflow、Executor、Hermes、Fact Layer 边界清晰稳定。
2. Workflow 成为统一任务治理底座。
3. 开发类任务可通过 Code Executor 实现规划、执行、验证、修复、汇报闭环。
4. PostgreSQL 成为唯一事实源。
5. 检索满足“少而准、有限、可控、可审计、可回放”。
6. 多用户隔离与 admin 治理真实生效。
7. Hermes 被整合为增强来源而非真相源。
8. 后续开发智能体可基于本规格包直接生成可靠开发计划。

# 文档 3：SAD V1（系统架构设计）

## 3.1 总体分层

### 3.1.1 接入与总控层（OpenClaw Layer）

组件：

- WeCom Adapter
- Feishu Adapter
- Web Portal
- Session Entry
- Identity Resolver
- Request Router
- Top-level Agent Coordinator
- Response Aggregator

职责：

- 接收用户请求
- 解析身份并绑定 `user_id`
- 把任务转交 Workflow 治理层
- 向用户回传阶段进度、最终结果与追问

不负责：

- 长任务持续执行
- 阶段内部自治循环
- 复杂 checkpoint 恢复
- 开发类 CLI 直接执行治理

### 3.1.2 Workflow 治理层

组件：

- Workflow Planner
- Stage Orchestrator
- State Machine Manager
- Checkpoint Manager
- Timeout / Retry Controller
- Pause / Resume Controller
- Approval Hook
- Progress Reporter
- Archive Manager
- Audit Writer

职责：

- 把任务变成阶段化工作流
- 管理状态迁移
- 管理超时、重试、暂停、恢复、失败处理
- 管理进度上报
- 管理归档与沉淀触发

### 3.1.3 Executor Layer

组件：

- Generic Task Executor
- Retrieval-aware Executor
- Verification Executor
- Repair Executor
- Code Executor

职责：

- 在单阶段内完成自治执行
- 调用工具与能力
- 产生阶段输出
- 进行自检与小步修复
- 输出证据与结构化结果

### 3.1.4 Fact / Retrieval Layer

组件：

- Structured Query Service
- Fulltext Retrieval Service
- Vector Retrieval Service
- Graph Enhancement Service
- Rerank Service
- Evidence Pack Builder
- Fact Writer
- Conflict Detector

职责：

- 统一事实存储与召回
- 对外提供受权限约束的检索与落库接口
- 把检索结果标准化为 Evidence Pack

### 3.1.5 Memory / Skill Enhancement Layer

组件：

- Hermes Adapter
- Memory Service
- Dream Processor
- Skill Builder
- Skill Registry
- Memory Policy Guard
- Memory Write-back Gate

职责：

- 接收 Hermes 能力
- 输出平台可治理的 memory / dream / skill 候选
- 严格通过平台写回与检索编排

### 3.1.6 Provider Layer

组件：

- LLM Provider Adapter
- Embedding Provider Adapter
- Rerank Provider Adapter
- External Tool Adapter
- Graph Query Adapter

职责：

- 隔离外部模型与第三方能力
- 提供统一输入输出协议
- 屏蔽供应商替换成本

### 3.1.7 Infrastructure Layer

组件：

- PostgreSQL（含 pgvector、AGE）
- Redis
- Object Storage
- Observability

## 3.2 关键边界

### 3.2.1 OpenClaw 与 Workflow 的边界

- OpenClaw 接住请求、路由请求、汇总结果。
- Workflow 治理任务生命周期。
- OpenClaw 不直接掌管复杂长执行细节。

### 3.2.2 Workflow 与 Executor 的边界

- Workflow 管阶段级生命周期与治理。
- Executor 管阶段内动作循环。
- Workflow 看“阶段是否达标”；Executor 看“这一步怎么做”。

### 3.2.3 Executor 与事实层的边界

- Executor 只能通过 Fact / Retrieval Layer 读写事实。
- 不能绕过策略直接改主事实表。
- 任何事实写入都要带证据、来源、作用域与审计信息。

### 3.2.4 Hermes 与平台的边界

- Hermes 只提供增强能力与候选结果。
- 平台负责 scope、policy、落库、审计、检索编排、回写规则。
- Hermes 不能被视为平台真相源。

## 3.3 统一运行主路径

1. 用户从企业微信、飞书或 Web 发起请求。
2. OpenClaw 完成身份解析、用户映射、请求路由。
3. Workflow Planner 根据任务类型、权限快照、预算、上下文与目标生成 Workflow。
4. Stage Orchestrator 选择当前阶段和对应 Executor。
5. Executor 拉取最小必要 Evidence Pack。
6. Executor 执行、验证、修复，并回写阶段结果。
7. Workflow 更新状态、保存 checkpoint、上报进度。
8. 任务结束后归档，并按策略触发 skill / dream 沉淀。

## 3.4 开发任务专用主路径

1. 请求被识别为开发类高复杂任务。
2. Workflow 生成开发任务阶段链。
3. `Code Executor Adapter` 创建执行会话。
4. Code Executor 在阶段内完成局部规划、代码修改、测试、修复、验证。
5. 关键产出通过 artifact、evidence、fact write、checkpoint 回写平台。
6. Workflow 根据验证结果决定继续、修复、等待用户、阻塞、失败或完成。

# 文档 4：Workflow & State Machine V1

## 4.1 Workflow 的统一定位

Workflow 是**平台统一任务治理底座**，覆盖：

- 知识检索
- 销售支持
- 售前方案
- 开发任务
- 实施任务
- 分析任务
- 文档生成
- 管理审批与治理任务

开发只是其中的高复杂专用分支。

## 4.2 Workflow 输入

生成 Workflow 时至少要读取：

- 用户目标
- 渠道来源
- `user_id`
- `policy_snapshot`
- 当前事实状态
- 可见公共资产
- 用户私有资产
- 任务类型
- 风险等级
- 检索预算
- 执行预算
- 时间预算

## 4.3 Workflow 输出最小结构

```json
{
  "workflow_id": "wf_xxx",
  "workflow_type": "development|analysis|sales|implementation|knowledge",
  "owner_user_id": "u_xxx",
  "scope_type": "private|public-governed",
  "workflow_plan_hash": "sha256:...",
  "stages": [],
  "selected_executors": [],
  "checkpoint_policy": {},
  "report_policy": {},
  "repair_policy": {},
  "approval_policy": {},
  "archive_policy": {}
}
```

## 4.4 标准状态集合

- `draft`
- `planned`
- `running`
- `waiting_user`
- `blocked`
- `verifying`
- `repairing`
- `reporting`
- `paused`
- `failed`
- `succeeded`
- `cancelled`
- `archived`

## 4.5 异常边界精确定义

### 4.5.1 `waiting_user`

定义：系统已经完成当前可执行部分，但继续推进必须依赖用户补充信息、确认或选择。

进入条件：

- 缺关键输入
- 需要用户确认方案分支
- 需要用户提供凭证、仓库、附件或授权
- 风险动作必须用户确认

退出条件：

- 用户补齐信息
- 用户明确拒绝继续，转 `cancelled`
- 超时未响应，按策略转 `paused` 或 `failed`

### 4.5.2 `blocked`

定义：系统仍想继续，但被外部依赖、策略限制或环境异常卡住，当前不能靠用户简单补一句话解决。

进入条件：

- 工具 / 外部服务不可用
- 代码仓或环境不可访问
- 依赖安装失败且无法自动修复
- 权限被策略拒绝
- 事实冲突导致无法继续自动推进

退出条件：

- 依赖恢复
- 管理员干预完成
- 策略放行
- 转人工接管
- 转 `failed`

### 4.5.3 `paused`

定义：任务被主动挂起，后续可从最近 checkpoint 恢复，挂起原因可以是人工、预算、排队、夜间窗口、长时间无响应等。

进入条件：

- 用户手动暂停
- admin 手动暂停
- 系统预算耗尽但允许后续恢复
- 调度层重阶段排队
- 到达维护窗口或限流窗口

退出条件：

- 用户 / admin 恢复
- 调度器恢复
- 超过最大暂停期限后转 `failed` 或 `cancelled`

### 4.5.4 `failed`

定义：当前 Workflow 在现有预算、策略、重试与恢复规则下已不能继续自动完成。

进入条件：

- 超过最大重试次数
- 超过最大修复次数
- 多次恢复仍失败
- 关键证据缺失且无法补齐
- 外部依赖长期不可用
- 验证不通过且无可行修复路径

退出条件：

- 仅允许进入 `archived`
- 如需重新开始，应新建 Workflow 或人工显式克隆失败实例

## 4.6 状态迁移规则

### 4.6.1 主迁移

- `draft -> planned`
- `planned -> running`
- `running -> verifying`
- `verifying -> repairing`
- `repairing -> verifying`
- `verifying -> reporting`
- `reporting -> running`
- `reporting -> succeeded`
- `running -> waiting_user`
- `running -> blocked`
- `running|verifying|repairing|reporting -> paused`
- `paused -> running`
- `planned|running|waiting_user|blocked|paused -> cancelled`
- `planned|running|waiting_user|blocked|paused|verifying|repairing -> failed`
- `succeeded|failed|cancelled -> archived`

### 4.6.2 禁止迁移

- `failed -> running`
- `succeeded -> running`
- `archived -> running`
- `waiting_user -> succeeded`
- `blocked -> succeeded`

说明：禁止跳过恢复与验证直接改写结果状态。

## 4.7 超时、重试、降级、恢复规则

### 4.7.1 超时规则

- 每个阶段必须配置 `soft_timeout_sec` 与 `hard_timeout_sec`
- 到达软超时：触发主动汇报，并评估是否进入 `paused` 或 `blocked`
- 到达硬超时：强制中断当前执行单元，进入恢复或失败流程

### 4.7.2 重试规则

- 同一阶段的同类工具错误必须有最大重试次数
- 默认优先小步重试，不允许无界重试
- 每次重试必须记录错误分类、输入摘要、预算消耗与结果

### 4.7.3 降级规则

典型降级：

- 图增强不可用时，只保留结构化 + 全文 + 向量召回
- rerank 不可用时，使用规则裁剪替代
- memory 服务不可用时，跳过 memory 检索但保留事实层
- Code Executor 子能力失败时，尝试切换到较弱执行模式或转人工确认

### 4.7.4 恢复规则

- 恢复只能从已提交 checkpoint 或已确认阶段边界开始
- 不允许从未落盘的推理中间态做伪恢复
- 恢复必须绑定原 `policy_snapshot_hash`
- 恢复必须保留原失败证据、恢复原因与恢复次数

### 4.7.5 回放边界

回放最小集合：

- `workflow_plan_hash`
- `policy_snapshot_hash`
- `checkpoint_id`
- `evidence_pack_hash`
- `tool_call_refs`
- `artifact_refs`

# 文档 5：Fact / Memory / Retrieval Orchestration V1

## 5.1 总原则

1. PostgreSQL 是唯一事实源。
2. memory 是辅助召回层，不替代事实层。
3. Hermes 不能直接写主事实。
4. 召回必须走混合链路，不能只靠向量。
5. 所有召回都必须先过权限过滤。
6. 输出必须标准化为 Evidence Pack。

## 5.2 事实层五层模型

1. 原始证据层：原始消息、文档、网页、日志、工具输出、md。
2. 对象层：客户、项目、需求、风险、决策、workflow、skill、repo、环境等。
3. 关系层：belongs_to、uses、affects、derived_from、references 等。
4. 事实声明层：某事实是什么、依据哪些证据、状态如何、版本如何。
5. 检索索引层：结构化查询、全文、向量、图增强。

## 5.3 记忆层定位

### 5.3.1 memory 用途

- 用户长期偏好
- 历史处理经验
- Dream 总结摘要
- skill 形成原料
- 少量高价值经验线索

### 5.3.2 memory 不得承担

- 客观业务主事实
- 权限绕过读取
- 未经证据支撑的状态覆盖
- 未经审计的隐式上下文注入

## 5.4 检索编排规则

### 5.4.1 规则总表

| 场景 | 优先检索 | 可选增强 | 禁止行为 |
|---|---|---|---|
| 查明确对象状态 | 结构化查询 | 全文补充 | 先上向量全库乱搜 |
| 查原文证据 | 全文/关键词 | 向量召回 | 只返回摘要不带证据 |
| 查相似案例 | 向量召回 | rerank、图扩展 | 绕过权限直接搜公共外全部 |
| 查关联对象 | 结构化 + 图增强 | 全文补充 | 无边界多跳漫游 |
| 查用户偏好/历史经验 | memory 检索 | 向量、rerank | 把 memory 当主事实 |
| 查开发上下文依赖 | 结构化 + 全文 + 向量 | 图增强 | 直接把整个仓库上下文塞入模型 |

### 5.4.2 结构化优先触发条件

以下场景必须先查结构化：

- 明确对象 ID / 名称 / owner 已知
- 查 workflow、stage、checkpoint、policy、audit、skill 元数据
- 查“我名下”“当前用户”“公共区”之类强作用域问题
- 查事实状态、执行状态、发布状态、审核状态

### 5.4.3 memory 检索触发条件

以下场景才进入 memory：

- 用户偏好、长期习惯、历史经验显著相关
- 当前任务需要参考过往处理方法，但这些信息不是主事实
- 需要 Dream 级摘要或经验性线索

以下场景默认不查 memory：

- 客观状态查询
- 权限判断
- 最终审批结论
- 主事实冲突裁决

### 5.4.4 图增强触发条件

以下场景启用 AGE：

- 需要一跳 / 二跳关系补充
- 需要依赖链、影响链、引用链
- 已经拿到主对象，需扩展周边高相关对象
- 需要路径约束做结果收束

以下场景跳过 AGE：

- 查询已足够精确
- 图相关预算耗尽
- 图增强服务不稳定
- 当前问题只是查单对象单字段

约束：

- `max_hops <= 2`
- 必须带类型过滤与数量上限
- 图结果必须能映射回 PostgreSQL 记录主键

### 5.4.5 rerank 触发条件

满足以下任一条件触发 rerank：

- 候选条目超过阈值
- 来源跨多个检索子系统
- 问题是模糊描述或多意图混合
- 需要把结构化、全文、向量、图结果统一排序

### 5.4.6 裁剪规则

进入模型上下文前必须裁剪：

- 只保留当前任务最相关片段
- 优先保留新近且证据强的材料
- 去掉重复片段
- 去掉跨 scope 的边缘材料
- 保留引用信息与最小解释头

### 5.4.7 降级规则

- 结构化失败：回退全文 / 关键词，但需标记结果可信度下降
- 向量失败：保留结构化 + 全文
- AGE 失败：跳过图扩展，记录降级原因
- rerank 失败：采用规则排序
- memory 失败：跳过 memory，不阻断主链路

## 5.5 Evidence Pack 最小结构

```json
{
  "evidence_pack_id": "ep_xxx",
  "query_text": "...",
  "intent_type": "object-status|evidence|relation|similar-case|dev-context",
  "scope_summary": {
    "user_id": "u_xxx",
    "allowed_scopes": ["private:u_xxx", "public:workflow", "public:skill"]
  },
  "retrieval_steps": [
    {"type": "structured", "ref": "..."},
    {"type": "vector", "ref": "..."},
    {"type": "graph", "ref": "..."}
  ],
  "items": [
    {
      "item_type": "document_chunk|memory|fact|entity|relation|artifact_excerpt",
      "item_ref": "...",
      "score": 0.93,
      "evidence_ref": "...",
      "source_scope": "private:u_xxx"
    }
  ],
  "clip_summary": "...",
  "evidence_pack_hash": "sha256:..."
}
```

## 5.6 冲突处理规则

- 不直接覆盖旧事实。
- 记录冲突对。
- 绑定新旧证据。
- 更新 `fact_status`。
- 必要时进入人工确认。
- 在确认前，冲突事实不得直接升级为公共可信结论。

# 文档 8：Code Executor Integration Spec V1

## 8.1 定位

Code Executor 是开发类高复杂任务的专用执行分支，承接：

- 局部任务规划
- 代码阅读
- 代码修改
- 测试执行
- 修复循环
- 结果汇报
- 证据产出

它不是平台总控，不管理全局 Workflow 生命周期。

## 8.2 集成原则

1. 优先复用成熟开源 CLI / 引擎能力。
2. 平台只补齐执行治理、状态回写、汇报、恢复、权限、审计。
3. 绝不让 OpenClaw 直接裸调 Code CLI。
4. 必须通过 `Code Executor Adapter` 接入。

## 8.3 执行会话模型

每次进入开发执行阶段时创建 `execution_session`（逻辑对象，可先落在 `workflow_stage + checkpoint + artifact refs` 组合中，后续再独立建表）。

会话至少包含：

- 会话 ID
- 关联 `workflow_instance_id`
- 关联 `workflow_stage_id`
- 仓库引用
- 分支引用
- 工作目录引用
- 执行预算
- 当前目标
- 当前验收规则
- 当前 checkpoint ref

## 8.4 标准执行循环

1. 读取阶段目标与约束。
2. 读取必要代码上下文与规格约束。
3. 生成局部实现计划。
4. 执行修改。
5. 执行验证命令 / 测试。
6. 收集失败证据。
7. 进行小步修复。
8. 回写进度、证据、patch、测试结果。
9. 达到阶段验收后交回 Workflow。

## 8.5 必须回写的平台信息

- 当前阶段摘要
- 修改文件清单
- patch / diff 引用
- 测试结果引用
- 错误日志引用
- 验证是否通过
- 当前剩余预算
- checkpoint 信息
- 下一步建议

## 8.6 验证与修复闭环

### 验证输入
- 阶段验收规则
- 测试命令或验证脚本
- 关键断言

### 验证输出
- `PASS|FAIL|PARTIAL`
- 失败分类
- 失败证据
- 是否允许自动修复

### 修复规则
- 必须基于失败证据修复
- 优先局部修复
- 达到最大修复次数后交回 Workflow 决策

## 8.7 与 Workflow 的边界

Workflow 决定：

- 是否进入开发分支
- 当前阶段目标是什么
- 修复是否继续
- 是否等待用户
- 是否失败终止
- 是否归档沉淀

Code Executor 决定：

- 阶段内部怎样规划动作
- 读取哪些必要代码上下文
- 采用怎样的小步修改与验证顺序

## 8.8 第一阶段复用策略

优先从 opencode / clawcode / 其他成熟 CLI 中复用：

- repo 操作能力
- 文件修改能力
- 测试命令执行能力
- patch 生成与回传能力
- 长循环执行经验

平台侧自研或薄适配：

- `Code Executor Adapter`
- 进度回调协议
- checkpoint 协议
- artifact / evidence 写回协议
- policy / audit 绑定
- Workflow 阶段级集成

# 文档 9：Capacity / Risk / Validation Plan V1

## 9.1 工程约束声明

- 单机 **4C16G** 支撑 **20 用户 / 20 运行态 Workflow** 是目标值，不是既成事实。
- 所有容量结论都必须来自 PoC、压测与限流验证。
- 高复杂开发阶段不能被默认等同于轻任务运行态。

## 9.2 Day 1 槽位建议

- Workflow 运行态槽位：20
- 通用 Executor 槽位：4
- 重检索 / rerank 槽位：4
- Code Executor 槽位：2~4
- 测试 / 修复槽位：2~4
- 文件处理槽位：2
- Embedding 建索引槽位：1~2

## 9.3 必做 PoC

### P0-PoC-1：OpenClaw 与 Workflow 解耦接入
验证点：
- OpenClaw 能否稳定把请求转为 Workflow 实例
- 进度与结果能否正确回传

### P0-PoC-2：Workflow + Checkpoint + Resume
验证点：
- 长任务中断后是否能从 checkpoint 恢复
- `waiting_user / blocked / paused / failed` 是否可区分

### P0-PoC-3：PostgreSQL + pgvector + 权限过滤
验证点：
- 强权限过滤下向量召回性能是否可接受
- 检索 trace 是否完整

### P0-PoC-4：AGE 图增强
验证点：
- 1~2 跳扩展的性能
- 与主事实表的一致性映射
- 在 4C16G 下的内存 / CPU 风险

### P0-PoC-5：Code Executor Adapter
验证点：
- 现有开源 CLI 是否可被封装为阶段内执行器
- 进度、checkpoint、测试结果是否能可靠回写

### P0-PoC-6：Hermes 增强接入
验证点：
- memory / dream / skill 候选是否能被平台治理吸收
- 是否能严格受 scope / policy 控制

## 9.4 主要技术风险

1. **AGE 风险**：在单机资源下的稳定性与查询成本需要实测。
2. **pgvector 风险**：强权限过滤叠加向量召回可能带来性能压力。
3. **Code Executor 风险**：开发长任务可能优先耗尽 token、等待时间与测试资源，而不是 CPU。
4. **恢复风险**：如果 checkpoint 粒度设计不合理，恢复会退化为重新执行。
5. **多用户风险**：如果 `scope / owner / snapshot` 字段体系不统一，极易出现串数。
6. **Hermes 集成风险**：若边界不清，容易把 memory 系统误当真相源。

## 9.5 压测与验证指标

至少验证：

- Workflow 平均启动耗时
- 阶段切换耗时
- 检索耗时（结构化 / 全文 / 向量 / 图增强）
- Evidence Pack 构建耗时
- Checkpoint 保存与恢复耗时
- Code Executor 平均阶段执行耗时
- 失败恢复成功率
- 跨用户越权读取次数（目标为 0）
- 审计事件完整率

# 文档 10：Acceptance Criteria V1

## 10.1 架构验收

- OpenClaw、Workflow、Executor、Fact Layer、Hermes Adapter 分层清晰。
- OpenClaw 不直接承担复杂长任务持续执行。
- PostgreSQL 是唯一事实源。
- AGE 仅为图增强层。

## 10.2 Workflow 验收

- Workflow 能覆盖至少开发、知识检索、分析三类任务。
- `waiting_user / blocked / paused / failed` 可明确区分。
- 超时、重试、恢复、归档规则可执行。
- 任务状态不依赖 md / heartbeat 维护。

## 10.3 检索验收

- 检索链路包含权限过滤、结构化、全文、向量、图增强、rerank、裁剪、Evidence Pack。
- memory 检索可控、可审计、可关闭。
- 冲突事实不会直接覆盖旧事实。

## 10.4 权限验收

- 用户彼此默认严格隔离。
- 只有 admin 可跨用户读取与编辑。
- 共享模型仅保留个人区与公共区。
- 公共区仅有 workflow 与 skill。

## 10.5 Code Executor 验收

- 开发任务可在 Workflow 下进入 Code Executor 分支。
- Code Executor 可回写 patch、测试结果、checkpoint、进度。
- 验证失败后可进入修复循环。
- 达到上限后能交回 Workflow 决策。

## 10.6 审计验收

- 身份解析、权限裁决、检索、工具调用、事实写入、公共发布均有审计。
- 任意一次检索可追溯到 retrieval trace 与 Evidence Pack。

## 10.7 恢复验收

- 中断后可从 checkpoint 恢复。
- 恢复仍绑定原 policy snapshot。
- 回放最小集合完整。

## 10.8 容量验收

- 在目标资源下，运行态 Workflow 与重阶段槽位可被调度与限流。
- 20 运行态不导致权限失效与审计丢失。
- 重开发阶段不会挤爆全部系统资源。
